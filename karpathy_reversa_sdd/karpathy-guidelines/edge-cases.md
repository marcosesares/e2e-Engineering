## EC-01 — Trivial Task Boundary

**Description:** The preamble states "for trivial tasks, use judgment." No formal definition of "trivial" is given.

**Risk:** The LLM may incorrectly classify a non-trivial task as trivial and skip pre-condition guards (Think Before Coding) or scope minimization (Simplicity First), leading to unchecked assumptions or over-engineered code.

**Resolution:** 🟢 CONFIRMADO — Maintainer defined 2026-05-15:
- **Trivial:** clearly local, low-risk, obvious correct answer — typo fixes, short copy edits, simple formatting, provided renames, changing a small literal value. For trivial tasks: act directly, avoid unnecessary explanation.
- **Non-trivial:** ambiguous, multi-file, behavior change, security/data impact, debugging, architecture, refactoring, migration, optimization, feature design. For non-trivial tasks: think first, state a concise plan, then proceed surgically.

## EC-02 — Duplicate Activation (Plugin + CLAUDE.md)

**Description:** A project may have both the plugin installed and `CLAUDE.md` containing a copy of the skill content.

**Risk:** The LLM's context contains the rules twice, potentially causing rule conflicts if the versions differ (e.g., CLAUDE.md was updated after plugin was installed).

**Current behavior:** No deduplication guard. Both copies are loaded. 🟡 INFERIDO

**Suggested mitigation:** Document that CLAUDE.md takes precedence, or add a marker comment so the LLM can detect duplication.

## EC-03 — Breaking Change to Rule Set

**Description:** A maintainer updates a behavioral rule (e.g., tightens BR-11) in a new version of `SKILL.md`. Projects using the plugin get the update automatically on next install; projects using a static `CLAUDE.md` copy do not.

**Risk:** Silent behavioral divergence between plugin users and CLAUDE.md users.

**Current behavior:** No versioning policy. No semver on rule changes. 🔴 LACUNA

**Suggested mitigation:** Add a version header to `SKILL.md` (e.g., `version: 1.1.0`) and document a policy: "any change to a rule is a minor bump; removing a rule is a major bump."

## EC-04 — Translation Divergence

**Description:** `SKILL.zh.md` is maintained independently with no automated sync to `SKILL.md`.

**Risk:** A Chinese-language user installs the skill and receives different behavioral constraints than an English-language user.

**Current behavior:** No CI check. Manual review only. 🟡 INFERIDO

**Suggested mitigation:** Add a CI job that diffs section counts and headings between `SKILL.md` and `SKILL.zh.md`. Content translation differs, but structure (number of principles, number of rules per principle) must match.

## EC-05 — Plugin Schema Regression

**Description:** The plugin manifest (`plugin.json`) has already broken once (ADR-006) due to schema errors introduced by a community contributor.

**Risk:** Future contributions modify `plugin.json` and reintroduce schema errors, silently breaking the plugin install path for all users.

**Current behavior:** Fixed as of `68b67a5`. No schema validation in CI. 🟡 INFERIDO

**Suggested mitigation:** Add a CI step: `npx ajv validate -s <plugin-schema> -d .claude-plugin/plugin.json`.

## EC-06 — Skill Content Loaded but LLM Ignores It

**Description:** The LLM's context window is full, or an earlier system prompt overrides the skill content.

**Risk:** All four principles are silently inactive even though installation succeeded.

**Current behavior:** No detection mechanism. No observability at install time. 🔴 LACUNA

**Suggested mitigation:** Add an activation acknowledgment: after install, prompt the LLM with "List the four Karpathy guidelines" to confirm they are accessible in context.
