# Simplicity First, Technical Design

> Spec generated by Reversa Writer · 2026-05-15
> Unit: `simplicity-first` · Feature-level · doc_level: detalhado

---

## Interface

No programmatic interface. The principle operates as a set of negative constraints applied during code generation. Each rule is a prohibition that fires if its condition is detected in the draft output. 🟢

---

## Decision Procedure

```
GENERATE draft implementation
├── CHECK: does output contain features not present in the request?
│   └── IF YES → remove unrequested features
├── CHECK: does output contain abstractions used only once?
│   └── IF YES → inline the abstraction
├── CHECK: does output add config/flags not in the request?
│   └── IF YES → remove configuration surface
├── CHECK: does output handle errors for impossible inputs?
│   └── IF YES → remove the guard
├── CHECK: is implementation 4× longer than the minimum viable solution?
│   └── IF YES → rewrite to minimum viable
└── APPLY senior engineer test: "Would a senior engineer call this overcomplicated?"
    └── IF YES → simplify further
    └── OUTPUT
```
🟢 CONFIRMADO — rules enumerated in `SKILL.md:14–23`

## Heuristic Tests

| Test | Trigger | Action |
|------|---------|--------|
| Size ratio | Draft is ≥4× lines of minimum viable | Rewrite |
| Senior engineer test | "Would a senior say this is overcomplicated?" | Simplify |

Both tests are self-applied by the LLM — no external evaluator. 🟡 INFERIDO — consistency depends on LLM judgment.

---

## Alternative Flows

- **User explicitly requests a complex solution:** BR-05–BR-10 yield to explicit user specification. If the user asks for configurability, add it. 🟡 INFERIDO — "beyond what was asked" implies explicit asks are honored.
- **Existing code is complex and the edit task is simple:** Simplicity First applies to the LLM's own output, not to surrounding code. BR-09 does not trigger on code the LLM did not write. 🟢 CONFIRMADO — Surgical Changes (BR-11–BR-12) governs this scenario instead.
- **"Impossible" scope:** "Impossible" = impossible under the local visible contract (caller guarantees visible in scope). If caller guarantees are not visible, the guard is acceptable. 🟢 CONFIRMADO — confirmed by maintainer 2026-05-15 [Revisão: 🔴→🟢]

## Dependencies

- No external dependencies. 🟢
- Interacts with Surgical Changes principle: Simplicity First governs what is written in new code; Surgical Changes governs what is touched in existing code. They are complementary, not overlapping. 🟢

## Design Decisions

| Decision | Evidence | Confidence |
|----------|----------|-----------|
| Size constraint is expressed as ratio (4×), not absolute lines | `SKILL.md:20`: "200 lines that could be 50" — ratio is 4:1 | 🟢 |
| "Senior engineer test" is an embedded heuristic, not a formal metric | `SKILL.md:21`: phrased as a question, not a measurable threshold | 🟢 |
| Rules are prohibitions (what NOT to do), not prescriptions (what TO do) | All 5 rules begin with "No" or "Don't" / "If ... could be ... rewrite" | 🟢 |
