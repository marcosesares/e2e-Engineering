## Interface

No programmatic interface. The principle operates as two sequential constraint sets applied during edit generation: one governing what the LLM touches in existing code, and one governing cleanup of orphans its changes create. 🟢

---

## Two Operating Modes

### Mode A — When editing existing code (4 rules)

Applied to any line the LLM touches in a file that existed before the task:

| Rule | Constraint |
|------|-----------|
| BR-11 | Adjacent code, comments, formatting → read-only |
| BR-12 | Non-involved functions/classes → read-only |
| BR-13 | Existing style → must be matched |
| BR-14 | Discovered dead code → mention, do not touch |

### Mode B — When own changes create orphans (2 rules)

Applied after the LLM's edits are complete, as a post-generation cleanup pass:

| Rule | Constraint |
|------|-----------|
| BR-15 | Imports/vars/functions made unused by LLM's own changes → remove |
| BR-16 | Pre-existing dead code → do not remove |

🟢 CONFIRMADO — two-section structure matches `SKILL.md:25–38`

## Decision Procedure

```
APPLY requested change to target location
│
├── MODE A (during edit)
│   ├── Scope: only lines directly required by request
│   ├── Adjacent code → leave untouched
│   ├── Style → match surrounding code
│   └── Discovered dead code → note in response, do not delete
│
└── MODE B (post-edit cleanup)
    ├── Scan for symbols made unused by LLM's own changes
    ├── Remove those symbols (BR-15)
    └── Leave all pre-existing unused symbols untouched (BR-16)

APPLY traceability test:
└── For each changed line: does it trace to the user's request?
    └── IF NO → revert
```
🟢 CONFIRMADO — `SKILL.md:36`: "Every changed line should trace directly to the user's request."

## Alternative Flows

- **User explicitly asks the LLM to clean up surrounding code:** BR-11 and BR-12 yield to explicit instruction. The cleanup becomes part of the request. 🟡 INFERIDO — "beyond what was asked" implies explicit asks are honored.
- **Shared helper that must change and affects callers outside the request:** Update the helper AND all callers that would break as a result, but only as far as necessary to preserve correctness. Do not update callers for reasons unrelated to the breaking change. 🟢 CONFIRMADO — confirmed by maintainer 2026-05-15 [Revisão: 🔴→🟢]
- **Existing style is inconsistent within the file:** LLM should match the style of the immediate surrounding context, not attempt to reconcile the inconsistency. 🟡 INFERIDO
- **Orphan is an exported/shared symbol:** Verify cross-file usage before removing. If cross-file usage cannot be confirmed, mention the potential orphan but do not delete. 🟢 CONFIRMADO — confirmed by maintainer 2026-05-15 [Revisão: 🔴→🟢]

---

## Embedded Test

> "Every changed line should trace directly to the user's request."

The LLM is expected to self-apply this test before outputting the diff. If any line fails the trace, it is reverted. 🟢

## Dependencies

- Complementary to Simplicity First: Simplicity First governs new code generation; Surgical Changes governs edits to existing code. Neither overrides the other. 🟢
- No external dependencies. 🟢

## Risks and Gaps

- 🟢 **Cross-file orphan scope defined** — verify cross-file usage for exported/shared symbols; mention but do not delete if unverifiable. [Revisão: 🔴→🟢]
- 🟡 **Style-matching in inconsistent codebases** — if the existing file has mixed styles, "match existing style" is ambiguous; LLM may choose wrong reference point
- 🟡 **Adjacent code boundary** — "adjacent" is not formally defined; LLM must judge how many lines away from the change counts as adjacent
