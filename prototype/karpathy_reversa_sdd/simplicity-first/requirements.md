## Overview

A behavioral principle that enforces minimum-viable code output. It combats the LLM tendency toward over-engineering: adding unrequested features, premature abstractions, defensive error handling for impossible cases, and unnecessarily large implementations. 🟢

---

## Responsibilities

- Prevent the LLM from generating features beyond what was explicitly requested 🟢
- Prevent premature abstraction for single-use code 🟢
- Prevent unrequested flexibility or configurability 🟢
- Prevent error handling for scenarios that cannot occur 🟢
- Enforce a size constraint: if a 200-line implementation could be 50 lines, rewrite it 🟢
- Apply the "senior engineer test" as a final complexity gate 🟢

## Acceptance Criteria

```gherkin
# Happy path — no extras added
Given the user asks "add a greet function that returns Hello, <name>"
When the LLM applies Simplicity First
Then the output is a single function with no config, no validation, no logging

# Happy path — size constraint
Given the LLM drafts a 180-line implementation
When Simplicity First is applied
Then the LLM identifies that the same logic fits in ~45 lines and rewrites before outputting

# Failure — unrequested feature
Given the user asks "add login endpoint"
When the LLM adds rate limiting, password strength check, and audit logging unprompted
Then the principle has been violated — those features must be removed

# Failure — impossible error handling
Given a function that always receives a non-null config object by construction
When the LLM adds a null-check guard with an error thrown
Then the principle has been violated — the check handles a scenario that cannot occur
```
