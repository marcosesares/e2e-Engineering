## Summary

**andrej-karpathy-skills** is a **content distribution system** for LLM behavioral guidelines — not a runtime application. It has no server, no database, no API, and no executable code. Its "architecture" is entirely a distribution and loading pipeline: from a GitHub repository through various install mechanisms into the Claude Code LLM context.

| Property | Value |
|----------|-------|
| Type | Claude Code skill plugin |
| Runtime | None (static content) |
| Primary language | Markdown |
| Executable components | 0 |
| External integrations | Claude Code runtime, GitHub, Claude Marketplace, skills.sh |
| Deployment | Static file hosting via GitHub |

---

## Architectural Style

🟢 **CONFIRMADO** — **Content-as-Code / Prompt Engineering as a Package**

The project applies software distribution patterns (versioning, manifests, marketplace listings, install commands) to a non-executable artifact: a set of behavioral instructions for an LLM. The "runtime" is Claude Code's context injection — the LLM reads the skill content and modifies its own behavior accordingly.

## Distribution Channels

Three independent install paths, each loading the same behavioral rules into Claude Code:

```
GitHub Repository
      │
      ├──── /plugin install ──────► Plugin Manifest (.claude-plugin/plugin.json)
      │                                   └── Skill Content (skills/karpathy-guidelines/)
      │                                           └── Claude Code skill runtime
      │
      ├──── curl CLAUDE.md ──────► CLAUDE.md (per-project)
      │                                   └── Claude Code project context
      │
      └──── skills.sh ───────────► skills/karpathy-guidelines/SKILL.md
                                          └── Claude Code skill runtime
```

## Deployment

🟢 **CONFIRMADO** — No deployment infrastructure. The project is statically hosted on GitHub. "Deployment" means pushing commits to the `main` branch. There is no CI/CD pipeline, no Dockerfile, no cloud configuration.

The only "release" action is updating version fields in `plugin.json` and `marketplace.json` — which is currently a manual process.
