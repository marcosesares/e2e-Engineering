## Complexity Evaluation Procedure

```mermaid
flowchart TD
    A([Implementation drafted]) --> B{Features beyond what was asked?}
    B -->|Yes| C[Remove speculative features\nOnly implement what was requested]
    C --> B
    B -->|No| D{Single-use abstraction introduced?}

    D -->|Yes — abstract class/interface for only one impl| E[Remove abstraction\nWrite direct implementation]
    E --> D
    D -->|No| F{Flexibility/configurability not requested?}

    F -->|Yes — optional params, feature flags, config systems for one use| G[Remove unasked-for configurability]
    G --> F
    F -->|No| H{Error handling for impossible scenarios?}

    H -->|Yes — catch blocks for scenarios that cannot happen| I[Remove impossible-scenario handling\nTrust internal guarantees]
    I --> H
    H -->|No| J{Line count check}

    J --> K["Apply the senior engineer test:\n'Would a senior engineer say this is overcomplicated?'"]
    K --> L{Overcomplicated?}
    L -->|Yes| M[Rewrite — target the minimal version\nTypically: 200 lines → aim for 50]
    M --> A
    L -->|No| N([Implementation approved — proceed])
```

## Anti-Patterns Detected (from EXAMPLES.md)

| Anti-Pattern | Example | Correct Behavior |
|-------------|---------|-----------------|
| Strategy pattern for single case | `DiscountStrategy` ABC + 2 subclasses for one discount calc | One function: `calculate_discount(amount, percent)` |
| Speculative caching | Adds `InMemoryCache` to `save_preferences` nobody asked for | Just run the `UPDATE` query |
| Speculative notification | Adds `notify_preference_change` nobody asked for | Delete it |
| Configuration system | `merge: bool`, `validate: bool`, `notify: bool` params for simple save | Remove all optional params until they're needed |

## Rules Summary

| ID | Rule | Type |

| SF-04 | No error handling for impossible scenarios | Hard prohibition |
