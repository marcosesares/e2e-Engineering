## Interface

No programmatic interface. The principle operates as a task-transformation and execution-loop pattern applied before and during implementation. 🟢

**Trigger:** Any multi-step or non-trivial task. Single-step trivial tasks may apply a lighter version (state goal, do once, verify once). 🟡 INFERIDO

---

## Task Transformation Pattern

The core mechanism converts imperative instructions to declarative goals:

| Imperative (input) | Declarative goal (output) |
|--------------------|--------------------------|
| "Add validation" | "Write tests for invalid inputs, then make them pass" |
| "Fix the bug" | "Write a test that reproduces it, then make it pass" |
| "Refactor X" | "Ensure tests pass before and after" |

🟢 CONFIRMADO — `SKILL.md:44–48`

## Multi-Step Plan Template

```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

Every step must have an explicit, checkable verify condition. The LLM produces this plan before executing step 1. 🟢 CONFIRMADO — `SKILL.md:50–51`

---

## Execution Loop

```
DEFINE success criteria from task
PRODUCE plan (if multi-step)
FOR EACH step in plan:
  EXECUTE step
  RUN verify check
  IF check fails:
    DIAGNOSE failure
    FIX issue
    RE-RUN verify check
  UNTIL check passes
  ADVANCE to next step
DECLARE done only after all checks pass
```
🟢 CONFIRMADO — `SKILL.md:52`; key principle: "loop until all success criteria are met"

## Alternative Flows

- **Single-step task:** Plan step is trivial (just the one step). Still requires an explicit verify condition. 🟡 INFERIDO
- **Verify check cannot be run mechanically:** Use explicit **human-gate steps** in the plan: `→ verify: [human confirms X]`. The loop pauses at that step; the LLM informs the user what to check and how to signal completion. 🟢 CONFIRMADO — confirmed by maintainer 2026-05-15 [Revisão: 🔴→🟢]
- **Loop does not converge:** After 3 failed fix attempts on the same verify check, stop, report the specific failure clearly, and ask the user for guidance. Do not continue speculating. 🟢 CONFIRMADO — confirmed by maintainer 2026-05-15 [Revisão: 🔴→🟢]

## Dependencies

- **Think Before Coding** (upstream): clarifying questions before coding feed into goal definition. A clear, unambiguous request produces tighter success criteria. 🟢
- No external tool dependencies. The verification loop is conceptual — the LLM reasons about pass/fail rather than running a real test runner (unless the skill is used in a context with tool access). 🟡 INFERIDO

## Design Decisions

| Decision | Evidence | Confidence |
|----------|----------|-----------|
| Goal-first pattern (define before implement) rather than implement-then-verify | `SKILL.md:44`: "Transform tasks into verifiable goals" — ordering is explicit | 🟢 |
| The loop is the LLM's own execution loop, not a runtime mechanism | No runtime tooling referenced; described conceptually | 🟡 |
| The principle uses TDD idiom as the canonical transformation for code tasks | `SKILL.md:44–48`: "write tests... then make them pass" | 🟢 |

## Key Quote

> "LLMs are exceptionally good at looping until they meet specific goals... Don't tell it what to do, give it success criteria and watch it go." — `README.md`

This quote encodes the design intent: success criteria unlock autonomous execution. Without them, the LLM needs constant human re-direction. 🟢 CONFIRMADO

## Internal State

Stateless per task. The plan is generated fresh for each task. 🟢

## Risks and Gaps

- 🟢 **Loop escape after 3 failed attempts** — stop, report specific failure, ask user for guidance. [Revisão: 🔴→🟢]
- 🟢 **Human-gate verify steps** — non-automatable checks declared as `→ verify: [human confirms X]`; loop pauses, LLM informs user. [Revisão: 🔴→🟢]
- 🟡 **Success criteria quality** — weak criteria ("make it work") lead to loops that never truly verify correctness; criteria quality is LLM-dependent
