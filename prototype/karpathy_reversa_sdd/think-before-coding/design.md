## Interface

No programmatic interface. The principle is expressed as LLM behavioral constraints active before any code generation begins.

**Trigger:** Any implementation request received by the LLM (the principle is a pre-condition guard, not an on-demand command). 🟢

---

## Decision Procedure

The principle encodes a sequential pre-condition check:

```
RECEIVE request
├── IF uncertain about any assumption
│   └── STATE assumption explicitly → ASK to confirm
├── IF multiple valid interpretations exist
│   └── PRESENT all interpretations → ASK which to pursue
├── IF simpler approach exists than implied
│   └── PROPOSE simpler path → EXPLAIN tradeoff
├── IF request is genuinely unclear
│   └── STOP → NAME the confusion → ASK
└── ELSE → proceed to implementation
```
🟢 CONFIRMADO — derived from `SKILL.md:5–12`

---

## Alternative Flows

- **Request is clear and unambiguous:** All four guards pass silently; LLM proceeds to implementation without any pre-coding dialogue. 🟢
- **User explicitly says "don't ask, just do it":** Honor the override but acknowledge assumptions made — e.g., "Proceeding with assumption X" — unless the action would be unsafe or destructive. The override applies to the current task only; it is not a permanent disable of the principle. 🟢 CONFIRMADO — confirmed by maintainer 2026-05-15 [Revisão: 🔴→🟢]
- **Multiple ambiguities at once:** LLM should surface all of them in a single message rather than asking sequentially. 🟡 INFERIDO — implied by efficiency but not stated.
