## Purpose

🟢 **CONFIRMADO** — This module ships a single Claude Code skill containing four behavioral principles designed to counteract well-known LLM coding failure modes. When installed (via plugin or CLAUDE.md), the principles become active constraints applied to every interaction.

The stated motivation (from README.md) maps directly to Andrej Karpathy's public observation that LLMs:
- Make wrong assumptions silently
- Overcomplicate code and APIs
- Make orthogonal edits to code they don't understand
- Lack verifiable success criteria

## Principles (Functional Units)

This module's "functions" are behavioral procedures activated by the skill runtime.

### Principle 1 — Think Before Coding

🟢 **CONFIRMADO** — Defined in `SKILL.md:5–12`, `CLAUDE.md:9–17`

**Intent:** Prevent silent assumptions, hidden confusion, and missing tradeoffs before any implementation begins.

**Trigger:** Any implementation request received by the LLM.

**Rules (4):**

| Rule ID | Rule | Guard condition |
|---------|------|----------------|
| TBC-01 | State assumptions explicitly | `IF uncertain THEN ask, NOT guess` |
| TBC-02 | Present multiple interpretations | `IF ambiguous THEN present all, NOT pick silently` |
| TBC-03 | Push back when simpler approach exists | `IF simpler path exists THEN say so` |
| TBC-04 | Stop when confused | `IF unclear THEN name the confusion AND ask` |

**Outputs:** Either clarifying questions before coding, or an explicit statement of assumptions before proceeding.

---

### Principle 2 — Simplicity First

🟢 **CONFIRMADO** — Defined in `SKILL.md:14–23`, `CLAUDE.md:19–28`

**Intent:** Enforce minimum-viable code. Combat the LLM tendency toward over-engineering.

**Trigger:** Any code generation or design decision.

**Rules (5 + 1 test):**

| Rule ID | Rule |
|---------|------|
| SF-01 | No features beyond what was asked |
| SF-02 | No abstractions for single-use code |
| SF-03 | No "flexibility" or "configurability" that wasn't requested |
| SF-04 | No error handling for impossible scenarios |
| SF-05 | If 200 lines could be 50, rewrite it |

**Embedded heuristic test (🟢 CONFIRMADO):**
> "Would a senior engineer say this is overcomplicated? If yes, simplify."

---

### Principle 3 — Surgical Changes

🟢 **CONFIRMADO** — Defined in `SKILL.md:25–38`, `CLAUDE.md:30–44`

**Intent:** Minimize diff blast radius. Every changed line must trace to the user's request.

**Trigger:** Any edit to existing code.

**Two operating modes:**

**Mode A — When editing existing code (4 rules):**

| Rule ID | Rule |
|---------|------|
| SC-01 | Don't "improve" adjacent code, comments, or formatting |
| SC-02 | Don't refactor things that aren't broken |
| SC-03 | Match existing style, even if you'd do it differently |
| SC-04 | If unrelated dead code is noticed, mention it — don't delete it |

**Mode B — When own changes create orphans (2 rules):**

| Rule ID | Rule |
|---------|------|
| SC-05 | Remove imports/variables/functions made unused by YOUR changes |
| SC-06 | Don't remove pre-existing dead code unless asked |

**Embedded test (🟢 CONFIRMADO):**
> "Every changed line should trace directly to the user's request."

---

### Principle 4 — Goal-Driven Execution

🟢 **CONFIRMADO** — Defined in `SKILL.md:40–55`, `CLAUDE.md:46–63`

**Intent:** Transform imperative tasks into declarative, verifiable goals to enable autonomous LLM loops.

**Trigger:** Any multi-step or ambiguous task.

**Core transformation pattern:**

| Input (imperative) | Output (verifiable goal) |
|--------------------|--------------------------|
| "Add validation" | "Write tests for invalid inputs, then make them pass" |
| "Fix the bug" | "Write a test that reproduces it, then make it pass" |
| "Refactor X" | "Ensure tests pass before and after" |

**Multi-step plan pattern (🟢 CONFIRMADO):**
```
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

**Key insight (🟢 CONFIRMADO, quoted from README.md):**
> "LLMs are exceptionally good at looping until they meet specific goals... Don't tell it what to do, give it success criteria and watch it go."

## Global Tradeoff Note

🟢 **CONFIRMADO** — Stated explicitly in `SKILL.md:1–3` and `README.md`:

> "These guidelines bias toward caution over speed. For trivial tasks, use judgment."

This is a meta-rule: the 4 principles are not universally applied mechanically — trivial tasks (simple typo fixes, obvious one-liners) bypass the full rigor. No formal definition of "trivial" is given; it is left to LLM judgment.
