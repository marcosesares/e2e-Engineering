## Interface

**Input:** User intent (unrefined idea for feature, fix, refactoring, or architectural change)  
**Output:** Approved design specification file (`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`) + Git commit

**Process Outputs at Each Stage:**

| Stage | Input to Next | Audience | Format |
|-------|-------------|----------|--------|
| 1. Context exploration | Project understanding (files read, recent commits noted) | Internal (agent state) | Chat message |
| 2. Visual companion offer | User consent to browser-based visuals | User | Dedicated message (if applicable) |
| 3–5. Clarifying questions & approaches | Refined intent, trade-offs documented | User | Chat exchange |
| 6. Design sections | Iterative design document | User | Chat message (each section) |
| 7. Spec written | Candidate design file on disk | Agent (internal review) | Markdown file |
| 8. Spec self-review | Fixed specification | User | Chat message (summary of fixes) |
| 9. User approval & transition | Approved spec + invoke writing-plans | writing-plans skill | Git commit + skill invocation |

## Main Flow

```
1. Explore Project Context
   → Read files, docs, recent commits
   → Understand current state & constraints
   → Detect scope early (multi-subsystem flag?)

2a. Large Project? → Decompose & Re-enter
   (If multi-subsystem project detected)
   → Identify independent pieces & ordering
   → Suggest decomposition to user
   → Brainstorm first sub-project only
   → Return to step 2b after first cycle

2b. Offer Visual Companion (if needed)
   → Standalone message with consent request
   → Wait for user response
   → (If declined, skip visual companion)

3. Ask Clarifying Questions (Sequential, One-per-Message)
   → Understand: purpose, constraints, success criteria
   → One question per response; wait for reply
   → Build refined intent iteratively

4. Propose 2–3 Alternative Approaches
   → Each approach: description, trade-offs, implementation effort
   → Recommendation provided with reasoning
   → Let user select preferred approach

5. Present Design (Incremental Sections with Approval Gates)
   → Split design logically (architecture, components, data flow, etc.)
   → Present section-by-section
   → Ask approval after each section; revise if needed
   → Repeat until design is complete & approved

6. Write Design Specification
   → File path: `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
   → Use design sections approved in step 5
   → Commit to git

7. Spec Self-Review (Inline Fixes)
   → Scan for placeholders ("TBD", "TODO", incomplete sections)
   → Check for contradictions between sections
   → Check for ambiguity (pick one interpretation, make explicit)
   → Fix issues inline; no separate review loop

8. User Reviews Written Spec
   → Request explicit user approval of spec file
   → If changes requested, edit & re-review
   → Continue once user approves

9. Invoke writing-plans Skill
   → Transition to implementation planning
   → ONLY writing-plans is invoked; no other skill
```

## Alternative Flows

### Over-scoped Project Detection
**Trigger:** User describes multiple independent subsystems (e.g., "build a platform with chat, file storage, billing, and analytics")

**Flow:**
1. Agent detects multi-subsystem scope before detailed questions
2. Agent explains need for decomposition
3. Agent suggests independent pieces and implementation order
4. Agent asks which sub-project to brainstorm first
5. Normal brainstorming flow continues for that sub-project only
6. (After first cycle, loop back for next sub-project with `/brainstorming` invocation)

**Evidence:** `SKILL.md` line 74: "If the project is too large for a single spec, help the user decompose..."

---

### Hard Gate Violation Attempt
**Trigger:** User or agent attempts to invoke implementation skill or write code before design is approved

**Flow:**
1. Agent detects attempt (direct code invocation, scaffold tool call, or implementation skill trigger)
2. Agent rejects with explicit error message, e.g.:  
   > "I can't invoke `<implementation-skill>` yet. We have a design presented, but it hasn't been approved by you. Please review the design in `<path>` and confirm approval before we move to implementation."
3. Agent references the unapproved design document
4. Agent prevents code execution (no fallback to implementation; hard rejection)

**Evidence:** `SKILL.md` lines 12–14 (HARD-GATE block)

---

### Visual Companion Accepted
**Trigger:** User consents to visual content during brainstorming

**Flow:**
1. Agent offers companion in standalone message
2. User approves
3. For each question: agent decides whether visual or text explanation is better
4. If visual content is needed, agent uses browser to show mockups, wireframes, diagrams, layout comparisons
5. If text content (requirements questions, conceptual choices), agent uses terminal
6. Agent references `skills/brainstorming/visual-companion.md` for detailed guide

**Evidence:** `SKILL.md` lines 146–164 (Visual Companion section)

---

### Visual Companion Declined
**Trigger:** User declines browser companion offer

**Flow:**
1. Agent acknowledges decline
2. Continues brainstorming with terminal-only explanations
3. All visual content (mockups, diagrams) described in text or ASCII art

---

### Spec Self-Review Detects Unresolved Issues
**Trigger:** Self-review identifies placeholders, contradictions, or ambiguity

**Flow:**
1. Agent fixes issues inline (replaces TBD with specific value, clarifies ambiguity, removes contradictions)
2. Agent notes fixes in chat message to user
3. Agent re-presents spec file to user for approval
4. Continue with step 8 (user approval)

**Evidence:** `SKILL.md` lines 116–124 (Spec Self-Review section)

## Internal State

**Brainstorming is stateless (session-local).** The skill does not maintain persistent internal state across sessions. All state is captured in:

1. **Chat context** — User responses, agent reasoning, clarifying question history
2. **Design spec file** (`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`) — The durable output
3. **Git commit** — Proof of design approval and sign-off

---

## Observability

**Chat-based feedback (user-facing):**
- Agent summarizes progress at each stage (e.g., "1. Project context understood. 2. Scope appears manageable. Moving to clarifying questions.")
- Agent explicitly states when transitioning stages (e.g., "Now that I understand the intent, here are 3 approaches...")
- Agent confirms user approval before moving forward (e.g., "Design approved. Writing spec now...")

**Audit trail (code-level):**
- Design spec file committed to git with explicit commit message
- Commit message references the design topic and approval date
- User's approval confirmed in chat (transcript available)

**Red flags during execution:**
- Multi-subsystem scope detected → Agent flags immediately, does not proceed
- Hard gate violation attempt → Agent rejects explicitly
- Unresolved placeholders in self-review → Agent reports and fixes inline
- User requests changes to spec → Agent tracks revisions

## Risks and Gaps

| Risk | Impact | Mitigation | Confidence |
|------|--------|-----------|-----------|
| Agent over-interprets scope | Large project may appear manageable, leading to incomplete brainstorming | Agent must flag multi-subsystem projects early, before detailed questions | 🟢 |
| User skips spec review | Design may proceed with unstated ambiguities or errors | Agent explicitly requests user review; blocks transition until approval | 🟢 |
| Hard gate bypassed by implementation-focused user | Code written without design; violates workflow | Agent rejects implementation skill invocation; error message explicit | 🟢 |
| Visual companion creates token bloat | Expensive rendering; user may not benefit | User consent required; agent decides per-question if visual is helpful | 🟢 |
| Spec self-review misses subtle contradictions | Contradictory design accepted | Agent performs manual scan; user review gate provides safety net | 🟡 |
| Project decomposition not clear to user | User continues with multi-subsystem brainstorming unnecessarily | Agent must explain decomposition clearly; offer to break into separate cycles | 🟡 |
