## Overview

Brainstorming is a mandatory workflow skill that guides agents through structured ideation and design specification before any implementation. It enforces a hard gate: **no code, no scaffolding, no implementation action may occur until a design has been presented to the user and approved.** This applies to all projects, regardless of perceived simplicity.

The skill transforms vague ideas into fully specified designs through collaborative dialogue, exploration of project context, clarifying questions, trade-off analysis, and iterative design validation.

---

## Responsibilities

1. Explore project context (files, docs, recent commits) to understand current state
2. Assess scope and flag large projects requiring decomposition
3. Offer visual companion (browser-based mockups/diagrams) if needed
4. Ask clarifying questions one at a time to refine user intent
5. Propose 2-3 alternative approaches with trade-offs and recommendation
6. Present design in sections scaled to complexity, get approval incrementally
7. Write validated design to spec file (`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`)
8. Perform spec self-review (fix placeholders, contradictions, ambiguity inline)
9. Obtain user approval of written spec before invoking writing-plans

---

## Business Rules

- **Hard Gate (Non-negotiable):** No implementation skill, code, or scaffolding may be invoked or started until user has approved a written design. 🟢
- **Anti-Pattern Rejection:** "This is too simple to need a design" is explicitly forbidden. Every project, no matter how simple, requires a design doc and approval. 🟢
- **One Question Per Message:** Clarifying questions must be asked sequentially, one per response, to avoid overwhelming the user. 🟢
- **Approaches Always Presented:** Every substantial design choice must include 2-3 alternative approaches with documented trade-offs before one is selected. 🟢
- **Visual Companion (Opt-in):** If visual content is anticipated, offer the browser companion in a dedicated message (no other content). The user must consent before it is used. 🟢
- **Terminal vs. Browser Decision:** For each question, decide whether text or visual explanation serves the user better. Visual is only for content that truly benefits from pictures. 🟢
- **Terminal State:** The ONLY skill invoked after brainstorming is `writing-plans`. No frontend-design, mcp-builder, or other implementation skill. 🟢
- **Scope Decomposition:** Projects spanning multiple independent subsystems must be decomposed into sub-projects first, each with its own brainstorming cycle. 🟡
- **Design Sections Iterative:** Design is presented section-by-section with approval checks, not as a single monolithic document. 🟢
- **Spec Stored in Git:** Design document must be committed to git as part of the design phase. 🟢

## Acceptance Criteria

```gherkin
Scenario: Happy Path — Simple Feature Design
  Given a user has a simple feature idea
  And the project context is understood
  When the agent presents a 3-section design (architecture, components, testing)
  And the user approves each section
  And the agent writes and self-reviews the spec
  And the user approves the written spec
  Then the agent invokes writing-plans skill
  And no implementation code has been written
  And the design spec is committed to git

Scenario: Over-scoped Project Detection
  Given a user describes a multi-subsystem project (e.g., chat + storage + billing + analytics)
  When the agent assesses scope before asking detailed questions
  Then the agent flags that the project needs decomposition
  And suggests independent sub-projects and ordering
  And asks which sub-project to brainstorm first
  And brainstorms only the selected sub-project

Scenario: Hard Gate Violation Attempt
  Given a design has been presented but not yet approved
  When the user or agent attempts to invoke an implementation skill or write code
  Then the agent rejects the action with explicit error
  And references the unapproved design
  And prevents code execution

Scenario: Visual Companion Consent Required
  Given questions will involve visual content (mockups, layouts, diagrams)
  When the agent anticipates visual questions
  Then the agent offers the browser companion in a standalone message
  And waits for explicit user consent
  And only uses visual content after consent

Scenario: Failure — Incomplete Spec Self-Review
  Given a design spec has been written
  When the agent fails to detect a placeholder (e.g., "TBD", incomplete section)
  Then the user points out the issue
  And the agent fixes it inline
  And the spec is re-approved before moving forward
```
