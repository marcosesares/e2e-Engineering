## Overview

Cavecrew is a decision-making framework that guides agents in choosing between spawning compressed-output subagents (`cavecrew-investigator`, `cavecrew-builder`, `cavecrew-reviewer`) versus using vanilla inline methods (`Explore`, main-thread editing, `Code Reviewer`). The primary value proposition is **token efficiency**: subagent output in caveman format is ~60% smaller than vanilla prose, significantly extending main-thread context budget across long sessions.

Cavecrew is NOT a new capability — it provides the same functionality as vanilla agents (code location, file editing, diff review) but returns results in a compressed format suitable for injection back into main-thread context.

---

## Responsibilities

1. **Decision-making:** Determine which subagent (investigator, builder, reviewer) to spawn based on task scope and output preference
2. **Delegation:** Spawn appropriate subagent with compressed-output contract
3. **Chaining:** Orchestrate multi-step patterns (locate → fix → verify) across multiple subagents
4. **Output interpretation:** Parse caveman-formatted results and use them for downstream decisions
5. **Fallback handling:** Recognize terminal responses (`too-big`, `needs-confirm`, `ambiguous`, `regressed`) and handle gracefully

---

## Business Rules

- **Compression Rationale:** Vanilla Explore returns ~2k tokens; cavecrew-investigator returns ~700 tokens. Across 20 delegations, saves ~26k tokens of main-thread budget. 🟢
- **Output Contract:** Each subagent returns structured output (file:line table, diff summary, review findings), never prose. Safe to grep with patterns like `path:\d+`. 🟢
- **Token Efficiency Test:** Use cavecrew if output should be ~1/3 the size; use vanilla if prose/explanation is needed. 🟢
- **Investigator Use:** "Where is X defined / what calls Y / list uses of Z" → `cavecrew-investigator`. 🟢
- **Builder Use:** Surgical edit of ≤2 files with obvious scope → `cavecrew-builder`. 🟢
- **Builder Precondition:** File paths must be known before invoking builder; skip if discovery needed. 🟢
- **Reviewer Use:** Audit diff, branch, or file for bugs → `cavecrew-reviewer`. 🟢
- **Chain Pattern:** Locate (investigator) → fix (builder) → verify (reviewer) is most common. 🟢
- **No Prose from Cavecrew:** Output is structured, sometimes terse. If human will read directly, paraphrase. 🟢
- **Terminal Responses:** Builder can return `too-big`, `needs-confirm`, `ambiguous`, `regressed` instead of a change. These end the operation and require main-thread decision. 🟢
- **Parallel Scouting:** Multiple `cavecrew-investigator` calls can be spawned in one message for different search angles (defs, callers, tests). 🟢
- **No Deep Refactoring with Builder:** ≥5 files or cross-cutting changes should use main thread or feature-dev skill, not builder (builder will return `too-big`). 🟢

## Acceptance Criteria

```gherkin
Scenario: Locate code with cavecrew-investigator
  Given agent needs to find "where is X defined"
  When cavecrew-investigator is spawned
  Then output is file:line table (3–10 entries) in caveman format
  And output can be parsed with regex `path:\d+`
  And main thread uses results to pick 1–2 sites for builder

Scenario: Surgical edit with cavecrew-builder
  Given agent knows exact file paths to edit
  And scope is ≤2 files
  When cavecrew-builder is spawned with paths and change description
  Then builder returns change summary (<10 words) or terminal response
  And if "verified: re-read OK", change is ready
  And if terminal response, operation halted

Scenario: Diff review with cavecrew-reviewer
  Given agent has a branch/PR to audit
  When cavecrew-reviewer is spawned
  Then output is line-by-line findings: "path:line: 🔴 severity: problem. fix."
  And findings sorted by file → line ascending
  And totals (N🔴 N🟡 N🔵 N❓) included

Scenario: Token budget extension (verify context lasts longer)
  Given session with 20 subagent delegations
  And vanilla agents return ~2k tokens per result
  When cavecrew agents return ~700 tokens per result
  Then main context budget saved ~26k tokens
  And session completes without context exhaustion

Scenario: Builder detects over-scoped task
  Given builder invoked for ≥5 files or cross-cutting refactor
  When builder scope exceeds surgical-edit threshold
  Then builder returns "too-big." (terminal response)
  And main thread recognizes terminal response
  And operation escalated to main thread or feature-dev skill

Scenario: Parallel scout with multiple investigators
  Given task requires investigation from multiple angles (defs, callers, tests)
  When 2–3 cavecrew-investigator calls spawned in one message
  Then all investigators execute in parallel
  And results aggregated in main thread
```

| Recognize terminal responses from builder | **Must** | Terminal responses halt operation; must be detected to avoid continuing with invalid state |
| Token efficiency (60% compression) | **Should** | Core value prop; extends context budget significantly; not critical to base functionality but main motivation for cavecrew |

| Parallel scout (multiple investigators) | **Could** | Performance optimization; useful for broad investigation but not required |
| Paraphrase caveman output for humans | **Could** | Improves readability; not critical if output is for agent-to-agent communication |
