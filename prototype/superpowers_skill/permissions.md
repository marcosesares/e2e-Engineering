## Overview

Superpowers defines three principal actors: **Agents** (AI implementations), **Humans** (users, maintainers), and **Harnessers** (IDE/CLI platforms). Permissions are coarse-grained (can vs. cannot) and hard-enforced via skill gates, not soft recommendations.

---

## Actors & Roles

### Agent Types

| Actor | Role | Trust Level | Authority |
|---|---|---|---|
| **Implementer Agent** | Write code, run tests, create branches | Trusted within gates | TDD cycle, debugging phases mandatory |
| **Code Reviewer Agent** | Review code quality, approve/reject | Independent, adversarial | Can block merge without justification |
| **Spec Reviewer Agent** | Review design document clarity | Independent, adversarial | Can reject spec without rewrite authority |
| **Subagent** | Execute single task from plan | Restricted (temporary) | Cannot create branches, cannot push, cannot merge |
| **Caveman Mode Agent** | Compress documentation, reduce tokens | Trusted (no code impact) | Can rewrite prose; cannot modify skill semantics |

### Human Roles

| Role | Decision Authority | Approval Gate |
|---|---|---|
| **Author/Implementer** | Propose implementation, choose harness | None (self-directed) |
| **Reviewer** | Approve design before brainstorming completes | Brainstorming hard gate |
| **Maintainer** | Accept/reject PRs, merge to main, release | High bar (94% rejection rate) |
| **Product Owner** | Define features, approve plan scope | Pre-implementation (before writing-plans) |

### Harness/Platform Roles

| Platform | Capability | Authority | Constraint |
|---|---|---|---|
| **Claude Code** | Full git (branch, push, PR), subagent spawning | Unrestricted | Agent-behavior gates still apply |
| **Codex App** | Managed worktrees, sandbox isolation, "Create branch" UI | Restricted (workspace-write, then detects) | Agent cannot push/branch in sandbox; handoff payload instead |
| **Codex CLI** | Full git (no managed worktrees) | Unrestricted | Agent uses `git worktree add` fallback |
| **Gemini CLI** | Full git (platform support TBD) | Unrestricted | Agent-behavior gates apply |
| **Custom IDE/CLI** | Depends on bootstrap autoload | Depends on impl. | Real integration iff brainstorming auto-triggers |

## Permission Enforcement (Hard Gates)

### TDD Cycle — Code Before Test Prohibition

**Rule:** No production code without failing test first.

**Enforcement:**
- Skill red flag: "Code before test? Delete & restart."
- Non-negotiable without human consent.
- Applies to all agents (Implementer, Subagent).

**Bypass Condition:** Human explicitly says "skip TDD for this task" or "write code first, test after" (rare, requires written justification).

🟢 **Confidence:** CONFIRMED (skill enforcement explicit)

---

### Brainstorming Design Gate

**Rule:** No implementation until design doc approved by human.

**Enforcement:**
- Brainstorming skill Step 8: user reads and approves spec before writing-plans invoked.
- Spec document path validation (docs/superpowers/specs/YYYY-MM-DD-*.md).
- Hard gate; cannot be skipped.

**Bypass Condition:** Human says "proceed without written spec" (very rare; creates technical debt).

🟢 **Confidence:** CONFIRMED (brainstorming skill structure)

---

### Debugging Phase Sequence

**Rule:** Phases 1, 2, 3, 4 must execute in order. Cannot skip Phase 1 investigation or jump to Phase 4 fix.

**Enforcement:**
- Systematic debugging skill lists 4 mandatory phases.
- Red flags for "quick fix" promises, guessing, multiple changes.
- Phase 4 escalation: if Fix #3+ fails, escalate to human.

**Bypass Condition:** Human says "this is a known fix, apply directly" (requires explicit context of why skipping is safe).

🟢 **Confidence:** CONFIRMED (skill defines phases, red flags explicit)

---

### Worktree Isolation Consent

**Rule:** If user has not stated worktree preference, agent must ask consent before creating.

**Enforcement:**
- using-git-worktrees Step 0: consent request if not already stated in user context.
- Respects prior consent declarations (avoid re-asking).

**Bypass Condition:** User already approved in session or instructions state preference.

🟢 **Confidence:** CONFIRMED (using-git-worktrees skill Step 0)

### Subagent Authority Boundary

**Rule:** Subagents cannot push, merge, or create branches (cannot directly impact main repo).

**Enforcement:**
- Subagent output reviewed by parent agent before any merge decision.
- Finishing skill (invoked by subagent) presents options but parent/human decides.
- File writes limited to isolated worktree; no main repo modification.

**Exception:** Parent agent (Implementer role) can authorize subagent output and push.

🟡 **Confidence:** INFERIDO (subagent-driven-development skill dispatch; exact boundaries inferred)

---

### Skill Content Modification Requirement

**Rule:** Modifying skill content (process definitions, red flags, gates) requires before/after eval evidence.

**Enforcement:**
- CLAUDE.md § "Skill Changes Require Evaluation" — must run adversarial testing.
- Must show eval results in PR description.
- Cannot modify behavior-shaping content (red flags, rationalization lists) casually.
- Maintainer review required; high bar for acceptance.

**Bypass Condition:** Bug fix with clear rationale (e.g., broken link, typo, factual error).

🟢 **Confidence:** CONFIRMED (CLAUDE.md explicit)

---

### Third-Party Dependency Prohibition

**Rule:** Core superpowers adds zero external tool dependencies.

**Enforcement:**
- CLAUDE.md explicitly rejects PR with external deps (unless new harness support).
- Skill content is markdown; zero implicit deps.
- PR requirement: "no new dependencies."

**Exception:** New harness integration may add platform-native tool mapping (e.g., Codex plugin).

🟢 **Confidence:** CONFIRMED (CLAUDE.md explicit, no external deps in codebase)

---

### Domain-Specific Skill Prohibition

**Rule:** Skills for specific domains (portfolio building, blockchain, game dev) do NOT belong in core.

**Enforcement:**
- CLAUDE.md § "Domain-Specific Skills" rejects these in PR.
- Decision gate: "Would this be useful to someone working on a completely different kind of project?"
- If NO → belongs in standalone plugin, not core.

**Corollary:** All 21 core skills tested across Go, JavaScript, Python, systems projects (multi-domain validation).

🟢 **Confidence:** CONFIRMED (CLAUDE.md explicit, skill scope verified by Archaeologist)

---

### Agent Fabrication & Slop Rejection

**Rule:** PRs with invented claims, hallucinated functionality, or fabricated problem descriptions are closed immediately.

**Enforcement:**
- CLAUDE.md § "If You Are an AI Agent" framing: "Your job is to protect your human partner from embarrassment."
- 94% rejection rate partly due to this filter.
- No detailed review given; closed with comment "slop" or "fabricated."

**Non-Bypass:** This is absolute; no exception path.

🟢 **Confidence:** CONFIRMED (CLAUDE.md explicit, git history shows swift rejections)

## Role Transition Rules

### Session Start → Implementer Role

```
User message → use-superpowers bootstrap → brainstorming (Planner role)
  → design doc written
  → writing-plans (Planner role)
  → plan written
  → execution model chosen
    ├─→ subagent-driven-development (Parent Implementer role)
    │   └─→ spawn Subagent Implementer (restricted)
    │
    └─→ executing-plans (Implementer role in same session)
```

### Code Review Trigger

```
Implementer completes code → requesting-code-review
  → spawn Code Reviewer Subagent (independent, adversarial)
    → Review output (approve/reject)
      ├─→ APPROVED: can merge
      └─→ REJECTED: return to Implementer for rework
```

### Debugging Trigger

```
Test failure detected → systematic-debugging
  → Implementer switches to Debugger role
    → [Phases 1-4 INVESTIGATION]
      → Fix implemented
        → Tests verify
          → return to Implementer role
```

## Enforcement Mechanisms

### Skill-Level Gates (Process Enforcement)

| Gate | Trigger | Consequence |
|---|---|---|
| TDD Red Phase | Test not failing | Skip to GREEN prohibited |
| Brainstorming Approval | Design doc not approved | Cannot invoke writing-plans |
| Debugging Phase Sequence | Phase skipped | Red flag triggered, escalation requested |
| Worktree Consent | No prior preference stated | Ask before creating |

### PR-Level Rejection (Quality Gate)

| Violation | Action | Outcome |
|---|---|---|
| Incomplete PR template | Close without review | Fast signal; easy to retry |
| Bulk/spray-and-pray PR | Close with comment | Clear boundary |
| Speculative fix (no real problem) | Close without review | Filters noise |
| Fabricated content | Close immediately | Zero tolerance |
| Third-party deps (no harness exception) | Close without review | Zero-dep enforcement |

### Eval Evidence Requirement (Behavior Change)

| Change Type | Evidence Required | Approval Gate |
|---|---|---|
| Skill content reword | Before/after eval across 3+ scenarios | Maintainer review |
| Red flag modification | Eval showing improved agent behavior | Maintainer review + adversarial test |
| Process gate removal | Evidence of safety (rare) | Very high bar, likely rejected |
| TDD/debugging phase change | Extensive evals showing no regression | Near-impossible bar |

## Permission Boundary Summary

### What Agents CAN Do (With Gates)

✅ Write code (following TDD)
✅ Create tests (TDD red phase)
✅ Debug root causes (following 4 phases)
✅ Review code (as Code Reviewer role)
✅ Create branches (with consent or in isolated worktree)
✅ Commit work (to feature branch)
✅ Request code review (submitting for approval)
✅ Suggest PR descriptions (human approves before creation)
✅ Reword non-skill prose (documentation, comments)
✅ Use caveman compression (on non-code prose)

### What Agents CANNOT Do

❌ Merge to main (human only)
❌ Push to remote without review (human decision)
❌ Skip TDD phases (hard gate)
❌ Skip debugging phases (hard gate, except human override)
❌ Modify skill content (requires eval evidence)
❌ Modify red flags / process gates (intentionally tuned)
❌ Add third-party dependencies (zero-dependency core)
❌ Create domain-specific skills in core (plugin-only)
❌ Open bulk PRs (one problem per PR)
❌ Open PR without showing human the diff first (approval gate)
