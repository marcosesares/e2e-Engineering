## Overview

Superpowers defines six primary state machines: Test Cycle (TDD), Debugging Investigation, Plan Execution, Skill Lifecycle, Worktree Management, and Agent Role States. Each is non-hierarchical but shares trigger conditions and guard clauses.

---

## 1. TDD Cycle (Test-Driven Development)

### States

| State | Description | Entry Condition | Exit Condition |
|-------|---|---|---|
| **IDLE** | Waiting for feature requirement | Session starts or feature assigned | Agent chooses to implement feature |
| **RED** | Write failing test | Feature assigned to agent | Test fails (non-trivially, feature missing) |
| **VERIFY_RED** | Confirm test failure | RED test written | Failure confirmed as expected |
| **GREEN** | Implement minimal code | VERIFY_RED succeeds | Code passes RED test |
| **VERIFY_GREEN** | Confirm test passes | GREEN code written | Test passes, no new failures |
| **REFACTOR** | Clean code without breaking tests | VERIFY_GREEN succeeds | Code is cleaner, tests still pass |
| **VERIFY_REFACTOR** | Final verification | REFACTOR changes committed | All tests pass, no regressions |
| **DONE** | Feature complete | VERIFY_REFACTOR succeeds | Code merged or archived |

### Transitions

```
IDLE --[feature assigned]--> RED
RED --[test written]--> VERIFY_RED
VERIFY_RED --[fails?]--> {
  PASS: GREEN
  FAIL: {error, return to RED, fix test}
}
GREEN --[code written]--> VERIFY_GREEN
VERIFY_GREEN --[passes?]--> {
  PASS: REFACTOR
  FAIL: {return to GREEN, fix code}
}
REFACTOR --[cleanup done]--> VERIFY_REFACTOR
VERIFY_REFACTOR --[all pass?]--> {
  PASS: DONE
  FAIL: {return to REFACTOR, fix breakage}
}
DONE --[end session/new feature]--> IDLE
```

### Mandatory Gates

- **RED Phase:** Must write test BEFORE production code. Violation: "Code before test? Delete & restart."
- **VERIFY_RED:** Test MUST fail. If test passes immediately, return to RED (test was trivial or incomplete).
- **VERIFY_GREEN:** No regressions. New failures = hypothesis wrong.
- **REFACTOR:** Code must be cleaner (duplication removed, names improved). "While I'm here" additions prohibited.

### Architecture Escalation

**Trigger:** Fix #3 fails (same feature, third attempted fix in different location).

**Action:** Escalate to human — signal of architectural problem, not hypothesis problem.

🟢 **Confidence:** CONFIRMED (detailed in test-driven-development skill)

## 2. Debugging Investigation (Systematic Debugging)

### States

| State | Description | Entry Condition | Exit Condition |
|-------|---|---|---|
| **IDLE** | Tests passing | Session running | Test failure or error observed |
| **INVESTIGATION** | Phase 1: Gather evidence | Error/failure occurs | Root cause hypothesis formed |
| **PATTERN_ANALYSIS** | Phase 2: Find working example | Investigation complete | Working code pattern identified |
| **HYPOTHESIS** | Phase 3: Form single hypothesis | Pattern analyzed | Hypothesis ready to test |
| **TESTING** | Phase 3b: Test hypothesis minimally | Hypothesis stated | Test result confirmed |
| **FIX_IMPLEMENTATION** | Phase 4: Implement root-cause fix | Hypothesis confirmed | Fix complete, test passes |
| **VERIFICATION** | Verify fix + no regressions | Fix implemented | All tests pass, no new failures |
| **DONE** | Bug fixed | Verification succeeds | Feature works correctly |

### Investigation Phase (Phase 1) Sub-States

```
INVESTIGATION
├── READ_ERROR_MESSAGE
│   └── Extract: file:line, error code, type, message
├── REPRODUCE_CONSISTENTLY
│   └── Execute exact steps, verify failure every time
├── CHECK_RECENT_CHANGES
│   └── git diff, new dependencies, config changes
├── GATHER_EVIDENCE
│   └── Logs at component boundaries, state at each layer
└── TRACE_DATA_FLOW
    └── Backward from symptom to source
```

### Mandatory Sequences

**Phase 1 → Phase 2 → Phase 3 → Phase 4 (linear, no skipping).**

- **Phase 1:** Read error, reproduce, check recent changes, trace data.
- **Phase 2:** Find working example in same codebase, identify differences.
- **Phase 3:** Form SINGLE hypothesis ("X is root cause because Y"), test minimally.
- **Phase 4:** Create failing test case, implement fix, verify.

### Architecture Escalation

**Trigger:** Fix #3+ fails in different location than Fix #1 or #2.

**Signal:** Not hypothesis failure, but architectural problem.

**Action:** Stop. Discuss with human partner before Fix #4.

### Red Flags (prevent deviation)

- ❌ "Issue simple, let me just try this fix" → Phase 1 skipping
- ❌ "Multiple fixes save time" → Phase 3 multi-hypothesis
- ❌ "Emergency, no time for phases" → Hard gate violation
- ❌ "One more attempt" (≥3 failures) → Architecture escalation ignored

🟢 **Confidence:** CONFIRMED (4-phase model explicit, red flags detailed)

## 3. Plan Execution State Machine

### States

| State | Description | Entry Condition | Exit Condition |
|-------|---|---|---|
| **LOAD** | Load plan from file | Execution skill invoked | Plan loaded, read, understood |
| **REVIEW** | Agent reviews plan structure | Plan loaded | Plan makes sense or human intervention |
| **TASK_QUEUE** | Prepare task list | Plan reviewed | First task ready to execute |
| **TASK_EXEC** | Execute single task | Task dequeued | Task steps complete |
| **TASK_VERIFY** | Verify task completion | Task executed | Verification passes or fails |
| **CHECKPOINT** | Save progress | Task verified | Checkpoint written, tracked |
| **NEXT_TASK** | Dequeue next task | Checkpoint saved | Queue empty? DONE : TASK_EXEC |
| **DONE** | All tasks complete | Queue empty | Plan execution finished |

### Execution Model Choice

```
Plan Written
    ├─ Model: subagent-driven-development
    │   ├─ [spawn Task 1 subagent + code reviewer]
    │   ├─ [spawn Task 2 subagent + code reviewer]
    │   ├─ [spawn Task N subagent + code reviewer]
    │   └─ [aggregate results, merge/PR/discard per user]
    │
    └─ Model: executing-plans
        ├─ [same session, sequential execution]
        ├─ [each task loaded, executed, verified in order]
        └─ [checkpoints saved per task]
```

### Checkpoint Format

```json
{
  "phase": "task_exec",
  "completed_tasks": [1, 2, 3],
  "current_task": 4,
  "total_tasks": 10,
  "last_verified_state": "task_3_passed_tests",
  "timestamp": "2026-05-17T14:23:45Z"
}
```

### Failure Recovery

- **Task fails test:** Agent re-runs verification, fixes code, re-verifies (loops in TASK_VERIFY).
- **Task blocked by dependency:** Skip to next unblocked task (priority via `depends_on` field).
- **Session timeout:** Checkpoint saved, user resumes with `/clear` + skill invocation.

🟢 **Confidence:** CONFIRMED (executing-plans skill, plan structure in writing-plans)

## 4. Skill Invocation Lifecycle

### States

| State | Description | Trigger | Next State |
|---|---|---|---|
| **NOT_LOADED** | Skill file not in context | Session starts | CANDIDATE (if bootstrap loads) |
| **CANDIDATE** | Skill name matches trigger condition | User message, auto-trigger, explicit `/skillname` | LOADED |
| **LOADED** | Skill file read, content available | Bootstrap or explicit invocation | ACTIVE |
| **ACTIVE** | Skill guidance used in agent reasoning | Skill in context, agent making decision | EXECUTING |
| **EXECUTING** | Agent following skill process | Agent at phase/step within skill | CHECKPOINT or DONE |
| **CHECKPOINT** | Progress saved between steps | Task boundary or context limit | LOADED (resume) or DONE |
| **DONE** | Skill process completed | Final step executed, goal met | NOT_LOADED (cleanup) |

### Bootstrap Auto-Trigger

**Trigger Mapping:**

| User Message / Condition | Auto-Trigger | Confidence |
|---|---|---|
| "let's make..." | brainstorming | 🟢 CONFIRMED |
| "write code" (no plan) | test-driven-development | 🟢 CONFIRMED |
| Test failure, error | systematic-debugging | 🟢 CONFIRMED |
| "need code review" | requesting-code-review | 🟡 INFERIDO |
| "merge/push/PR" decision | finishing-a-development-branch | 🟡 INFERIDO |
| Multiple independent tasks | dispatching-parallel-agents | 🟡 INFERIDO |

**Non-Triggers (explicit invocation required):**
- caveman mode (user must type `/caveman`, `/caveman-help`, or `caveman mode`)
- caveman-commit (triggered on staging changes OR `/caveman-commit`)

🟡 **Confidence:** INFERIDO (trigger set documented in using-superpowers; exact heuristics not specified)

## 5. Worktree State Machine

### States

| State | Description | Entry Condition | Exit Condition |
|---|---|---|---|
| **MAIN_REPO** | On main branch of primary repo | Session starts | User/skill decision to isolate |
| **DETECT_ISOLATION** | Check if already isolated | Skill invoked | Detection result (isolated or not) |
| **ALREADY_ISOLATED** | In existing worktree | `GIT_DIR != GIT_COMMON` (not submodule) | Proceed to project setup, skip creation |
| **REQUEST_CONSENT** | Ask user if worktree wanted | `GIT_DIR == GIT_COMMON` + no prior consent | User allows / declines / already stated |
| **NATIVE_CREATION** | Use platform-native tool | Consent given, native tool available | Worktree created, checkout successful |
| **GIT_FALLBACK** | Use `git worktree add` | Consent given, no native tool | Worktree created, checkout successful |
| **SETUP_BASELINE** | Install deps, run tests | Worktree created | Tests passing, ready to work |
| **WORK_IN_PROGRESS** | Feature implementation | Setup complete | All tasks done, ready to finish |
| **FINISHING** | Determine merge strategy | Work complete | Branch decision made (merge/PR/keep/discard) |
| **CLEANUP** | Remove worktree (if owned by skill) | Finishing complete | Worktree deleted or preserved per decision |
| **MAIN_REPO** | Return to main checkout | Cleanup complete | Session continues or ends |

### Environment Detection

**Read-Only Detection (no side effects):**

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
BRANCH=$(git branch --show-current)
SUPERPROJECT=$(git rev-parse --show-superproject-working-tree 2>/dev/null)
```

**Decision Matrix:**

| GIT_DIR vs GIT_COMMON | Superproject? | BRANCH | Environment | Action |
|---|---|---|---|---|
| Equal | No | Exists | Normal repo | Create worktree |
| Equal | No | Empty | Detached HEAD (unusual) | Create worktree, warn at finish |
| Different | Yes (non-empty) | Any | Submodule | Proceed (not isolated) |
| Different | No | Exists | Worktree (named branch) | Skip creation, use existing |
| Different | No | Empty | Worktree (detached HEAD, Codex App) | Skip creation, emit handoff payload |

### Cleanup Guard

**Step 5 (Finishing):**

Before running `git worktree remove`:

1. Re-run GIT_DIR detection (do NOT rely on earlier output)
2. If `GIT_DIR != GIT_COMMON`, **skip removal** (harness owns this worktree)
3. Otherwise, remove as normal

**Rationale:** Skill may re-run in different session; detect current state independently.

🟢 **Confidence:** CONFIRMED (codex-app compatibility spec details, using-git-worktrees skill Step 0)

## 6. Agent Role States

### Session Agent Roles

| Role | Trigger | Responsibilities | Constraints |
|---|---|---|---|
| **Implementer** | Task assigned or code requested | Write code, run tests, commit | Must follow TDD; must create test before code |
| **Code Reviewer** | `requesting-code-review` invoked | Review implementation, approve/reject | Must verify tests, no "LGTM without reading" |
| **Spec Reviewer** | Brainstorming design doc written | Verify spec clarity, completeness | Must check for ambiguities, contradictions |
| **Debugger** | Test failure / error occurs | Investigate root cause systematically | Must follow 4-phase debugging process |
| **Planner** | `writing-plans` invoked | Break design into bite-sized tasks | Tasks ≤ 2h each; dependencies explicit |

### Subagent Dispatch

**Pattern:**

```
Parent Agent (Implementer role)
  ├─ Task 1 → Subagent A (temporary Implementer role)
  │   └─ Code Reviewer examines output
  │       └─ Feedback to Subagent A → ITERATE or APPROVED
  │
  ├─ Task 2 → Subagent B (temporary Implementer role)
  │   └─ Code Reviewer examines output
  │       └─ Feedback to Subagent B → ITERATE or APPROVED
  │
  └─ Aggregate results
      └─ Parent merges, creates PR, or discards per user decision
```

**Filesystem Sharing:** Subagents share parent's worktree (confirmed empirical test in Codex app compatibility spec).

**Reviews:** 2-stage (initial rejection triggers iteration, approval gates merge).

🟡 **Confidence:** INFERIDO (subagent-driven-development skill structure; detailed dispatch logic not fully analyzed)

## 7. Caveman Mode Intensity States

### States

| State | Token Efficiency | Prose | Technical | Examples |
|---|---|---|---|---|
| **NORMAL** | 100% (baseline) | Full articles, pleasantries | Exact terminology | Verbose explanations |
| **CAVEMAN_LITE** | ~85% | Drop articles, fillers | Exact terminology | Brief examples, one per topic |
| **CAVEMAN_FULL** | ~50-60% | Drop articles, fillers, pleasantries, hedging | Exact terminology | One-liners, fragments |
| **CAVEMAN_ULTRA** | ~30-40% | Extreme fragments | Exact terminology | Absolute minimum; tone changes |
| **WENYAN_LITE** | ~85% | Remove fluff, keep English sentence structure | Exact terminology | Classical Chinese-inspired compression |
| **WENYAN_FULL** | ~50-60% | Remove fluff, use parallel structures | Exact terminology | Allusions, parallel phrases |
| **WENYAN_ULTRA** | ~30-40% | Minimal parallel structures, poetic | Exact terminology | Extreme stylization |

### Transition Rules

- **Entry:** User types `/caveman [level]`, `caveman mode`, or system auto-detects token efficiency needed
- **Persistence:** Level persists until `/caveman [new-level]`, `stop caveman`, or `normal mode`
- **Boundary:** Code/commits/PRs always written in normal mode (no compression)
- **Auto-Drop:** Drop caveman temporarily for security warnings, irreversible action confirmations, complex sequences

🟡 **Confidence:** INFERIDO (caveman skill defines modes; token injection mechanism unclear)

## 8. Harness Worktree Contract States

### Codex App Specific

| State | Condition | Skill Behavior | User Control |
|---|---|---|---|
| **MANAGED_WORKTREE_WRITE** | Detached HEAD, workspace-write sandbox | Skip creation, commit work, emit handoff | "Create branch" button |
| **MANAGED_WORKTREE_FULL** | Detached HEAD, Full access | Can branch/push via App | Finish 4-option menu OR "Create branch" |
| **NATIVE_BRANCH** | On named branch (App or manual) | Full finishing menu available | Merge/PR/keep/discard decision |
| **LOCAL_REPO** | Normal git repo checkout | Full skill behavior (unchanged) | Standard git workflow |

### Handoff Payload (Codex App Detached HEAD)

**Format:**

```
Implementation complete. All tests passing.
Current HEAD: <commit-sha>

This workspace is externally managed (detached HEAD).
I cannot create branches, push, or open PRs from here.

⚠ These commits are on a detached HEAD. If you do not create a branch,
they may be lost when this workspace is cleaned up.

If your host application provides these controls:
- "Create branch" — to name a branch, then commit/push/PR
- "Hand off to local" — to move changes to your local checkout

Suggested branch name: <ticket/description>
Suggested commit message: <summary>
```
