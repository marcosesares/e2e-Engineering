## EC-01 — "Impossible Scenario" Is Actually Possible

**Description:** BR-08 prohibits error handling for impossible scenarios. But the LLM may lack full system context and incorrectly classify a reachable error state as impossible — omitting handling that is actually needed.

**Risk:** A real runtime error goes unhandled because the LLM decided it "can't happen," causing a silent failure or crash in production.

**Current behavior:** "Impossible" is not formally defined. Left to LLM inference-time judgment. 🔴 LACUNA

**Suggested mitigation:** The rule should be scoped to "scenarios impossible given the current function's contract and caller guarantees, as visible in the code." If the LLM cannot see the caller, it should not omit the guard.

## EC-02 — Legitimate Complexity Exceeds 4× Ratio

**Description:** BR-09 says rewrite if 200 lines could be 50. But some problems genuinely require 200+ lines (e.g., a parser, a state machine, a protocol implementation).

**Risk:** LLM rewrites valid complex code into an undersized version that is wrong or incomplete, driven by the size heuristic.

**Current behavior:** Rule is a heuristic, not an absolute. "Could be 50" is the key qualifier — the rule only fires if the simplification is genuinely available. 🟢

**Suggested clarification:** Add to `SKILL.md`: "only applies when a correct, complete solution exists at the smaller size." Helps LLM distinguish legitimate complexity from bloat.

## EC-03 — User Requests Configurability That Is Actually a Good Idea

**Description:** BR-07 prohibits adding flexibility that wasn't requested. But the user's request has an obvious extension point that, if not added now, will require a breaking change later.

**Risk:** The LLM obeys the rule and ships a rigid implementation, creating tech debt that is more expensive to fix post-deployment.

**Current behavior:** Rule is categorical: no unrequested flexibility. Trade-off acknowledged in preamble ("guidelines bias toward caution"). 🟢

**Suggested handling:** The LLM should implement without the flexibility but mention it: "I've kept this simple per the guidelines — note that adding X later would require [small/large] change." This preserves simplicity while preserving the team's ability to make an informed decision.

## EC-04 — Senior Engineer Test Is Applied Inconsistently

**Description:** BR-10 asks "Would a senior engineer say this is overcomplicated?" This is a subjective self-evaluation. The LLM's notion of "senior engineer" may not match the team's standard.

**Risk:** The heuristic produces inconsistent results across different LLMs, different model versions, or even different conversation contexts.

**Current behavior:** Fully subjective, no calibration. 🟡 INFERIDO

**Suggested mitigation:** Provide 2–3 canonical examples in `EXAMPLES.md` of what "overcomplicated" looks like in this project's context, so the test is anchored to observable patterns rather than abstract judgment.
