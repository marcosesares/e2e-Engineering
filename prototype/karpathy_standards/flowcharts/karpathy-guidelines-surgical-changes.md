## Edit Scope Minimization Procedure

```mermaid
flowchart TD
    A([Edit request received]) --> B{Editing existing code?}

    B -->|Yes| C[Identify the minimal set of lines\nthat trace directly to the request]

    C --> D{Tempted to improve adjacent code?}
    D -->|Yes — adjacent comments, formatting, style| E[DO NOT touch it\nLeave as-is even if suboptimal]
    E --> F{Tempted to refactor something nearby?}
    D -->|No| F

    F -->|Yes — something that 'could be better'| G[DO NOT refactor\nIt's not broken, don't fix it]
    G --> H{Style differs from own preferences?}
    F -->|No| H

    H -->|Yes| I[Match existing style anyway\nDO NOT convert quotes, add type hints, etc.]
    I --> J{Noticed unrelated dead code?}
    H -->|No| J

    J -->|Yes| K[Mention it in response\nDO NOT delete it]
    K --> L[Write the minimal diff]
    J -->|No| L

    L --> M{Did own changes make anything unused?\nimports, variables, functions}
    M -->|Yes — orphan created by THIS change| N[Remove the orphan\nScope: only what THIS change created]
    N --> O{Any pre-existing dead code visible?}
    M -->|No| O

    O -->|Yes| P[Leave it alone\nDO NOT remove unless explicitly asked]
    O -->|No| Q([Apply the traceability test])
    P --> Q

    Q --> R{"Every changed line traces\ndirectly to the user's request?"}
    R -->|No — extra changes crept in| S[Revert extra changes]
    S --> L
    R -->|Yes| T([Change approved])

    B -->|No — new code| U([Apply Simplicity First instead])
```

## Anti-Patterns Detected (from EXAMPLES.md)

| Anti-Pattern | Example | Correct Behavior |
|-------------|---------|-----------------|
| Drive-by refactoring | Fixing empty email crash → also adds username length/alphanumeric validation | Only fix the empty email guard |
| Quote style drift | Changes `'` → `"` while adding logging | Keep original quote style |
| Unsolicited type hints | Adds `file_path: str` while fixing bug | Leave type annotations as-is |
| Unsolicited docstrings | Adds `"""Upload file..."""` while adding logging | Don't add docstrings |
| Boolean refactor | Rewrites `if/else return True/False` to `return response.status_code == 200` while adding logging | Leave boolean pattern as-is |

## Rules Summary

| ID | Rule | Mode | Type |
|----|------|------|------|
| SC-01 | Don't improve adjacent code, comments, formatting | Edit | Hard prohibition |
| SC-02 | Don't refactor things that aren't broken | Edit | Hard prohibition |
| SC-03 | Match existing style | Edit | Hard obligation |
| SC-04 | Mention unrelated dead code — don't delete | Edit | Soft obligation |
| SC-05 | Remove orphans YOUR changes created | Orphan cleanup | Obligation (scoped to own changes) |
| SC-06 | Don't remove pre-existing dead code | Orphan cleanup | Hard prohibition |
| SC-T1 | Traceability test: every changed line traces to request | Verification | Heuristic |
