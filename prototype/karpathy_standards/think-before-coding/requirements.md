## Overview

A behavioral principle that forces the LLM to surface assumptions, ambiguities, and tradeoffs before writing any code. It addresses the failure mode of silent assumption-making, where the LLM picks an interpretation without disclosing it and proceeds with potentially wrong implementation. 🟢

---

## Responsibilities

- Detect uncertainty in an incoming request and surface it explicitly before proceeding 🟢
- Detect multiple valid interpretations and present all of them rather than silently choosing one 🟢
- Identify when a simpler implementation path exists and propose it 🟢
- Stop and name confusion when a request is unclear, rather than guessing 🟢

## Acceptance Criteria

```gherkin
# Happy path — uncertain assumption
Given the user says "add caching to the API"
When the LLM applies Think Before Coding
Then the LLM states "I'm assuming you mean in-memory cache, not Redis — is that correct?" before writing any code

# Happy path — multiple interpretations
Given the user says "refactor the auth module"
When the LLM applies Think Before Coding
Then the LLM presents at least two interpretations ("extract to service" vs "simplify inline logic") and asks which to pursue

# Happy path — simpler approach exists
Given the user requests a 50-line solution to a problem solvable in 5 lines
When the LLM applies Think Before Coding
Then the LLM proposes the simpler approach and explains the tradeoff before implementing

# Failure — genuinely unclear request
Given the user says "fix the thing"
When the LLM applies Think Before Coding
Then the LLM stops, names what is unclear ("which thing?"), and asks — does not guess
```
