## EC-01 — User Explicitly Overrides ("Just Do It")

**Description:** The user says "don't ask questions, just implement it" or "I don't care about assumptions, just start."

**Risk:** BR-01 and BR-04 require asking before coding. If the user disables this, the LLM has conflicting instructions: the skill says ask, the user says don't.

**Current behavior:** Not addressed in `SKILL.md`. 🔴 LACUNA

**Suggested handling:** The override should be honored for the current task but not treated as a permanent disable. LLM should acknowledge: "Proceeding without confirmation — note that assumptions are [X] and [Y]." This satisfies the user's intent while preserving the spirit of the rule.

---

## EC-02 — All Interpretations Are Equally Valid

**Description:** A request has multiple interpretations and the LLM genuinely cannot determine which is more likely or more appropriate.

**Risk:** BR-02 says present all interpretations — but if there are 5+ valid interpretations, presenting all of them creates a poor user experience and may stall the interaction.

**Current behavior:** No cap or prioritization guidance in `SKILL.md`. 🟡 INFERIDO — left to LLM judgment.

**Suggested handling:** Cap at 3 interpretations (most likely, simplest, most conservative). If more exist, group remainder as "other variations" with a brief description.

---

## EC-03 — Request Is Clear But the LLM Is Wrong About It

**Description:** The LLM believes the request is unambiguous and proceeds without asking (BR-04 guard does not trigger). But the LLM's understanding is incorrect.

**Risk:** Confident but wrong implementation — the failure mode Think Before Coding is meant to prevent, but triggered by false certainty rather than ambiguity.

**Current behavior:** Not addressed. This is a fundamental LLM limitation, not a rule failure. 🟡 INFERIDO

**Suggested handling:** Cannot be fully mitigated at the skill level. The best available guard is BR-19 (Goal-Driven Execution): stating success criteria before implementing increases the chance the user catches wrong assumptions early.

---

## EC-04 — Cascading Clarification Loop

**Description:** The user answers a clarifying question, but the answer introduces new ambiguity, triggering another clarifying question, and so on.

**Risk:** The interaction never reaches implementation. User frustration.

**Current behavior:** No cycle-break mechanism in `SKILL.md`. 🔴 LACUNA

**Suggested handling:** After two rounds of clarification on the same topic, the LLM should state its best-guess assumption and proceed: "I'll assume X and proceed — let me know if that's wrong."
