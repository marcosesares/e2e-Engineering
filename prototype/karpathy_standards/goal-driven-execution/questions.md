## Q-GDE-01 — What is the escape condition when the verification loop does not converge? 🔴

**Context:** `goal-driven-execution/design.md:Alternative Flows`; `edge-cases.md:EC-02`

**Spec affected:** [`_reversa_sdd/goal-driven-execution/design.md`]

**Question:** BR-20 says "loop until each verification check passes." There is currently no escape condition. If the LLM cannot find and fix the root cause, it loops indefinitely, making increasingly speculative changes. Should there be a defined escape heuristic — e.g., "after 3 failed attempts on the same check, stop, report the specific failure, and ask the user for guidance"?

**Impact:** Adding an escape heuristic would change BR-20 wording and add one rule to the skill text. Without it, the loop can degrade into scope-creep speculation.

## Q-GDE-02 — How should the LLM handle verify checks that cannot be run mechanically? 🔴

**Context:** `goal-driven-execution/design.md:Alternative Flows`; `edge-cases.md:EC-03`

**Spec affected:** [`_reversa_sdd/goal-driven-execution/design.md`]

**Question:** BR-20 requires looping until verify checks pass, but some checks are not automatable: "verify: the UI looks correct," "verify: the user approves the result," "verify: the deployment succeeds in CI." Should these be designated as human-gate steps in the plan (e.g., `→ verify: [human confirms X]`), with the loop pausing and the LLM informing the user what to check and how to signal completion?

**Impact:** Adds a sub-category of verify step to the skill text ("human-gate steps") and defines the pause behavior. Without this, the LLM either skips non-runnable checks (false success) or halts indefinitely waiting for something it cannot trigger.
