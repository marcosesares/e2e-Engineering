# Goal-Driven Execution

> Spec generated by Reversa Writer · 2026-05-15

---

## Overview

A behavioral principle that transforms imperative tasks ("do X") into declarative, verifiable goals ("X is done when Y passes"). It enables the LLM to loop autonomously until success criteria are met, rather than producing output and waiting for human re-direction. 🟢

---

## Responsibilities

- Transform any imperative task into a verifiable success criterion before implementing 🟢
- For multi-step tasks, state an explicit step-by-step plan with verify steps before starting 🟢
- Loop until each verification check passes — not skip, not approximate 🟢

## Functional Requirements

| ID | Requirement | Priority | Acceptance Criterion |
|----|-------------|----------|---------------------|
| RF-01 | LLM must define success criteria before writing any code for non-trivial tasks | Must | A verifiable goal statement appears before the first code block |
| RF-02 | LLM must produce a step-by-step plan for multi-step tasks, each step with a verify check | Must | Plan lists ≥2 steps, each ending with `→ verify: [check]` |
| RF-03 | LLM must loop until verification passes — not declare success without checking | Must | No "Done" claim without a passed verification step |
| RF-04 | LLM must use the canonical transformation patterns for known task types | Should | "Add validation" → test-first; "Fix bug" → reproduce then fix; "Refactor" → tests before and after |

## Acceptance Criteria

```gherkin
# Happy path — task transformation
Given the user says "add validation to the login form"
When the LLM applies Goal-Driven Execution
Then the LLM states "Done when: tests for invalid email, empty password, and short password all pass" before writing any code

# Happy path — multi-step plan
Given the user says "migrate the users table to the new schema"
When the LLM applies Goal-Driven Execution
Then the LLM produces a plan:
  "1. Write migration script → verify: applies without error
   2. Update ORM model → verify: tests pass
   3. Update API layer → verify: integration test passes"
Before executing any step

# Happy path — verification loop
Given the LLM's first implementation attempt fails the verify check
When Goal-Driven Execution is applied
Then the LLM diagnoses and fixes the failure; does not declare success until the check passes

# Failure — premature success claim
Given the user asks to fix a bug
When the LLM writes a fix but does not run or simulate a verification check
Then BR-20 has been violated — success cannot be claimed without verification
```
