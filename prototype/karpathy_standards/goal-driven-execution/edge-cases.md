## EC-01 — Success Criteria Are Too Vague to Verify

**Description:** The user's task does not provide enough information to form a specific, verifiable success criterion. E.g., "make it better" or "fix performance."

**Risk:** BR-18 requires transforming the task into a verifiable goal — but if the task is too vague, any criterion the LLM invents may not reflect the user's actual intent.

**Current behavior:** Not addressed in `SKILL.md`. The principle assumes the task is specific enough to derive a criterion. 🟡 INFERIDO

**Suggested handling:** If the task is too vague for a verifiable criterion, this is a Think Before Coding trigger (BR-04). The LLM should surface the vagueness and ask for clarification before proceeding to Goal-Driven Execution.

## EC-02 — Verification Loop Does Not Converge

**Description:** BR-20 says loop until the check passes. But if the LLM cannot find and fix the root cause, the loop runs indefinitely.

**Risk:** The LLM loops, making increasingly speculative changes, potentially causing regressions far from the original task scope.

**Current behavior:** No escape condition defined. 🔴 LACUNA

**Suggested mitigation:** Add an escape heuristic: after 3 failed fix attempts on the same check, the LLM should stop, report the specific failure, and ask the user for guidance rather than continuing to speculate.

---

## EC-03 — Verify Check Cannot Be Run Mechanically

**Description:** Some verify conditions are not automatable: "verify: the UI looks correct," "verify: the user approves the result," "verify: the deployment succeeds."

**Risk:** BR-20 says loop until the check passes, but if the check requires human input or external state, the loop must pause and wait — which the skill's wording does not account for.

**Current behavior:** Not addressed. 🔴 LACUNA

**Suggested handling:** Non-runnable checks should be declared as human-gate steps in the plan: `→ verify: [human confirms X]`. The loop pauses at that step and the LLM informs the user what to check and how to signal completion.

---

## EC-04 — Task Is a Single Trivial Step

**Description:** BR-19 specifies the plan format for multi-step tasks. A trivial single-step task may not need a full plan, but BR-18 still requires a success criterion.

**Risk:** Over-applying the plan template to trivial tasks adds friction without benefit, contradicting the preamble's "for trivial tasks, use judgment" guidance.

**Current behavior:** `SKILL.md` says "for multi-step tasks" explicitly — single-step tasks are implicitly exempt from the plan format but not from the criterion. 🟢

**Note:** Rule wording handles this correctly. The only open question is what "trivial" means (shared with EC-01 in `karpathy-guidelines/edge-cases.md`).
