## Overview

A behavioral principle that minimizes the blast radius of every edit. It enforces that the LLM touches only what the user's request directly requires — preventing adjacent improvements, style clean-ups, and opportunistic refactors that inflate diffs, obscure intent, and introduce unintended regressions. 🟢

---

## Responsibilities

- Prevent the LLM from editing code, comments, or formatting adjacent to the requested change 🟢
- Prevent opportunistic refactoring of code not directly involved in the task 🟢
- Enforce style matching with the existing codebase 🟢
- Mandate mentioning (not deleting) unrelated dead code discovered during the task 🟢
- Require cleanup of orphans created by the LLM's own changes 🟢
- Prohibit deletion of pre-existing dead code unless explicitly requested 🟢
- Enforce traceability: every changed line must trace to the user's request 🟢

## Acceptance Criteria

```gherkin
# Happy path — minimal diff
Given the user asks "rename variable x to userId in function login()"
When the LLM applies Surgical Changes
Then only the occurrences of x inside login() are renamed; nothing else changes

# Happy path — orphan cleanup
Given the LLM's change makes import { helper } unused
When the LLM applies Surgical Changes
Then the import is removed as part of the same change

# Failure — adjacent improvement
Given the user asks "fix the null check in processOrder()"
When the LLM also reformats processOrder() and adds a comment to the function above it
Then BR-11 has been violated — those unrequested edits must be reverted

# Failure — pre-existing dead code deletion
Given a function legacyFormat() exists in the file and is unused
When the LLM deletes it while fixing an unrelated bug
Then BR-16 has been violated — the deletion must be undone
```
