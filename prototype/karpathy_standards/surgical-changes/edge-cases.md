## EC-01 — Orphan Is Used in Another File

**Description:** BR-15 says remove symbols made unused by the LLM's own changes. But "unused" may mean unused in the current file while still imported/called by another file in the codebase.

**Risk:** LLM removes a function or variable that is still used elsewhere, introducing a runtime error in another module.

**Current behavior:** BR-15 does not specify cross-file scope. 🔴 LACUNA

**Suggested clarification:** Add to `SKILL.md`: "Remove only symbols unused in the current file AND confirmed unused across the codebase. If cross-file usage cannot be verified, mention the potential orphan instead of deleting."

## EC-02 — Inconsistent Style Within the Same File

**Description:** BR-13 says match existing style. But a file may have mixed styles (e.g., some functions use `camelCase`, others use `snake_case`) due to multiple contributors over time.

**Risk:** "Match existing style" is ambiguous — which style does the LLM match?

**Current behavior:** Not addressed. Left to LLM judgment. 🟡 INFERIDO

**Suggested handling:** Match the style of the immediate surrounding context (the function or block being edited), not the file's majority style.

---

## EC-03 — "Adjacent" Boundary Is Unclear

**Description:** BR-11 says do not improve adjacent code. But "adjacent" is not defined. Is it the next line? The same function? The same file?

**Risk:** LLM either interprets "adjacent" too narrowly (only the changed line) and still modifies nearby code, or too broadly (the entire file) and becomes overly cautious.

**Current behavior:** Not formally defined. 🟡 INFERIDO

**Suggested clarification:** "Adjacent" means any line not in the direct path of the requested change: same function is acceptable context for reading, but only the specific lines that must change are touched.

## EC-04 — Dead Code Is Actually In-Progress Work

**Description:** BR-16 says do not remove pre-existing dead code. BR-14 says mention it. But what appears to be dead code may be code that another developer has in progress and hasn't wired up yet.

**Risk:** Mentioning "dead code" as a suggestion to remove could trigger the user to delete work-in-progress code.

**Current behavior:** Rule correctly defaults to mention (not delete). This is the right behavior. 🟢

**Note:** The rule is correctly designed here. No mitigation needed — mention + non-deletion is the safe default.

---

## EC-05 — Change Requires Touching a Helper Used in Many Places

**Description:** The requested change affects a shared utility function. Modifying it correctly would improve all callers, but those callers were not part of the request.

**Risk:** LLM either modifies callers (violating BR-12) or produces an inconsistent result where the helper changed but callers still use the old signature.

**Current behavior:** Not addressed. 🔴 LACUNA

**Suggested handling:** If a shared helper must change, the LLM should (a) update only the helper as required, (b) update the callers only if the change would break them (required, not optional), and (c) mention any callers not updated that may need review.
