## 1. Module overview

The Setup & Configuration module is the **distribution and bootstrap layer** of the BeckTech.QA.Tools framework. It owns three concerns:

1. **Subtree distribution** — publish the `claude-skills/` tree to a separate `published-skills` branch (root-leveled), and pull it into consuming repos under `<prefix>/.claude/`.
2. **Hook-driven guardrails** — block local edits to subtree-owned files (`PreToolUse`) and notify on staleness (`SessionStart`).
3. **Per-machine / per-repo bootstrap** — install hooks, register MCP servers (Atlassian, ADO, Playwright), set REST credentials (PATs), and stamp/track template versions for drift detection.

Two layers exist:

| Layer | Artifacts | Trigger |
|-------|-----------|---------|
| **Scripts** (5) | `setup/*.ps1` | Invoked manually or by hooks |
| **Skills** (9 setup skills + 2 orchestrators) | `setup-skills/*/SKILL.md`, `skills/onboard/SKILL.md`, `skills/refresh-setup/SKILL.md` | Slash-command (e.g., `/onboard`, `/setup-claude-hooks`) |

The module has **no business logic** — it is pure plumbing. Its correctness criteria are deterministic file presence, version-marker equality, and shell-command exit codes.

## 3. Control flow — scripts

### 3.1 `generate-manifest.ps1` 🟢

**Purpose:** Rebuild `claude-skills/SHARED_MANIFEST` so it reflects the current set of tracked + untracked-but-not-ignored files under `claude-skills/`.

**Input:** none (must run from repo root). **Output:** overwrites `claude-skills/SHARED_MANIFEST`.

**Algorithm (lines 16–50):**

```
1. Assert cwd contains "claude-skills/" (else exit 1).
2. files = git ls-files --cached --others --exclude-standard -- claude-skills
3. Pipeline transform:
   a. Normalize backslashes → forward slashes
   b. Filter: startsWith("claude-skills/")
   c. Filter: Test-Path exists (drops staged-but-deleted)
   d. Strip "claude-skills/" prefix
   e. Filter: != "SHARED_MANIFEST" (manifest does not list itself)
   f. Sort-Unique
4. Join with LF + trailing LF; write via [System.IO.File]::WriteAllText (LF-stable across platforms).
```

**Non-trivial detail (line 31):** uses `--cached --others --exclude-standard` (NOT just `--cached`) so the manifest is complete on a feature branch before `git add`. Maintainer can run, review the diff, then stage everything together.

**Failure modes:**

| Condition | Effect |
|---|---|
| Not at repo root | `Write-Error` + exit 1 |
| `git ls-files` non-zero | `Write-Error` + exit 1 |
| Renamed-but-not-staged file | Dropped by `Test-Path` filter |

### 3.2 `publish-skills.ps1` 🟢

**Purpose:** Push the contents of `claude-skills/` to the `published-skills` branch (root-leveled) so consumers can subtree-pull.

**Input:** `-NonMainOk` switch. **Output:** force-pushed branch `origin/published-skills`.

**Algorithm (lines 16–81):**

```
1. Assert cwd contains "claude-skills/" (else exit 1).
2. Read current branch via `git branch --show-current`.
3. Branch gate:
   - If branch ∉ {main, master}:
     - If -NonMainOk: warn, proceed.
     - Else: interactive Read-Host "Continue anyway? (y/N)"; only "y" continues, else exit 0.
4. Manifest drift check:
   a. Invoke generate-manifest.ps1.
   b. git status --porcelain claude-skills/SHARED_MANIFEST.
   c. If non-empty → ERROR + exit 1 (manifest must be committed first).
5. If local branch `published-skills` exists → git branch -D published-skills.
6. git subtree split --prefix=claude-skills -b published-skills (else exit 1).
7. git push origin published-skills --force (else exit 1).
8. Print "Done" + consumer pull instructions.
```

**Non-trivial detail (lines 56–60):** must delete any pre-existing local `published-skills` branch before `git subtree split -b` because the command refuses to write into an existing branch. The force-push at step 7 is intentional — the branch is a rebuild from `main`, not an additive history.

**Idempotency:** safe to re-run; each invocation rebuilds the manifest, the split branch, and the remote ref.

### 3.3 `sync-shared-skills.ps1` 🟢

**Purpose:** Mirror the remote `published-skills` branch into the consumer repo at `<prefix>/.claude/`. Track which files were last synced so removed files can be cleaned up. Optionally only check whether updates exist (`-UpdateCheck`).

**Parameters:**

| Param | Default | Purpose |
|---|---|---|
| `-Remote` | `shared-skills` | Git remote pointing back at BeckTech.QA.Tools |
| `-TargetPrefix` | `.claude` | Where to land synced files (relative to repo root) |
| `-Force` | false | Sync even if no version change |
| `-UpdateCheck` | false | Don't sync; exit 0 = up-to-date, 1 = updates pending |

**Algorithm (lines 196–289):**

```
1. RepoRoot = walk up parents until a .git/ exists (else error).
2. RecordFile = <RepoRoot>/<TargetPrefix>/.sync-record.
3. git fetch <Remote> published-skills (silent).
4. newVersion = git rev-parse <Remote>/published-skills (else error).
5. fileList = git ls-tree -r --name-only <Remote>/published-skills (else error).
6. (previousVersion, previousFiles) = Get-SyncRecord(RecordFile)
   - Empty hashtable if record absent or unreadable.
7. addedFiles   = fileList − keys(previousFiles)
   removedFiles = keys(previousFiles) − fileList
   fileSetUnchanged = (addedFiles.Count == 0 AND removedFiles.Count == 0)
8. If -UpdateCheck:
   - exit 0 if newVersion == previousVersion AND fileSetUnchanged
   - exit 1 otherwise
9. Short-circuit when no work to do:
   - if newVersion == previousVersion AND fileSetUnchanged AND not -Force → "up to date" + exit 0.
10. Sync-FilesFromRemote: see §3.3.a.
11. Save-SyncRecord: write { version, timestamp, files } as JSON.
12. Cleanup orphaned `.temp-plan-*.json` from repo root and .cache/.
13. Print "Next steps" (git diff, commit, push).
```

**Non-trivial detail (lines 225–228):** the file-set comparison is **not** count-based. A rename keeps `count == count` while changing the set; the explicit `addedFiles`/`removedFiles` diff is what triggers a re-sync.

#### 3.3.a `Sync-FilesFromRemote` (lines 71–154) 🟢

```
For each file in FileList:
  target = <RepoRoot>/<TargetPrefix>/<file>
  Create parent dirs if needed.
  content = git show <Remote>/published-skills:<file>
  If exit 0: Set-Content (UTF8, -Force); newSyncRecord[file] = true; syncedCount++.
  Else: warn + errorCount++.

For each oldFile in keys(previousSyncRecord) NOT in newSyncRecord:
  oldPath = <RepoRoot>/<TargetPrefix>/<oldFile>
  Remove-Item.
  Walk up parent dirs:
    - If empty AND not == <RepoRoot>/<TargetPrefix>: Remove-Item parent; recurse.
    - Stop on first non-empty or boundary hit.

Return { Synced, Deleted, Errors, NewRecord }.
```

**Non-trivial detail (lines 130–138):** empty-parent cleanup walks **upward** until hitting either a non-empty directory or the target root. This prevents orphaned empty `<prefix>/.claude/scripts/`-style directories when a whole subtree is removed upstream.

**Non-trivial detail (line 103):** uses `git show <ref>:<path>` instead of checkout, so the working tree is never disturbed. Side effect: line-ending normalization happens via `Set-Content -Encoding UTF8` and may not match the LF-only manifest semantics.

### 3.4 `check-updates.ps1` 🟢

**Purpose:** SessionStart hook. Fast, silent on success, prints a JSON payload that becomes additional context only when updates are actually pending.

**Algorithm (lines 11–73):**

```
1. ScriptDir = MyInvocation parent.
   ClaudeDir = ScriptDir parent (script lives at <ClaudeDir>/setup/check-updates.ps1).
2. RepoRoot = walk up parents of ClaudeDir until .git/ found; else exit 0.
3. Prefix = ClaudeDir.Substring(RepoRoot.Length).TrimStart('\','/').
   If empty → exit 0.
4. Cache key:
   CacheDir = $env:USERPROFILE/.claude/cache (mkdir -p).
   RepoKey = RepoRoot with [\\/:*?"<>|] replaced by '_'.
   CacheFile = $CacheDir/shared-skills-check-$RepoKey.txt.
5. Cache hit gate:
   If CacheFile.LastWriteTime < 4h ago → exit 0 (skip check).
6. Locate sync script at <ClaudeDir>/setup/sync-shared-skills.ps1; if missing → exit 0.
7. Push-Location $RepoRoot
   & syncScript -Remote shared-skills -TargetPrefix $Prefix -UpdateCheck
   updateAvailable = ($LASTEXITCODE -eq 1)
8. Touch CacheFile regardless of result.
9. If updateAvailable:
   Emit JSON: { hookSpecificOutput: { hookEventName: 'SessionStart', additionalContext: <msg> } }
```

**Non-trivial detail (lines 28–36):** the 4-hour cache is **per-repo** and lives in the user profile, not the repo. Replacing reserved Windows path characters in the cache key prevents collisions across drive letters and case-only differences.

**Non-trivial detail (line 8):** `$ErrorActionPreference = 'SilentlyContinue'` — the hook must never break session startup; any failure exits cleanly with no output.

### 3.5 `guard-shared-skills.ps1` 🟢

**Purpose:** PreToolUse hook for `Edit | Write | MultiEdit`. Blocks edits to subtree-owned files; allows everything else.

**Algorithm (lines 14–74):**

```
1. Escape hatch: if $env:QA_ALLOW_SHARED_EDIT == '1' → exit 0.
2. Read stdin (tool-input JSON). Empty input → exit 0 (allow).
3. ConvertFrom-Json; malformed → exit 0 (allow, never break user workflow).
4. filePath = payload.tool_input.file_path; missing → exit 0.
5. ClaudeDir = script's grandparent (i.e., consumer repo's .claude/).
6. manifestFile = <ClaudeDir>/SHARED_MANIFEST; missing → exit 0.
7. Resolve to absolute paths, forward-slash form:
   absTarget = [Path]::GetFullPath(filePath) (catch → exit 0).
   absClaude = [Path]::GetFullPath(ClaudeDir).
8. Containment check:
   If NOT absTarget.StartsWith(absClaude + '/', OrdinalIgnoreCase) → exit 0.
9. relative = absTarget.Substring(absClaude.Length + 1).
10. manifest = Get-Content manifestFile, skip blank lines and lines starting with '#'.
11. match = manifest first where Trim() -ieq relative.
12. If no match → exit 0 (file is in .claude/ but not subtree-owned, e.g., repo-context).
13. Else → write BLOCKED message to stderr, exit 2 (Claude Code halts the tool call).
```

**Non-trivial detail (line 17):** the `QA_ALLOW_SHARED_EDIT=1` env var is a session-scoped escape hatch for legitimate local debugging — checked once at the top before any work. Setting it requires restarting Claude Code (user-scope env var semantics).

**Non-trivial detail (line 29):** **fail-open** posture on malformed input. The hook should never be the reason a valid edit fails; any structural issue (missing manifest, unparseable JSON, bad path) defaults to allow.

**Non-trivial detail (line 52):** case-insensitive prefix comparison (`OrdinalIgnoreCase`) because Windows filesystems are case-insensitive but Git tracks paths case-sensitively. Without this, a user editing `.Claude\README.md` would bypass the guard.

### 3.6 `notify.ps1` (notifications) 🟢

**Purpose:** Windows balloon toast on Claude Code Stop/Notification events.

## 4. Control flow — orchestrator skills

### 4.1 `onboard` (per-user setup walk-through) 🟢

**Idempotent walk** of three repo sanity checks + seven per-user steps. For each step: detect state → tell user "✓ already configured" OR invoke the sub-skill.

**Pre-flight (lines 19–53):**

1. `shared-skills` remote presence (`git remote -v | grep shared-skills`); add if missing.
2. `repo-context` filled in (no `<!-- ... -->` placeholders left); offer `/setup-repo-context`.
3. `CLAUDE.md` in QA-owned directory; offer `/setup-claude-md`.
4. `git fetch shared-skills published-skills`; if updates available (sync `-UpdateCheck` exit 1), offer to sync.

**Per-user steps (lines 55–102):**

| Step | Detection | Action if missing |
|---|---|---|
| 1. Claude hooks | `.claude/settings.json` contains SessionStart→check-updates AND PreToolUse→guard-shared-skills AND `_qaHookVersion` matches | Invoke `setup-claude-hooks` |
| 2. Atlassian MCP | `claude mcp list` contains `atlassian` | Invoke `setup-atlassian-mcp` |
| 3. Atlassian creds | `ATLASSIAN_EMAIL` AND `ATLASSIAN_API_TOKEN` User-scope | Invoke `setup-atlassian-credentials` |
| 4. ADO MCP | `claude mcp list` contains `ado` | Invoke `setup-azure-devops-mcp` |
| 5. ADO creds | `[Environment]::GetEnvironmentVariable('ADO_PAT','User')` | Invoke `setup-ado-credentials` |
| 6. Playwright MCP | `claude mcp list` contains `playwright` | Ask; if yes invoke `setup-playwright-mcp` |
| 7. Notifications | `~/.claude/hooks/notify.ps1` exists AND `~/.claude/settings.json` has Stop/Notification hook | Ask; if yes invoke `setup-notifications` |

**Summary markers (lines 104–128):** `✓` already configured · `+` newly installed (typically needs restart) · `-` skipped by user · `!` flagged but not blocking.

### 4.2 `refresh-setup` (drift audit) 🟢

**Three artifacts, three different version-marker mechanisms.** The skill is read-only — it never writes generated files directly; it routes refreshes back to the owning setup skill so the same review/approve flow runs on re-installs.

**Process (lines 24–86):**

```
1. Locate .claude/ (repo root or subdir).
2. For each owning skill, read the `version` frontmatter field:
   - setup-claude-md      → target for CLAUDE.md
   - setup-repo-context   → target for repo-context
   - setup-claude-hooks   → target for settings.json hooks
3. For each artifact, read its marker:
   - CLAUDE.md: <!-- qa-version: claude-md=N --> at top
   - repo-context: frontmatter qa-version: N
   - settings.json: top-level _qaHookVersion: N
   Status:
     missing      = file doesn't exist
     unversioned  = file exists but no marker
     <version>    = parsed value
4. Compare:
   | Observed                 | vs Target | Status      | Action                                  |
   |--------------------------|-----------|-------------|-----------------------------------------|
   | equal                    | equal     | current     | none                                    |
   | lower OR unversioned     | higher    | stale       | offer to re-run owning skill (refresh)  |
   | missing                  | any       | missing     | offer to run owning skill from scratch  |
   | higher                   | lower     | ahead       | flag anomaly; never auto-downgrade      |
5. Present compact report with ✓ ! + ? markers.
6. For approved items, invoke owning skill (sequentially, one at a time).
7. Print done line; never auto-commit.
```

**Non-trivial detail (line 38):** missing skill directories in `.claude/skills/` are tolerated — note in summary, skip the audit for that artifact rather than failing the whole report.

## 5. Control flow — leaf setup skills

All nine leaf setup skills follow the same shape: **detect existing state → write/merge → stamp version → instruct restart**. Differences are only in *what* state is detected and *what* is written.

### 5.1 `setup-claude-hooks` (v=2) — project-scoped `.claude/settings.json` 🟢

**Writes:**

- `_qaHookVersion: 2` at top level (drift marker).
- `hooks.SessionStart[].hooks[]` → `powershell.exe -NoProfile -ExecutionPolicy Bypass -File "${CLAUDE_PROJECT_DIR}/.claude/setup/check-updates.ps1"`.
- `hooks.PreToolUse[]` with `matcher: "Edit|Write|MultiEdit"` → equivalent command pointing at `guard-shared-skills.ps1`.
- `permissions.allow[]` ← `["Write(~/.claude/fetch-jira-item/**)", "Bash(mkdir -p *fetch-jira-item*)", "Bash(powershell.exe*download-attachment.ps1*)"]`.

**Merge rule:** existing `SessionStart`, `PreToolUse`, `permissions.allow` arrays are extended, not replaced. Deduplicate by exact `command` string for hooks; exact rule string for allow-list.

**Drift detection:** if existing `_qaHookVersion` equals current `version` AND all required hooks present → "already installed and current" + exit.

**Maintainer rule (frontmatter comment):** bump `version` whenever the hook command, matcher, or any permissions.allow rule changes. Pure-internal fixes inside `check-updates.ps1`/`guard-shared-skills.ps1` do **not** require a bump (those scripts sync in independently).

## 6. Algorithms (cross-cutting)

### 6.1 Version-marker drift detection 🟢

Three independent marker styles, all detected by `refresh-setup`:

| Artifact | Marker location | Marker syntax | Owner |
|---|---|---|---|
| `CLAUDE.md` | top of file | `<!-- qa-version: claude-md=N -->` | `setup-claude-md` |
| `repo-context/SKILL.md` | frontmatter | `qa-version: N` | `setup-repo-context` |
| `.claude/settings.json` (project) | JSON top level | `"_qaHookVersion": N` | `setup-claude-hooks` |
| `~/.claude/settings.json` (user) | JSON top level | `"_qaNotificationsVersion": N` | `setup-notifications` |

**Comparison rule:**

```
status =
  missing       if file does not exist
  unversioned   if file exists but marker absent
  N             if marker present, integer N

action =
  none           if status == target
  refresh        if status < target  OR  status == unversioned
  install        if status == missing
  flag-anomaly   if status > target  (subtree is behind whatever wrote the file)
```

**Confidence:** 🟢 — three skills explicitly document the field name and target version; `refresh-setup` references all three.

---

### 6.2 Subtree-owned file lookup (guard) 🟢

```
relative = absolute(targetFile) − absolute(<consumer .claude>) − leading '/'
match    = any line of SHARED_MANIFEST (skip blank, skip '#' comments) where Trim() ==(case-insensitive) relative
```

- One file = one line; deduplicated and sorted by `generate-manifest.ps1`.
- The manifest entries are **subtree-relative** (e.g., `setup/sync-shared-skills.ps1`, not `claude-skills/setup/sync-shared-skills.ps1`) — see `generate-manifest.ps1:42` (`$_.Substring($prefix.Length)`).

**Confidence:** 🟢.

---

### 6.3 Sync delta computation 🟢

```
addedFiles      = remote − local-record       (new files arrived)
removedFiles    = local-record − remote       (files removed upstream)
fileSetChanged  = (added > 0) OR (removed > 0)
versionChanged  = remote-HEAD ≠ recorded-HEAD

needsSync = versionChanged OR fileSetChanged OR -Force
```

**Why not just count comparison:** a one-for-one rename produces equal counts but a different set. The explicit added/removed diff catches it.

**Confidence:** 🟢.

---

### 6.4 4-hour TTL cache for SessionStart 🟢

```
cacheKey  = repoRoot with [\\/:*?"<>|] → '_'
cacheFile = $env:USERPROFILE/.claude/cache/shared-skills-check-<cacheKey>.txt
gate:
  if exists AND (Now - LastWriteTime) < 4h → exit 0 (skip)
  else → run sync -UpdateCheck; touch cacheFile regardless of result
```

**Why timestamp-only:** the file's contents are unused — only `LastWriteTime` matters. Cheap to write, no parsing.

**Confidence:** 🟢.

## 7. Data structures

### 7.1 `SHARED_MANIFEST` (plain text) 🟢

- Encoding: UTF-8, LF line endings, trailing LF.
- Each line: one subtree-relative path with forward slashes.
- Excludes `SHARED_MANIFEST` itself.
- Sorted lexicographically; deduplicated.
- Blank lines and `#`-prefixed lines are tolerated by the guard (skipped on read) — but `generate-manifest.ps1` does not write them.
- **Count (current):** 76 entries — matches `git ls-files --cached --others --exclude-standard -- claude-skills` minus self.

---

### 7.2 `.sync-record` (JSON, written by sync-shared-skills) 🟢

```json
{
  "version": "<git-sha-of-published-skills-HEAD>",
  "timestamp": "<ISO 8601>",
  "files": {
    "<subtree-relative-path>": true,
    ...
  }
}
```

- Location: `<RepoRoot>/<TargetPrefix>/.sync-record` (e.g., `.claude/.sync-record`).
- `files` is a hashtable serialized as a JSON object; each key was synced on the last successful run, value always `true`.
- `version` is the **40-char hex SHA** of the remote `published-skills` HEAD at sync time.
- Read robustness: malformed JSON returns `@{ version = $null; files = @{} }` (treated as first sync).

### 7.3 `.claude/settings.json` (project-scoped) 🟢

```json
{
  "_qaHookVersion": 2,
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "powershell.exe -NoProfile -ExecutionPolicy Bypass -File \"${CLAUDE_PROJECT_DIR}/.claude/setup/check-updates.ps1\""
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "powershell.exe -NoProfile -ExecutionPolicy Bypass -File \"${CLAUDE_PROJECT_DIR}/.claude/setup/guard-shared-skills.ps1\""
          }
        ]
      }
    ]
  },
  "permissions": {
    "allow": [
      "Write(~/.claude/fetch-jira-item/**)",
      "Bash(mkdir -p *fetch-jira-item*)",
      "Bash(powershell.exe*download-attachment.ps1*)"
    ]
  }
}
```

| Field | Type | Purpose |
|---|---|---|
| `_qaHookVersion` | int | Drift marker; mirrors `version` in `setup-claude-hooks/SKILL.md` |
| `hooks.SessionStart[]` | array | Each entry has `hooks: [{ type: "command", command: "..." }]`; runs on session start |
| `hooks.PreToolUse[]` | array | Each entry has `matcher`, `hooks`. Matcher is a pipe-separated regex of tool names |
| `permissions.allow[]` | array | Pre-approved Write/Bash patterns; suppresses interactive permission prompts |

**Merge semantics:** arrays are *extended* (push deduplicated entries), never *replaced*. Other unrelated keys are preserved untouched.

## 12. Notes

- **The setup module is the trust boundary of the distribution model.** Everything else in this repo assumes `published-skills` is faithful to `main` (publish-skills enforces it) and that consumers cannot diverge silently (guard enforces it).
- **No domain logic.** Setup files are scaffolding; if a script breaks, no QA workflow runs, but no data is lost. By design.
- **Idempotency is the only correctness contract.** Re-running any setup script or skill on a fully-configured machine/repo MUST be a no-op (with at most a "current" message). Verified by reading the version-marker gates in each skill.
