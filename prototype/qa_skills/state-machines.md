## 1. ADO Test Case — System.State

The primary entity with a formal state machine is the **ADO Test Case** work item.

### States and Transitions

```mermaid
stateDiagram-v2
    [*] --> Design : wit_create_work_item
    Design --> Ready : wit_update_work_item\n(State=Ready)\nReason auto: Completed
    Ready --> Closed : manual close in ADO\nReason auto: (varies)
    Closed --> Ready : wit_update_work_item\n(State=Ready)\nReason auto: Reactivated

    note right of Design
        Initial state after creation.
        Steps are added here via
        testplan_update_test_case_steps.
    end note

    note right of Ready
        "Published" — visible in test plans.
        Standard operational state.
    end note

    note right of Closed
        Archived. Not included in
        active test runs by default.
    end note
```

### Business Rules

| Rule | Detail | Confidence |
|------|--------|------------|
| Creation always lands in `Design` | `wit_create_work_item` produces `Design` state | 🟢 |
| `Design → Ready` requires separate PATCH | Cannot combine creation fields + state transition in one call | 🟢 |
| `Closed → Design` is NOT a valid transition | Re-open always targets `Ready` | 🟢 |
| Never set `System.Reason` manually | Workflow assigns `Completed` / `Reactivated` automatically | 🟢 |
| Disposition `Absorbed` / `Dropped` → state unchanged | Field updates only; no state transition needed | 🟡 |

## 2. Work Item Routing — work-on Router

The `work-on` skill acts as a state machine routing Jira issue types to downstream workflows.

```mermaid
stateDiagram-v2
    [*] --> Fetching : User mentions Jira key\nor invokes /work-on

    Fetching --> RouteDecision : fetch-jira-item returns issue_type

    RouteDecision --> BugWorkflow : issue_type = Bug
    RouteDecision --> StoryWorkflow : issue_type = Automation Story\nor QA Story
    RouteDecision --> SpikeDecision : issue_type = Spike
    RouteDecision --> SubtaskResolution : issue_type = Subtask
    RouteDecision --> UserDecision : issue_type = Other / Unknown

    BugWorkflow --> [*] : fix-qa-bug completes
    StoryWorkflow --> [*] : implement-story completes

    SpikeDecision --> UserDecision : Present spike question
    UserDecision --> BugWorkflow : User picks Bug path
    UserDecision --> StoryWorkflow : User picks Story path
    UserDecision --> [*] : User declines / one-off task

    SubtaskResolution --> RouteDecision : Check parent type
    SubtaskResolution --> UserDecision : Self-contained subtask
```

### Shortcut Rule

If user's message makes type obvious (`fix QA-1234`, `implement DC-456`), `RouteDecision` is skipped. Jira fetch still runs.

## 3. apply-test-plan — Execution Pipeline

`apply-test-plan` has a linear gate-based state machine: each stage must pass before the next is allowed.

```mermaid
stateDiagram-v2
    [*] --> Locate : User invokes /apply-test-plan <path>

    Locate --> Parse : Markdown file found
    Locate --> [*] : File not found → error

    Parse --> Validate : Sidecar JSON generated\nor loaded from cache

    Validate --> ApprovalGate1 : Validation passed\n(required fields present)
    Validate --> [*] : Validation failed → error + instructions

    ApprovalGate1 --> CreateUpdate : User approves ADO mutations\n(explicit confirmation required)
    ApprovalGate1 --> [*] : User declines

    CreateUpdate --> JiraWriteback : feature plan only\ncreate-ado-test-cases + update-ado-test-cases run
    CreateUpdate --> Done : regression plan\n(no Jira step)

    JiraWriteback --> Done : TestPlan/Suite IDs written\nto covered Jira stories

    Done --> [*]
```

**Gate invariant:** `apply-test-plan` never authors or edits plan content. Any content issue → redirect to `/revise-test-plan`.

## 6. Guard Hook — File Edit Decision

```mermaid
stateDiagram-v2
    [*] --> HookReceives : Edit/Write/MultiEdit fires\n(PreToolUse)

    HookReceives --> ParsePayload : Read stdin JSON

    ParsePayload --> TargetInClaude : Extract file_path

    TargetInClaude --> ManifestLookup : Target is under .claude/
    TargetInClaude --> Allow : Target outside .claude/\n(fail-open)

    ManifestLookup --> ManifestRead : Manifest file found

    ManifestRead --> Lookup : Normalize path, case-insensitive
    ManifestRead --> Allow : Manifest missing or unreadable\n(fail-open)

    Lookup --> Block : Path matches a manifest entry\nstderr BLOCKED + exit 2
    Lookup --> Allow : No match → exit 0

    ParsePayload --> Allow : Any structural error\n(fail-open)

    Allow --> [*]
    Block --> [*]
```
