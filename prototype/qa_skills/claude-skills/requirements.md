## Overview

Claude Skills is the primary distribution artifact of BeckTech.QA.Tools. It exposes 19 reusable QA automation workflows as stateless, composable Claude Code skills (SKILL.md files), invoked via `/command` syntax. Skills are distributed to consuming repos via git subtree and protected from accidental modification by a PreToolUse hook.

**Core pattern:** High-order entry-point skills (e.g., `/work-on`, `/apply-test-plan`) delegate to lower-order skills and PowerShell scripts, forming a directed call graph rooted at user-facing commands.

---

## Responsibilities

- Route work items by Jira issue type to the correct automation workflow 🟢
- Fetch and cache Jira issues with timestamp-based invalidation 🟢
- Generate test plan Markdown for feature and regression scopes 🟢
- Apply test plans to Azure DevOps (validate → approve → create → suite) 🟢
- Create and transition ADO test case work items with all required custom fields 🟢
- Manage ADO test suites (resolve by name or ID, create if absent, add TCs) 🟢
- Distribute skills to consuming repos via git subtree and guard against edits 🟢
- Enforce seven QA automation principles across all workflows 🟢
- Guide new users through full environment onboarding 🟢
- Detect and prompt skill updates on session start 🟢

## Business Rules

- **BR-WR-01** Jira `Bug` → route to `fix-qa-bug` 🟢
- **BR-WR-02** Jira `Automation Story` or `QA Story` → route to `implement-story` 🟢
- **BR-WR-03** Jira `Spike` → present question, no auto-route 🟢
- **BR-WR-04** Jira `Subtask` → route by parent type; if self-contained, ask user 🟢
- **BR-WR-05** Unknown Jira type → present summary, ask user 🟢
- **BR-WR-06** If message intent is unambiguous (e.g., "fix QA-123"), skip routing prompt; Jira fetch still runs 🟢
- **BR-TC-01** All required ADO custom fields set in a single `wit_create_work_item` call to avoid `TF401320` 🟢
- **BR-TC-02** Steps set via `testplan_update_test_case_steps` after TC creation — never in the create call 🟢
- **BR-TC-03** `Custom.ManualReason` required whenever `Custom.Manual = true` 🟢
- **BR-TC-04** `AutomationStatus = Planned` for regression-intent TCs; `Not Automated` for all others 🟢
- **BR-TC-05** State transitions to `Ready` via separate PATCH; do not set `System.Reason` 🟢
- **BR-TC-06** `Custom.CreatedIn` must not be set on new TCs (being phased out) 🟢
- **BR-TC-07** `|` is the step delimiter in `testplan_update_test_case_steps`; strip `|` from step text 🟢
- **BR-TC-08** Always use `wit_create_work_item` with `workItemType: "Test Case"` — not `testplan_create_test_case` 🟢
- **BR-TC-09** TC creation is NOT idempotent — re-running produces duplicates 🟢
- **BR-AP-01** `apply-test-plan` must not author or edit plan content; editing belongs to `revise-test-plan` 🟢
- **BR-AP-02** Explicit user approval required before any ADO mutation 🟢
- **BR-AP-03** Feature plans write TestPlan + Suite IDs back to Jira after creation; regression plans do not 🟢
- **BR-JI-01** Only `raw.json` counts as a valid Jira cache — absence forces re-fetch 🟢
- **BR-JI-02** Cache freshness: compare `updated` field; mismatch triggers re-fetch 🟢
- **BR-JI-03** 401 from Jira → instruct user to re-auth via `/mcp`; do not proceed without auth 🟢
- **BR-JI-04** Feature plans write `customfield_10257` (Test Plan ID) + `customfield_10258` (Test Suite ID) back to Jira stories after ADO TC creation. Regression plans do not. (Cross-ref `BR-AP-03`.) 🟢
- **BR-JI-05** Jira Cloud ID hardcoded: `ec1222f5-ecd6-4b14-a4b4-68c3ed82ef14` 🟢
- **BR-SD-03** Guard hook exits 0 (allow) on any structural error — fail-open posture 🟢
- **BR-SD-04** `QA_ALLOW_SHARED_EDIT=1` bypasses guard — maintainer use only 🟢
- **BR-SD-05** Update check cached per repo root with 4-hour TTL 🟢
- **BR-PL-01** Test code favors readability and diagnosability over production-grade defensive coding 🟢
- **BR-PL-02** Scope discipline: changes limited to the work item 🟢
- **BR-PL-03** Tests interact through real interfaces (UI clicks, HTTP requests) 🟢
- **BR-PL-04** Diagnosable failures over silent recovery — except teardown cleanup 🟢
- **BR-PL-05** Avoid hardcoded sleeps when a targeted wait condition exists 🟢
- **BR-PL-06** repo-context conventions take precedence over `principles` skill when they conflict 🟢

## Acceptance Criteria

```gherkin
# Work item routing — happy path
Given a Jira issue key of type "Bug"
When the user runs /work-on QA-1234
Then work-on fetches the issue, routes to fix-qa-bug, and presents the bug fix workflow

# Work item routing — unknown type
Given a Jira issue key of type "Spike"
When the user runs /work-on QA-9999
Then work-on presents the spike question and waits for user direction

# apply-test-plan — approval gate
Given a valid feature plan Markdown at output/DC-4719-plan.md
When the user runs /apply-test-plan output/DC-4719-plan.md
Then the skill validates the plan, summarizes ADO mutations, and does not call ADO until user explicitly approves

# apply-test-plan — ADO creation
Given user approval is received
When apply-test-plan proceeds
Then create-ado-test-cases.ps1 creates TCs with state=Ready and all required Custom.* fields set

# Guard hook — blocked edit
Given a file listed in .sync-manifest (e.g., .claude/skills/apply-test-plan/SKILL.md)
When the user attempts to edit it via any file tool
Then PreToolUse hook exits non-zero and returns a message directing the user to submit a PR to BeckTech.QA.Tools

# Guard hook — error condition
Given guard-shared-skills.ps1 encounters a missing or corrupt .sync-manifest
When PreToolUse fires
Then hook exits 0 (allow) — fail-open posture

# Jira cache — cache hit
Given raw.json exists for QA-1234 and Jira's updated timestamp matches the cached value
When fetch-jira-item runs for QA-1234
Then the cached brief is returned; no full REST fetch is made

# Jira cache — cache miss
Given raw.json exists for QA-1234 but Jira's updated timestamp is newer
When fetch-jira-item runs for QA-1234
Then full fetch is made, raw.json is overwritten, and brief is regenerated
```
