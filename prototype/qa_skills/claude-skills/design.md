## Interface

Claude Skills expose 19 entry-point and orchestrator workflows via `/command` syntax. Skills are stateless and compose through explicit calls; no shared state or global context.

### Entry Points (User-Invocable)

| Command | Skill | Input | Output | Purpose |
|---------|-------|-------|--------|---------|
| `/work-on KEY` | work-on | Jira issue key | Routed to fix-qa-bug or implement-story | Route work item by type |
| `/fetch-jira-item KEY` | fetch-jira-item | Jira issue key | Structured brief + cache | Fetch + cache Jira issue |
| `/fix-qa-bug KEY` | fix-qa-bug | Jira issue key | Plan + commit link | Bug fix workflow |
| `/implement-story KEY` | implement-story | Jira issue key | Plan + commit link | Story implementation workflow |
| `/plan-change DESCRIPTION` | plan-change | Change description | Approved implementation plan | Create change plan |
| `/plan-feature-coverage EPIC` | plan-feature-coverage | Epic/story key | Test plan Markdown | Feature coverage plan |
| `/plan-regression-coverage` | plan-regression-coverage | (none — scope from repo-context) | Test plan Markdown | Regression coverage plan |
| `/apply-test-plan PATH` | apply-test-plan | File path to plan Markdown | ADO test cases + suite | Apply plan to ADO |
| `/revise-test-plan PATH` | revise-test-plan | File path to plan Markdown | Updated plan Markdown | Revise plan (no ADO) |
| `/onboard` | onboard | (none) | Setup completion report | New user setup |
| `/refresh-setup` | refresh-setup | (none) | Update check + hook refresh | Refresh setup + check versions |
| `/principles` | principles | (none) | Display 7 principles | View QA automation principles |

### Approval Gate Phrase Patterns

Both `plan-change` and `review-change` enforce approval gates documented in their SKILL.md prose. Approval is **model-discretionary**, recognized against these example phrases (no regex, no whitelist enum):

- `plan-change/SKILL.md`: *"Phrases like 'looks good', 'let's go', 'approved', or 'go ahead' count. Questions or suggestions mean keep iterating."*
- `review-change/SKILL.md`: *"Phrases like 'looks good', 'approved', 'ship it', or 'go ahead' count. Questions, concerns, or suggestions mean the changes need more work."*

Iteration continues until explicit approval. Ambiguous user input ("looks good enough", "fine") is judged at the model's discretion against the gate text; tend toward continuing iteration when unclear.

### Concurrency Assumption

Single-user, single-session per repo (matches `architecture.md §2`). Per-issue Jira cache isolation makes parallel `/work-on QA-A` + `/work-on QA-B` safe across two Claude Code sessions; parallel `/apply-test-plan` runs against the same repo are unsafe (no operation-JSON / sidecar locking). 🟢

## Fluxo Principal

### Flow 1: `/work-on` Routing

```
1. work-on fetches Jira issue via fetch-jira-item
2. Extract issueType from Jira response
3. Match type against decision matrix (BR-WR-01–06)
4. If type is Bug → invoke fix-qa-bug [key]
5. If type is Story → invoke implement-story [key]
6. If type is Spike → present spike question
7. If type is Unknown → present summary, ask user
```

**Confidence:** 🟢 (observed in SKILL.md §1–2)

### Flow 2: `/apply-test-plan` — Complete Pipeline

```
1. Locate Markdown file at provided path
2. Parse Markdown: extract plan_type (feature|regression), header, test cases table, steps
3. Validate against repo-context:
   - Check QA Functional Areas exist
   - Validate Area/Iteration paths in ADO
   - Flag mismatches
4. Summarize ADO mutations (count, preview)
5. GET explicit user approval ("continue", "proceed", "go", "yes" → approval)
6. If approved:
   a) Invoke create-ado-test-cases.ps1 → creates TCs
   b) Invoke manage-ado-test-suite.ps1 → adds TCs to suite
   c) If feature plan: invoke set-jira-test-plan-ids.ps1 → write back to Jira
   d) Return success envelope
7. If not approved: return and await user adjustment
```

**Confidence:** 🟢 (8-step documented in apply-test-plan/references/instructions.md)

### Flow 3: Jira Fetch with Caching

```
1. Receive [issue-key] argument
2. Check ~/.claude/fetch-jira-item/<key>/raw.json
3. If missing → fetch full issue (Step 5)
4. If exists:
   a) Extract cached 'updated' timestamp
   b) Lightweight REST call: GET /issue/{key}?fields=updated
   c) Compare timestamps
   d) If match → cache hit, skip to Step 6
   e) If differ → cache miss, go to Step 5
5. Full REST GET /issue/{key}?fields=*all → save raw.json
6. Download attachments (if present) to cache/jira-attachments/
7. Write brief.md (structured Markdown)
8. Return brief to user
```

**Confidence:** 🟢 (fully detailed in fetch-jira-item/SKILL.md §2–5)

## Fluxos Alternativos

- **User rejects plan changes:** `plan-change` loops with user until approval or user quits
- **ADO API fails during TC creation:** Script returns errors array; partial success reported
- **Jira auth fails (401):** Instruct user to re-auth via `/mcp`; do not proceed
- **Guard hook detects synced file edit:** Return error message; do not allow edit
- **Guard hook encounters missing manifest:** Fail-open (allow edit)

## Dependências

- **Azure DevOps MCP** — All ADO calls (test case creation, suite management, wiki updates)
- **Jira/Atlassian MCP** — All Jira calls (fetch, comment, transition)
- **Playwright MCP** — Optional; used by qa-investigate agent for UI testing
- **Agents (qa-investigate, qa-implement)** — Spawned by fix-qa-bug / implement-story
- **PowerShell Scripts (22)** — Low-level REST wrappers for Jira, ADO, utilities
- **repo-context Skill** — Per-repo QA configuration (area paths, functional areas, defaults)
- **principles Skill** — Loaded by all workflow skills; governs output quality

## Decisões de Design Identificadas

| Decisão | Evidência | Confiança |
|---------|-----------|-----------|
| Stateless skills with composition | SKILL.md files are pure functions; state in repo files (plans, caches) | 🟢 |
| Jira fetch caching in user home dir | BR-JI-01; cache in ~/.claude/fetch-jira-item/<key>/ | 🟢 |
| Lightweight timestamp check before full fetch | BR-JI-02; reduces API calls and improves session start speed | 🟢 |
| Approval gates before ADO mutations | BR-AP-02; required by policy | 🟢 |
| Operations JSON as intermediate format | Schema in code-analysis.md §3.3; used by create-ado-test-cases.ps1 | 🟢 |
| Feature plans write back to Jira; regression plans do not | BR-AP-03; feature plans update Jira story fields 10257, 10258 | 🟢 |
| Guard hook fail-open on error | BR-SD-03; prevents blocking user edits due to manifest issues | 🟢 |
| 4-hour TTL for update check cache | BR-SD-05; balance between freshness and session speed | 🟢 |
| All custom fields set in single wit_create_work_item call | BR-TC-01; avoids TF401320 error on field population | 🟢 |

## Estado Interno

Skills do not maintain internal state. External state lives in:

- **Jira cache:** `$USERPROFILE/.claude/fetch-jira-item/<key>/raw.json` (Jira issue snapshot)
- **Test plans:** `<repo-root>/test-plans/` (user-authored Markdown)
- **Operations JSON:** Generated in-memory during apply-test-plan; passed to scripts
- **Sync manifest:** `.claude/.sync-manifest` (JSON list of synced file hashes)
- **Version markers:** In generated files (CLAUDE.md, settings.json, repo-context/SKILL.md)

## Observabilidade

- **Logs:** Skills output structured messages (approval gates, validation errors, API call summaries)
- **Caching signals:** Jira fetch reports "cache hit" or "fetched" to user
- **ADO mutations:** apply-test-plan previews changes before approval
- **Setup completion:** onboard reports each step status (done/error/skipped)

## Riscos e Lacunas

- 🔴 Exact error handling in PowerShell scripts beyond validation — exception recovery not fully documented
- 🟡 Concurrent skill execution safety — no documented guard against parallel /work-on and /apply-test-plan in same repo
- 🟡 Performance for large test batches (>100 TCs) — not measured
- 🔴 Rollback on partial ADO creation failures — what happens if 50 TCs created, 51st fails?
