## 1. Glossary

| Term | Definition | Confidence |
|------|-----------|------------|
| **Skill** | A Claude Code workflow defined as a `SKILL.md` file under `.claude/skills/<name>/`. Entry-point-driven (`/command`), stateless, composable. | ЁЯЯв |
| **Agent** | An autonomous Claude Code worker with declared tool permissions and a model. Invoked by skills; produces files, not interactive output. | ЁЯЯв |
| **Script** | A PowerShell `.ps1` file that wraps a single REST API surface. Leaf node in the skill call graph; returns a `Standard Output Envelope`. | ЁЯЯв |
| **Shared Skill** | Any skill distributed from `BeckTech.QA.Tools` to a consuming repo via git subtree. Listed in `SHARED_MANIFEST`. | ЁЯЯв |
| **Repo-Context** | A consuming-repo-owned `SKILL.md` that holds project-specific constants (ADO field defaults, QA Functional Areas). Loaded by agents. Not shared. | ЁЯЯв |
| **Consumer Repo** | A downstream repo (e.g., `BeckTech.QA.Estimator`) that pulls shared skills via `git subtree pull`. | ЁЯЯв |
| **Producer** | `BeckTech.QA.Tools` itself тАФ the authoritative source for shared skills. | ЁЯЯв |
| **published-skills branch** | A derived Git branch containing only the content of `claude-skills/`. Rebuilt from scratch on each publish via `git subtree split --force-push`. | ЁЯЯв |
| **SHARED_MANIFEST** | A plain-text file listing all paths distributed via subtree. Used by `guard-shared-skills.ps1` to block edits on consuming repos. | ЁЯЯв |
| **Test Plan** | A Markdown document generated by `test-plan` skill, representing a feature or regression test scope. Has two shapes: **feature** (per epic/story) and **regression** (AUT-wide). | ЁЯЯв |
| **AUT** | Application Under Test тАФ the product name (e.g., `Cost Library`, `Estimator`). Stored in `Custom.AUT` on ADO test cases. | ЁЯЯв |
| **Operations JSON** | The input format for `create-ado-test-cases.ps1` and `update-ado-test-cases.ps1`. Schema: `{ defaults, test_cases[] }`. | ЁЯЯв |
| **Sidecar** | A JSON file generated during `apply-test-plan` execution that caches the parsed test plan. Aligned with Operations JSON schema to avoid re-serialization. | ЁЯЯв |
| **Standard Output Envelope** | The JSON response format of all PowerShell scripts: `{ success: bool, data: object\|null, error: string\|null }`. | ЁЯЯв |
| **QA Functional Area** | A per-product ADO picklist (`Custom.QAFunctionalArea`) that categorizes test cases (e.g., `Cost Library - Units`). | ЁЯЯв |
| **Disposition** | The classification of a test case in the regression-coverage workflow. Four values: **NEW**, **Manual**, **Absorbed**, **Dropped**. | ЁЯЯв |
| **Hook** | A Claude Code lifecycle event handler. `SessionStart` fires on session open; `PreToolUse` fires before any tool call. | ЁЯЯв |
| **Drift** | The condition when a version-marker in a generated file (`CLAUDE.md`, `settings.json`, `repo-context`) is older than the current template version. | ЁЯЯв |
| **Subtask** | A Jira issue type that is a child of a parent Bug or Story. Route depends on parent type. | ЁЯЯв |
| **Spike** | A Jira research task type. May not produce code тАФ `work-on` asks the user how to proceed. | ЁЯЯв |
| **TF401320** | Azure DevOps API error indicating missing required fields at test case creation time. Avoided by populating all required custom fields in the initial `wit_create_work_item` call. | ЁЯЯв |
| **CoverageType** | ADO field `Custom.CoverageType`. Values: `Story`, `Regression`, `Exploratory`. | ЁЯЯв |
| **AutomationStatus** | ADO field `Microsoft.VSTS.TCM.AutomationStatus`. Values: `Planned`, `Not Automated`, `Automated`. Context-dependent, not a flat default. | ЁЯЯв |
| **qa-version** | Version integer stamped in generated artifacts to track freshness. Distinct marker per artifact type. | ЁЯЯв |
| **4-hour TTL** | The staleness window for the `check-updates.ps1` cache. Within 4 hours of a session start, the update check is silently skipped. | ЁЯЯв |
| **Fail-open** | The guard posture: when `guard-shared-skills.ps1` cannot determine if a file is shared (missing manifest, parse error, path error), it allows the operation (`exit 0`). | ЁЯЯв |
| **force-push** | The `published-skills` branch is rebuilt and force-pushed on each publish. History on that branch is intentionally disposable. | ЁЯЯв |
| **nextPageToken** | Jira Cloud `POST /search/jql` pagination token. `fetch-jira-items-batch.ps1` loops while `isLast == false`. | ЁЯЯв |
| **jira-fetch-brief.md** | Structured Markdown summary written per Jira issue to `$USERPROFILE/.claude/fetch-jira-item/<key>/`. Canonical input for agent context. | ЁЯЯв |

## 2. Business Rules

### 2.1 Work Item Routing

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-WR-01** | Jira `Bug` type тЖТ route to `fix-qa-bug` | ЁЯЯв |
| **BR-WR-02** | Jira `Automation Story` or `QA Story` тЖТ route to `implement-story` | ЁЯЯв |
| **BR-WR-03** | Jira `Spike` тЖТ present spike question, ask user how to proceed. No auto-route. | ЁЯЯв |
| **BR-WR-04** | Jira `Subtask` тЖТ check parent type, route by parent. If self-contained, ask user. | ЁЯЯв |
| **BR-WR-05** | Unknown type тЖТ present summary, ask user which workflow applies. | ЁЯЯв |
| **BR-WR-06** | If user's message makes type obvious (e.g., "fix QA-123"), skip routing decision and go directly to workflow. Jira fetch still runs first. | ЁЯЯв |

### 2.2 Test Case Creation

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-TC-01** | All required custom fields must be set in the **same** `wit_create_work_item` call to avoid `TF401320`. | ЁЯЯв |
| **BR-TC-02** | Steps are set via `testplan_update_test_case_steps` after creation тАФ never in `wit_create_work_item`. | ЁЯЯв |
| **BR-TC-03** | `Custom.ManualReason` is required whenever `Custom.Manual = true`. | ЁЯЯв |
| **BR-TC-04** | `AutomationStatus = Planned` for new regression-intent TCs; `Not Automated` for everything else. | ЁЯЯв |
| **BR-TC-05** | `System.State` must transition to `Ready` after creation (separate PATCH). Do not set `System.Reason`. | ЁЯЯв |
| **BR-TC-06** | `Custom.CreatedIn` field is **not** to be set on new TCs тАФ being phased out. | ЁЯЯв |
| **BR-TC-07** | The `|` character is the step delimiter in `testplan_update_test_case_steps` format. Strip `|` from step text. | ЁЯЯв |
| **BR-TC-08** | `testplan_create_test_case` cannot set required Custom fields. Always use `wit_create_work_item` with `workItemType: "Test Case"`. | ЁЯЯв |
| **BR-TC-09** | Creation is **not idempotent** тАФ re-running `create-ado-test-cases.ps1` at the same input produces duplicate work items. | ЁЯЯв |

### 2.3 Regression Coverage Dispositions

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-RC-01** | NEW TC (Section 1): `CoverageType=Regression`, `RegressionCoverageEvaluation=Contributes to Coverage`, `Manual=false`, `AutomationStatus=Planned`. | ЁЯЯв |
| **BR-RC-02** | Manual regression (Section 2): `CoverageType=Regression`, `RegressionCoverageEvaluation=Contributes to Coverage`, `Manual=true`, `ManualReason=Multi-User\|Low Value To Effort`, `AutomationStatus=Not Automated`. | ЁЯЯв |
| **BR-RC-03** | Absorbed TC: `CoverageType=Regression`, `RegressionCoverageEvaluation=Evaluated`, `Manual=true`, `ManualReason=Covered By Other Automation`, `AutomationStatus=Not Automated`. | ЁЯЯв |
| **BR-RC-04** | Dropped TC: `CoverageType=Story` (unchanged), `RegressionCoverageEvaluation=Evaluated`, `Manual=true`, `ManualReason=Story Coverage Type`, `AutomationStatus=Not Automated`. | ЁЯЯв |

### 2.4 State Transition Rules (ADO Test Case)

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-ST-01** | State machine: `Design тЖТ Ready тЖТ Closed`. Re-open: `Closed тЖТ Ready` (never `Closed тЖТ Design`). | ЁЯЯв |
| **BR-ST-02** | When setting `System.State`, do **not** set `System.Reason`. ADO assigns reason automatically. | ЁЯЯв |
| **BR-ST-03** | `Reason` for `Design тЖТ Ready` is automatically `Completed`. For `Closed тЖТ Ready` it is `Reactivated`. | ЁЯЯв |

### 2.5 Jira Integration

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-JI-01** | Only `raw.json` counts as a valid Jira cache. Presence of brief but absence of `raw.json` тЖТ re-fetch. | ЁЯЯв |
| **BR-JI-02** | Cache freshness check: compare `updated` timestamp from `raw.json` vs live Jira field. If different тЖТ re-fetch. | ЁЯЯв |
| **BR-JI-03** | Auth failure (401) тЖТ instruct user to re-authenticate via `/mcp`. Do NOT proceed without auth. | ЁЯЯв |
| **BR-JI-04** | For feature test plans, `apply-test-plan` writes TestPlan + Suite IDs back to covered Jira stories after creation. Regression plans do not touch Jira. | ЁЯЯв |
| **BR-JI-05** | Jira Cloud ID is hardcoded: `ec1222f5-ecd6-4b14-a4b4-68c3ed82ef14`. | ЁЯЯв |

### 2.6 Skill Distribution

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-SD-01** | `published-skills` branch is a projection of `claude-skills/`. It is force-pushed on each release. History is disposable. | ЁЯЯв |
| **BR-SD-02** | `publish-skills.ps1` refuses to publish if `SHARED_MANIFEST` has drifted from current tracked files. | ЁЯЯв |
| **BR-SD-03** | Guard hook: if `guard-shared-skills.ps1` encounters any structural error (missing manifest, parse failure, path resolution), it exits 0 (allow). Fail-open to prevent blocking valid edits. | ЁЯЯв |
| **BR-SD-04** | `QA_ALLOW_SHARED_EDIT=1` env var bypasses the path guard entirely. Intended for maintainer use. | ЁЯЯв |
| **BR-SD-05** | Update check (`check-updates.ps1`) is cached per repo root (4-hour TTL). Cache key is the repo root path with special chars тЖТ `_`. | ЁЯЯв |
| **BR-SD-06** | Consumer sync uses **set-based delta** (not count): added = remote тИТ local-record; removed = local-record тИТ remote. Required because renames preserve file count. | ЁЯЯв |

### 2.7 apply-test-plan Gate Rules

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-AP-01** | `apply-test-plan` must NOT author or edit plan content. If peer review feedback applies тЖТ recommend `/revise-test-plan`. | ЁЯЯв |
| **BR-AP-02** | Explicit approval required before invoking ADO mutations. | ЁЯЯв |
| **BR-AP-03** | Feature plans: Jira writeback (TestPlan/Suite IDs) after ADO creation. Regression plans: no Jira writeback. | ЁЯЯв |

### 2.8 Principles (Unwritten Laws)

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-PL-01** | Test code prioritizes **readability and diagnosability** over production-grade defensive coding. | ЁЯЯв |
| **BR-PL-02** | Scope discipline: changes limited to the work item. "While I'm here" fixes belong in separate work items. | ЁЯЯв |
| **BR-PL-03** | Tests interact through **real interfaces** (UI clicks, HTTP requests). Direct property manipulation is disallowed for test actions; allowed for observation. | ЁЯЯв |
| **BR-PL-04** | Diagnosable failures over silent recovery. Silent catch blocks that mask failures are prohibited. Exception: teardown cleanup code. | ЁЯЯв |
| **BR-PL-05** | Avoid hardcoded sleeps when a targeted wait condition exists. | ЁЯЯв |
| **BR-PL-06** | repo-context conventions take precedence over shared `principles` skill when they conflict. | ЁЯЯв |

### 2.9 CI/CD Release Rules

| Rule | Detail | Confidence |
|------|--------|------------|
| **BR-CI-01** | NuGet package versions are driven by `<Version>` in each `src/*.csproj`. Not by Git tags or build numbers. | ЁЯЯв |
| **BR-CI-02** | Re-running `publish-testkit.yml` at same `<Version>` is a no-op (`--skip-duplicate`). | ЁЯЯв |
| **BR-CI-03** | Neither pipeline runs on PRs (`pr: none`). `publish-testkit.yml` double-gates with `ne(Build.Reason, 'PullRequest')`. | ЁЯЯв |
| **BR-CI-04** | Package glob uses `Get-ChildItem` (PowerShell) not shell glob тАФ prevents NuGet SDK version sensitivity. | ЁЯЯв |
