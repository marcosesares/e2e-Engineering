## 1. Module overview

The CI/CD Pipelines module is the **release-automation layer** of the repository. It owns two independent pipelines, both declared as **Azure DevOps Pipelines YAML** (despite the Scout's earlier mention of GitHub Actions — the actual `azureDevOps`-specific schema is used: `trigger`, `pool.vmImage`, `task: UseDotNet@2`, `task: PublishTestResults@2`, `${{ parameters.* }}`, `$(Build.ArtifactStagingDirectory)`):

| Pipeline file | Output | Trigger |
|---------------|--------|---------|
| `publish-claude-skills.yml` | `published-skills` branch updated via `git subtree split --prefix=claude-skills` + force-push | Push to `master` touching `claude-skills/*` |
| `publish-testkit.yml` | `.nupkg` packages pushed to `DESTINI-Web` Azure Artifacts feed | Push to `master` touching `dotnet/BeckTech.QA.TestKit/**` |

Both pipelines:
- Run on `windows-latest` hosted agents.
- Use path filters so unrelated commits do not trigger releases.
- Disable PR validation (`pr: none`) — releases are post-merge only.
- Rely on **persisted git credentials** (`persistCredentials: true`) or **Azure Artifacts auth** (`NuGetAuthenticate@1`) instead of explicit secrets in the YAML.

The module has **no business logic** — it is pure release plumbing. Correctness criteria are: exit code 0 from each step, presence of the published branch / `.nupkg` artifact, and idempotency at the same input version.

## 3. `publish-claude-skills.yml` — subtree publisher

### 3.1 Triggers and resources

```yaml
trigger:
  branches:
    include: [master]
  paths:
    include: [claude-skills/*]

pr: none

resources:
  repositories:
    - repository: self
      type: git

pool:
  vmImage: 'windows-latest'
```

🟢 CONFIRMADO:
- Path filter `claude-skills/*` (single-level glob, **not** recursive `**`). ADO treats `claude-skills/*` as matching files directly under `claude-skills/` *and* descendant changes — the YAML schema documentation states `*` matches across path segments inside path filters. 🟡 INFERIDO behaviour: descendant edits do fire the pipeline based on observed `Merged PR` history (`bc5df85`, `4a67068`, `71a0d10`) all touching deep `claude-skills/skills/**/SKILL.md` files.
- `pr: none` disables CI on PR branches — pipeline only runs on the merge commit landing on `master`.
- `persistCredentials: true` (in `checkout`) is required by `git push origin` later in the same job.

### 3.2 Steps

| # | Step | Type | Notes |
|---:|---|---|---|
| 1 | `checkout: self` | builtin | `fetchDepth: 0` (full history — required by `git subtree split`); `persistCredentials: true` |
| 2 | `Publish claude-skills to published-skills branch` | `powershell` (inline) | Single block: configure user, delete stale local branch, `git subtree split`, force-push |

### 3.3 PowerShell step — control flow

```powershell
git config user.email "build@beck-technology.com"
git config user.name "Build Service"

git show-ref --verify --quiet refs/heads/published-skills
if ($LASTEXITCODE -eq 0) {
    Write-Output "Deleting existing local published-skills branch..."
    git branch -D published-skills | Out-Null
}

Write-Output "Splitting claude-skills/ into published-skills branch..."
git subtree split --prefix=claude-skills -b published-skills

if ($LASTEXITCODE -ne 0) {
    Write-Error "git subtree split failed."
    exit 1
}

Write-Output "Pushing published-skills branch to origin..."
git push origin published-skills --force

if ($LASTEXITCODE -ne 0) {
    Write-Error "git push failed."
    exit 1
}
```

Key invariants 🟢:
- **Branch reset before split.** `git subtree split -b <name>` aborts if `<name>` already exists locally. Because `fetchDepth: 0` may have hydrated a local tracking ref for `published-skills`, the script defensively deletes it first.
- **Force push is intentional.** Each release rewrites the entire `published-skills` history (the branch is a *projection* of `claude-skills/`, not an additive log).
- **Two error gates.** Every `git` invocation that can fail (`subtree split`, `push`) is followed by an `$LASTEXITCODE -ne 0` check → `Write-Error` + `exit 1`. The `show-ref` and `branch -D` calls intentionally do **not** check exit codes — `show-ref` reports absence with non-zero (handled by the `if`), and `branch -D` runs only inside the success branch.
- **Identity hard-coded.** Author of the resulting commits on `published-skills` is `Build Service <build@beck-technology.com>` (not parameterized, not pulled from `System.AccessToken` identity).

### 3.4 Reuse of equivalent logic in `claude-skills/setup/publish-skills.ps1`

This pipeline duplicates the script `claude-skills/setup/publish-skills.ps1` (analyzed in `code-analysis-setup.md`). The script-form is intended for **local/manual** publishing; the YAML is the **automated** path. Behavioural deltas:

| Aspect | `publish-claude-skills.yml` | `setup/publish-skills.ps1` |
|---|---|---|
| Branch delete on existence | `git branch -D` only if `show-ref` succeeds | Same pattern (uses `try`/`catch`) 🟡 |
| Push remote | `origin` hard-coded | `origin` hard-coded |
| Force push | `--force` | `--force` |
| User identity | `Build Service <build@beck-technology.com>` | Inherits local git config |
| Prefix | `--prefix=claude-skills` | `--prefix=claude-skills` |

🟡 INFERIDO: any divergence in subtree behaviour between local and pipeline would manifest as a `published-skills` branch with a different root tree hash — no drift mechanism is wired up; trust is implicit.

## 4. `publish-testkit.yml` — NuGet release pipeline

### 4.1 Triggers, parameters, variables

```yaml
trigger:
  branches:
    include: [master]
  paths:
    include: [dotnet/BeckTech.QA.TestKit/**]

pr: none

parameters:
  - name: azureArtifactsPath
    type: string
    default: https://pkgs.dev.azure.com/becktech/7a23932a-63a6-4ad9-9313-800a9913891c/_packaging/DESTINI-Web/nuget/v3/index.json

variables:
  solution: 'dotnet/BeckTech.QA.TestKit/BeckTech.QA.TestKit.sln'
  buildConfiguration: 'Release'

pool:
  vmImage: 'windows-latest'
```

🟢:
- Path filter uses `**` (recursive) — any descendant change triggers.
- `azureArtifactsPath` is a runtime parameter, so re-runs of the pipeline can override the target feed without YAML edits. Default points to `DESTINI-Web` feed in org `becktech`, project GUID `7a23932a-63a6-4ad9-9313-800a9913891c`.
- `buildConfiguration: Release` is hard-coded; no Debug variant is published.

### 4.2 Step graph

```
checkout (depth=1)
  └─► UseDotNet@2 (sdk 9.0.x)
        └─► dotnet restore "$(solution)"
              └─► dotnet build --configuration Release --no-restore     [emits .nupkg via GeneratePackageOnBuild]
                    └─► dotnet test --no-build --logger trx --logger console
                          └─► PublishTestResults@2     [condition: succeededOrFailed]
                                └─► CopyFiles@2  ──► $(Build.ArtifactStagingDirectory)/nuget
                                      │            [condition: succeeded AND not PR]
                                      └─► NuGetAuthenticate@1
                                            └─► pwsh: enumerate .nupkg + dotnet nuget push --skip-duplicate
                                                   [condition: succeeded AND not PR]
```

Step-by-step:

| # | Step | Task / Type | Conditions | Confidence |
|---:|---|---|---|---|
| 1 | `checkout: self` | builtin | `fetchDepth: 1` (shallow — full history not needed) | 🟢 |
| 2 | Use .NET 9 SDK | `UseDotNet@2` | `packageType: sdk`, `version: 9.0.x` | 🟢 |
| 3 | Restore | `script` (`dotnet restore`) | unconditional | 🟢 |
| 4 | Build | `script` (`dotnet build --no-restore`) | unconditional; `GeneratePackageOnBuild` in `*.csproj` emits `.nupkg` here | 🟢 |
| 5 | Test | `script` (`dotnet test --no-build --logger trx --logger console;verbosity=detailed`) | unconditional | 🟢 |
| 6 | Publish test results | `PublishTestResults@2` | `condition: succeededOrFailed()` — publishes even when tests fail | 🟢 |
| 7 | Copy nupkgs to staging | `CopyFiles@2` | `condition: and(succeeded(), ne(variables['Build.Reason'], 'PullRequest'))` | 🟢 |
| 8 | Authenticate to NuGet | `NuGetAuthenticate@1` | unconditional | 🟢 |
| 9 | Push to feed | `pwsh` (inline) | `condition: and(ne(variables['Build.Reason'], 'PullRequest'), succeeded())` | 🟢 |

### 4.3 Release versioning model

🟢 CONFIRMADO (from header comment):
- Release version is **driven by the `<Version>x.y.z</Version>` element in each `src/*.csproj`**. Not by git tags, not by build numbers, not by branch.
- To cut a release: bump `<Version>` → commit → merge to `master`.
- `dotnet nuget push --skip-duplicate` makes **same-version re-runs idempotent** (existing version on the feed yields a no-op, not an error).

This makes the pipeline **stateless w.r.t. previous runs** — the source of truth is the `<Version>` field in the csproj at the commit being built.

### 4.4 The `pwsh` push step — wildcard resolution

```pwsh
$packages = Get-ChildItem '$(Build.ArtifactStagingDirectory)/nuget' -Filter *.nupkg
if (-not $packages) { throw "No .nupkg files found in $(Build.ArtifactStagingDirectory)/nuget." }
dotnet nuget push $packages.FullName -s '${{ parameters.azureArtifactsPath }}' -k az --skip-duplicate
if ($LASTEXITCODE -ne 0) { throw "dotnet nuget push failed with exit code $LASTEXITCODE." }
```

🟢 Algorithm:
1. Glob `*.nupkg` from staging dir with `Get-ChildItem` (returns either `$null`, a single object, or an array).
2. If empty: throw — surface "no packages" as a hard failure, not a silent success.
3. Pass **resolved full paths** (`$packages.FullName`, expanded by PowerShell) to `dotnet nuget push`. The CLI receives N concrete paths instead of a glob string.
4. `-k az` selects the `az` API key, which `NuGetAuthenticate@1` wires up to use the Azure Artifacts service connection.
5. `--skip-duplicate` makes re-pushes of the same version a no-op (exit 0, not failure).
6. Explicit `$LASTEXITCODE -ne 0` rethrow — `dotnet nuget push` does not throw a terminating error on its own.

🟢 **Why the wildcard is resolved in PowerShell** (per inline comment, lines 75–80 of YAML):
- `pwsh` does not auto-glob native-command arguments (unlike bash).
- Whether `dotnet nuget push` itself expands `*.nupkg` has varied across NuGet client versions inside SDK patches — earlier regressions have broken publishes.
- Enumerating with `Get-ChildItem` removes that dependency: the script always hands the CLI a list of concrete paths.

This was fixed by PR 14547 (commit `bc5df85`, "Fix publish-testkit NuGet push: resolve nupkg wildcard in pwsh"). Prior version 🟡 INFERIDO relied on glob expansion by the CLI and broke on a SDK patch upgrade.

### 4.5 PR-aware gating

`Build.Reason` is the ADO predefined variable that distinguishes triggers. The path-filter only fires on push to `master`, but the **gating is layered** in two more places:

- `CopyFiles@2`: `and(succeeded(), ne(variables['Build.Reason'], 'PullRequest'))`
- `pwsh` push: `and(ne(variables['Build.Reason'], 'PullRequest'), succeeded())`

🟡 INFERIDO defensiveness: with `pr: none` already set, `Build.Reason == 'PullRequest'` cannot reach these steps under normal triggers; the duplicated check guards against accidental `pr:` re-enablement or manual run of the pipeline against a PR ref. The combined effect: only `IndividualCI` / `BatchedCI` / `Manual` runs ever push.

## 5. Cross-cutting concerns

### 5.1 Secrets and identity

| Pipeline | Mechanism | Notes |
|---|---|---|
| `publish-claude-skills.yml` | `persistCredentials: true` on `checkout`; `git push` uses the ADO build service identity (`Build Service` per git config set inside the step) | No PAT / secret variable in YAML |
| `publish-testkit.yml` | `NuGetAuthenticate@1` task; service connection inferred from `dotnet nuget push -k az` | The `az` API key is a sentinel — `NuGetAuthenticate@1` wires the real credentials into the NuGet config at runtime |

🔴 LACUNA: no service-connection name or `azureSubscription:` is declared inside `publish-testkit.yml`. `NuGetAuthenticate@1` uses **implicit** Azure DevOps service-connection / Build Service identity. To audit who can publish, inspect ADO **project settings → Service connections** (out of scope for static analysis).

### 5.2 Failure behaviour

| Pipeline | Failure surface |
|---|---|
| `publish-claude-skills.yml` | Two explicit `Write-Error; exit 1` paths (subtree split, push). Step fails → job fails. No retries. |
| `publish-testkit.yml` | Each `script:` step inherits PowerShell's "fail on non-zero exit code" default. The push step throws explicitly on `$LASTEXITCODE -ne 0` for clearer error messages. `PublishTestResults@2` runs even on failure (`succeededOrFailed()`). |

### 5.3 Test reporting

Only `publish-testkit.yml` produces test results:
- `dotnet test --logger "trx;LogFileName=test-results.trx"` writes `**/test-results.trx`.
- `PublishTestResults@2` picks them up with `testResultsFormat: VSTest`, `mergeTestResults: true`.

No code-coverage logger is configured. No code-coverage threshold step. 🔴 LACUNA: gating on test pass/fail is implicit — `dotnet test` returning non-zero would fail the step; nothing else enforces coverage or quality bars.

### 5.4 Idempotency model

| Pipeline | Idempotency |
|---|---|
| `publish-claude-skills.yml` | Re-run = recompute split + force-push → branch always matches `claude-skills/` tree at the triggering commit. Safe to re-run. |
| `publish-testkit.yml` | Re-run = same `<Version>` → `--skip-duplicate` → no-op. Bump `<Version>` + commit + merge → produces new feed version. Safe to re-run. |

### 5.5 Coupling to setup scripts

The publish pipelines are **functionally redundant** with manual scripts in `claude-skills/setup/`:

| Pipeline | Manual equivalent | Drift risk |
|---|---|---|
| `publish-claude-skills.yml` | `claude-skills/setup/publish-skills.ps1` | 🟡 Two PowerShell implementations of the same flow; no shared library; comments call out one another but no mechanism enforces parity. |
| `publish-testkit.yml` | (none — no local NuGet publish script) | N/A |
