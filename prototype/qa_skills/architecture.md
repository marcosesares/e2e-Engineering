# Architecture Overview — BeckTech.QA.Tools

> Generated by Reversa Architect · 2026-05-23
> Confidence: 🟢 CONFIRMADO | 🟡 INFERIDO | 🔴 LACUNA

---

## 1. System Purpose

BeckTech.QA.Tools is a **QA automation framework** that packages reusable Claude Code skills, PowerShell scripts, and AI agents as distributable artifacts. It has two primary output channels:

1. **Claude Code skills** — distributed as a git subtree into consuming repos (`.claude/`)
2. **NuGet packages** — `BeckTech.QA.TestKit` and `BeckTech.QA.TestKit.Playwright` published to Azure Artifacts

The system is a **producer**. It has no persistent storage. All state lives in external systems (Jira, ADO, consuming repo filesystems).

---

## 2. Architecture Style

| Attribute | Value | Confidence |
|-----------|-------|------------|
| Pattern | **Composable skill pipeline** — stateless, entry-point-driven | 🟢 |
| Composition model | Higher-order skills call lower-order skills; agents are terminal nodes | 🟢 |
| Distribution | git subtree (skills) + NuGet (TestKit libraries) | 🟢 |
| State persistence | External only (Jira, ADO, consuming-repo filesystem) | 🟢 |
| Runtime | Claude Code session (conversational; single-user) | 🟢 |
| Coupling | Loose — scripts communicate via Standard Output Envelope `{success, data, error}` | 🟢 |
| Concurrency | **Single-user, single-session per repo**. Two parallel `/work-on KEY-A` + `/work-on KEY-B` are safe (per-issue Jira cache isolation). Parallel `/apply-test-plan` runs against the same repo are **unsafe** — no locking on ops JSON / sidecar. | 🟢 |

## 4. Key Components

### 4.1 Orchestration Skills (Entry Points)

| Skill | Role | Key Collaborators |
|-------|------|-------------------|
| `work-on` | Router: dispatches by Jira issue type | `fetch-jira-item`, `fix-qa-bug`, `implement-story` |
| `fix-qa-bug` | Full bug fix lifecycle | `plan-change`, `qa-investigate`, `qa-implement`, `review-change`, `commit-change` |
| `implement-story` | Full story lifecycle | Same as fix-qa-bug |
| `apply-test-plan` | Gate-based plan executor | `create-ado-test-cases`, `update-ado-test-cases`, `set-jira-test-plan-ids.ps1` |
| `plan-feature-coverage` | Feature coverage evaluator | `test-plan`, ADO queries |
| `plan-regression-coverage` | Regression coverage evaluator | `test-plan`, Jira batch query |

### 4.2 Agents

| Agent | Model | Access | Output Contract |
|-------|-------|--------|-----------------|
| `qa-investigate` | Sonnet | Read, Glob, Grep, Bash | `investigation.md` (file) |
| `qa-implement` | Sonnet | Full file I/O | Modified source files |

### 4.3 Script Layer (`claude-skills/scripts/`)

Standard Output Envelope: `{ success: bool, data: object|null, error: string|null }` 🟢
Reconciled with `_reversa_sdd/scripts/legacy-mapping.md` (2026-05-23 Reviewer pass).

| Group | Count | External Target |
|-------|-------|-----------------|
| Jira integration | 7 | Jira Cloud REST API (Basic auth, email:apitoken) |
| ADO integration | 14 | Azure DevOps REST API (Basic auth, :PAT) |
| Test Plan Generation | 1 | Local filesystem (markdown renderer; no HTTP) |

Setup/distribution scripts (`generate-manifest.ps1`, `publish-skills.ps1`, `sync-shared-skills.ps1`, `check-updates.ps1`, `guard-shared-skills.ps1`) live under `claude-skills/setup/` and belong to the Setup unit — see `_reversa_sdd/setup/legacy-mapping.md`.

### 4.4 Distribution Infrastructure

| Component | Mechanism | Trigger |
|-----------|-----------|---------|
| `publish-skills.ps1` | `git subtree split --force-push` | Manual / CI |
| `publish-claude-skills.yml` | **Azure DevOps Pipelines** | `push to master` touching `claude-skills/*`; `pr: none` |
| `sync-shared-skills.ps1` | `git subtree pull --squash` | Manual (in consumer repo) |
| `guard-shared-skills.ps1` | PreToolUse hook (exit 2) | Claude Code edit attempt |
| `check-updates.ps1` | SessionStart hook (4h TTL) | Each Claude Code session |

## 5. Data Flows

### 5.1 Bug Fix / Story Implementation

```
User → /work-on QA-123
  → fetch-jira-item (fetches + caches in ~/.claude/fetch-jira-item/QA-123/)
  → route by issue_type
  → fix-qa-bug / implement-story
      → plan-change (user approval gate)
      → qa-investigate (read-only codebase scan)
      → qa-implement (code changes)
      → review-change (user approval gate)
      → commit-change → commit to git + Jira comment
```

### 5.2 Test Plan Application

```
User → /apply-test-plan <path>
  → locate + parse Markdown plan → sidecar JSON
  → validate required fields
  → user approval gate
  → create-ado-test-cases.ps1 (Operations JSON → ADO WIT API)
  → update-ado-test-cases.ps1 (steps, state transition)
  → [feature plans only] set-jira-test-plan-ids.ps1 (writeback to Jira)
```

### 5.3 Skill Distribution

```
Maintainer → edits claude-skills/ → PR → merge to master
  → publish-claude-skills.yml (Azure DevOps Pipelines, push trigger on master)
  → git subtree split -b published-skills --force-push
  → published-skills branch updated

Consumer repo:
  check-updates.ps1 (SessionStart, 4h TTL)
  → git show + delta check
  → [updates available] user notified → manual sync
  → sync-shared-skills.ps1 → git subtree pull --squash
```

## 7. Technical Debts

| ID | Description | Severity | Confidence |
|----|-------------|----------|------------|
| **TD-01** | Jira auth inconsistency: 3 scripts require explicit `-Pat` parameter; 4 others use env vars. No unified credential resolver. | Medium | 🟢 |
| **TD-02** | No automated tests for `guard-shared-skills.ps1`, `sync-shared-skills.ps1`, or `check-updates.ps1`. Behavior verified manually only. | High | 🟢 |
| **TD-03** | `create-ado-test-cases.ps1` is **not idempotent** — re-run on same input creates duplicate test cases. No deduplication guard. | High | 🟢 |
| **TD-04** | Jira Cloud ID hardcoded (`ec1222f5-ecd6-4b14-a4b4-68c3ed82ef14`). Multi-tenant or org migration requires code change. | Low | 🟢 |
| **TD-05** | SHARED_MANIFEST sync has no checksum validation — relies entirely on path list integrity. File content drift undetectable. | Medium | 🟡 |
| **TD-06** | ADO PAT required scopes not formally documented for PR create/update operations. | Low | 🟢 |
| **TD-07** | TestKit source (`dotnet/BeckTech.QA.TestKit/`) deleted **in working tree only** (197 files in `git status ` `D` state; no removal commit on master). Pipeline `publish-testkit.yml` is therefore still active until a deletion commit lands. | Medium | 🟢 |
| **TD-08** | `notify.ps1`: 13-line Windows-only balloon toast via `System.Windows.Forms.NotifyIcon`. Hardcoded title `'Claude Code'`, body `'Waiting for your input'`. No PII, no network. macOS/Linux users have no equivalent — opt-in only via `/onboard` Step 7. | Low | 🟢 |
| **TD-09** | `QA_ALLOW_SHARED_EDIT=1` bypass is undocumented in consumer-facing docs. Maintainers discover it by reading guard script source. | Low | 🟡 |

## 8. Architectural Decisions Summary

Key ADRs extracted from Git history (full content in `_reversa_sdd/adrs/`):

| ADR | Decision | Rationale |
|-----|----------|-----------|
| 001 | git subtree for skill distribution | No git submodule tracking overhead; consumers get a clean directory |
| 002 | Skills as Markdown (SKILL.md) | Engine-agnostic; readable by humans and AI; no binary dependency |
| 003 | Multi-phase ADO test case creation | Single `wit_create_work_item` call with all required fields avoids TF401320 |
| 004 | Fail-open guard posture | Structural errors in guard script never block legitimate edits |
| 005 | 4-hour TTL for update cache | Balances freshness with session-start latency |
| 006 | Per-artifact version markers | Enables `refresh-setup` to detect stale generated files independently |
| 007 | PowerShell glob for nupkg | Avoids NuGet SDK version sensitivity in wildcard expansion |
| 008 | Sidecar schema alignment | `apply-test-plan` sidecar reuses Operations JSON schema; no re-serialization |
| 009 | `.csproj` version as release source | Version of record lives in source; no tag/build-number lookup needed |
| 010 | Set-based delta for sync | Detects renames correctly where file-count comparison fails |
