The tests use C# helpers like `teststep_AddBidderToPackage_CommandService` and the actual endpoint paths are not visible in the test corpus. None of the helpers expose them as strings reachable by the reviewer in this session.
**Spec afetada:** `_reversa_sdd/Bidders/flows.md`, `_reversa_sdd/Bidders/design.md`, similar API claims wherever they appear.
**Pergunta:** Where can the canonical CommandService endpoint catalog be found (OpenAPI spec, controller files in the API project, swagger.json artifact)? Without that, all endpoint paths in the specs should be downgraded to 🟡 or removed.
**Impacto:** Bidders/flows.md and other endpoint claims demoted to 🟡 INFERIDO; once the catalog is provided they can be restated as 🟢.

**Resposta:** The canonical CommandService endpoint catalog is **inside the test repo itself** at `TestSteps/TestStep.CommandServiceAPI.cs` — every endpoint the tests touch is hard-coded there as a string literal. Catalog extracted from grep of `"/api/"` in that file:
- **Preferences:** `PUT /api/Preferences`
- **BidDay System Settings:** `POST /api/BidDaySystemSettings` (10+ distinct command bodies — single endpoint, `commandBody` discriminator)
- **Users:** `POST /api/User`, `PUT /api/User`
- **Project:** `POST /api/Project/create-project`, `POST /api/Project/update-project-alternate-enabled-state`, `POST /api/Project/fees`, `POST /api/project/update-project-bidders-ranges`, `POST /api/project/close-project`
- **BidPackage:** `POST /api/bid-package/add-bidders`, `POST /api/bid-package/set-adjustment`, `POST /api/bid-package/add-adjustment-notes`, `POST /api/bid-package/notes`, `POST /api/bid-package/add-bidder-notes`, `POST /api/bid-package/file-uploads/upload-bidder-document`, `POST /api/bid-package/add-line-items`, `POST /api/bid-package/update-line-items`, `POST /api/bid-package/set-bidder-requirement-fulfillment`, `POST /api/bid-package/set-line-item-bid`, `POST /api/bid-package/set-line-item-rollup-bid`, `POST /api/bid-package/set-contingency`, `POST /api/bid-package/set-selected-bidder`, `POST /api/bid-package/update-lock`

Corrections for the corpus:
- "`/commands/add-bidder`" / "`/commands/update-bidder`" / "`/commands/delete-bidder`" in `Bidders/flows.md` are **invented** — real path is `POST /api/bid-package/add-bidders` (plural, scoped to bid-package). No update-bidder or delete-bidder endpoint is visible in the test catalog. Bidder updates likely go through a re-`add-bidders` call or another endpoint not exercised by tests; the bidder-delete flow uses UI confirmation (`teststep_DeleteBidderAsync`) without an explicit `/api/` string visible — needs verification.
- Almost every command is `POST` — `PUT` is used only for `/api/Preferences` and `/api/User`.
- Auth flow at `TestStep.CommandServiceAPI.cs:18-36`: `CommandServiceUtility.Login(email, password)` returns `CommandServiceTokens`; subsequent calls send the token; on `401` the helper re-acquires and retries (`:60-64`). This is ROPC (password-grant) against `_testConfiguration.ropcUrl` with `applicationId` + `scopeUrl`. Specs that name "Azure AD B2C" are extrapolating from ROPC parameters — keep them 🟡 until confirmed.
- Rewrite endpoint claims in every `flows.md` against this catalog; downgrade unverified backend claims (event-store schema, role-permissions table) to 🟡. 🔴→🟢.
