## Canonical Glossary

> Source of truth for all terminology used across this project. Extracted from `CONTEXT.md`, commit messages, `.out-of-scope/` entries, and skill content.

### Core entities

**Issue tracker** 🟢
The tool that hosts a repo's issues — GitHub Issues, GitLab Issues, local `.scratch/` markdown convention, or a user-described custom workflow. Skills like `to-issues`, `to-prd`, and `triage` read from and write to it.
_Avoid_: backlog manager, backlog backend, issue host, "backlog" as a noun for the tool.
_Historical note_: the term "backlog" was used until commit `179a14e` (Apr 28 2026) when it was formally replaced with "issue tracker" across CONTEXT.md and all skill files.

**Issue** 🟢
A single tracked unit of work inside an Issue tracker — a bug, task, PRD, or vertical slice produced by `to-issues`.
_Avoid_: ticket (use only when quoting external systems that call them tickets).

**Triage role** 🟢
A canonical state-machine label applied to an Issue during triage. Each role maps to a real label string in the Issue tracker via `docs/agents/triage-labels.md`. Two categories of roles exist: **state roles** and **category roles** (see state machines).

**Skill** 🟢
A self-contained agent instruction set stored as a folder with a `SKILL.md`. The "code" of this repository. Each skill defines a workflow an AI agent executes when invoked.

**Bucket** 🟡
An organizational category for skills: `engineering`, `productivity`, `misc`, `in-progress`, `deprecated`, `personal`. Not a formal domain term but used consistently in CLAUDE.md and READMEs.

**Agent Brief** 🟢
A structured comment posted on a GitHub issue when it moves to `ready-for-agent`. The authoritative specification that an AFK agent works from. Durable by design: no file paths, no line numbers.
_Key constraint_: must be behavioral ("what the system should do"), not procedural ("how to implement it").

**Deep module** 🟢
A module with a small interface and a lot of implementation (high leverage). Defined formally in `tdd/deep-modules.md`. The opposite of a **shallow module** (large interface, thin implementation).

**Seam** 🟢
Where an interface lives — a place behavior can be altered without editing in place. Defined in `improve-codebase-architecture/LANGUAGE.md`. Use this term; avoid "boundary."

**Vertical slice** 🟢
A thin but complete path through all integration layers (schema, API, UI, tests) that is demoable on its own. Used in `tdd` and `to-issues`. Opposite of a **horizontal slice** (one layer across all features).

**Tracer bullet** 🟢
The first vertical slice — the one test + minimal implementation that proves the end-to-end path works. Term used in `tdd` and `to-issues`.

**Beat** 🟡
A single narrative move in a `writing-beats` session. Sized by what it needs: one sentence to multiple paragraphs. Stops at the point where the next beat can pivot.

**Fragment** 🟡
Any piece of text in `writing-fragments` that might survive into a final article. Heterogeneous: claims, vignettes, sharp sentences, half-thoughts, quotes.

**AFK agent** 🟢
An AI agent that picks up a `ready-for-agent` issue and implements it without human interaction. Contrasted with HITL (Human-In-The-Loop) work.

**HITL** 🟢
Human-In-The-Loop. Work that requires human interaction (architectural decisions, design reviews, external access). Used in issue slices typed as `HITL` vs `AFK`.

**Sandcastle** 🟡
A TypeScript library for orchestrating coding agents in isolated sandboxes. Provides: a `.sandcastle` directory, Dockerfile/sandbox definition, environment variables, GitHub issues as backlog source, and a planner → implementer(s) → reviewer → merger agent pipeline. Enables AFK factory-style parallel execution. External to the skills library itself.
_Source_: tutorial Part 8.

**Git worktree** 🟡
A mechanism for checking out multiple branches of the same repository in separate folders simultaneously. Allows multiple agents to work independently on different features without interfering. Each worktree has its own branch, its own changes, and can have its own agent. Key safety rule: agents must push to the specific feature branch, not `main`.
_Source_: tutorial Part 9.

**research.md** 🟡
A temporary file used to cache research results within a session. Created when an agent needs to explore an unfamiliar domain before implementing. Intentionally transient — deleted or archived after use, not kept as permanent documentation (to avoid stale research infecting future sessions).
_Source_: tutorial (step 4 of complete workflow).

**QA plan** 🟡
A structured checklist produced at the end of an implementation cycle for human QA. Contents: core happy path, edge cases, regression checks, visual checks, data integrity checks, error states, performance concerns, accessibility. The QA plan closes the implementation loop — human QA findings become new issues that re-enter the triage pipeline.
_Source_: tutorial Part 12.

**Standards review** 🟢
One of two parallel axes in the `/review` skill. Checks: code conventions, module boundary (seam) compliance, test meaningfulness, absence of unnecessary abstractions, scope discipline. A change can pass spec review but fail standards review.
_Source_: tutorial Part 10.

**Spec review** 🟢
One of two parallel axes in the `/review` skill. Checks: implementation satisfies the issue/PRD, all acceptance criteria met, user stories preserved. A change can pass standards review but fail spec review.
_Source_: tutorial Part 10.

## Domain Relationships

```
Issue tracker  1──* Issue
Issue          1──1  state_role (TriageStateRole)
Issue          1──1  category_role (CategoryRole)
Issue          0──1  AgentBrief        (when state = ready-for-agent)
Issue          0──1  TriageNotes       (when state = needs-info)

Skill folder   1──1  SKILL.md (required)
Skill folder   0──*  supporting .md files
Skill folder   0──1  scripts/ directory

Bucket folder  1──*  Skill folder
Bucket folder  1──1  README.md

Plugin registry (.claude-plugin/plugin.json)  *──*  Skill (published only)
link-skills.sh  *──*  Skill (non-deprecated only → ~/.claude/skills/)
```

## Implicit Business Rules (extracted from code + git history)

### Skill authoring rules
🟢 `description` field in SKILL.md frontmatter must be ≤ 1024 characters and include "Use when [specific triggers]".
🟢 SKILL.md should target ≤ 100 lines; split into supporting files when exceeded.
🟢 `disable-model-invocation: true` marks a skill whose SKILL.md content IS the prompt — no AI reasoning occurs on load (used only by `zoom-out` and `setup-matt-pocock-skills`).
🟡 `<what-to-do>` and `<supporting-info>` XML sections separate imperative instructions from background context (pattern introduced in `writing-*` skills and `grill-with-docs`, commit `b843cb5`).

### Publication and linking rules
🟢 Only skills in `engineering/` and `productivity/` are listed in `.claude-plugin/plugin.json` (the skills.sh publication registry).
🟢 `misc/` skills exist in the repo but are NOT published via plugin.json.
🟢 `deprecated/` skills are excluded from `link-skills.sh` (added explicitly in commit `494e4b2`).
🟢 `deprecated/` and `personal/` skills must not appear in the top-level `README.md` or `plugin.json`.
🟢 Every skill in `engineering/`, `productivity/`, or `misc/` must have an entry in the top-level `README.md` AND in `plugin.json`.

### Issue tracker support scope
🟢 Only **mainstream** issue trackers receive first-class support: GitHub, GitLab, local-markdown.
🟢 Niche/new/single-vendor trackers are out of scope — maintenance cost is only justified for tools "a typical engineer would recognise." (`.out-of-scope/mainstream-issue-trackers-only.md`, issue #99).
🟢 Escape hatches exist for non-mainstream: `local markdown` or `other/custom` (user describes workflow as prose).

### Grilling rules
🟢 All grilling sessions (grill-me, grill-with-docs, triage) ask one question at a time — never batch.
🟢 No maximum question cap on grilling sessions — open-ended by design. (`.out-of-scope/question-limits.md`, issue #44).
🟢 If a question is answerable by exploring the codebase, explore instead of asking.

### Domain documentation rules
🟢 `CONTEXT.md` is a **glossary only** — no implementation details, no specs, no decisions. (Reinforced in commit `e74f006`, May 13 2026).
🟢 ADRs are created only when ALL THREE conditions are true: hard-to-reverse + surprising-without-context + genuine trade-off.
🟢 `CONTEXT.md` is created lazily — only when the first term is resolved; not pre-emptively.

### Triage rules
🟢 Every triaged issue carries exactly one state role AND one category role.
🟢 Every AI-generated triage comment must begin with `> *This was generated by AI during triage.*`
🟢 `.out-of-scope/` matching uses **concept similarity**, not keyword matching.
🟢 `.out-of-scope/` only records rejected **enhancements** (not bugs).

### Prototype rules
🟢 Prototypes are throwaway — must be deleted or absorbed ("delete-or-absorb lifecycle").
🟢 Logic prototype: pure logic module (reducer/state-machine/functions/class) is separate from TUI shell. TUI is deleted; logic module may be lifted into production.
🟢 UI prototype: hidden in production builds (`NODE_ENV !== 'production'` gate on switcher).

### TDD rules
🟢 No horizontal slices (all tests, then all code). Vertical slices only (1 test → 1 impl, repeat).
🟢 Tests must exercise public interfaces only — never private methods or internal state.
🟢 Never refactor while RED (failing test). Get to GREEN first.

### Pre-commit / shoehorn rules
🟢 `shoehorn` (`fromPartial`, `fromAny`, `fromExact`) is test code only — never production.
🟢 `setup-pre-commit`: `.prettierrc` creation is idempotent — only create if no Prettier config exists.

### Writing skills rules
🟢 All writing skills (`writing-beats`, `writing-shape`, `writing-fragments`) re-read the target file from disk before every write — user edits between turns are authoritative.
🟢 `writing-shape`: raw material file is read-only — never edit it.
🟢 `writing-beats`: write one beat at a time. End = natural end of journey, NOT empty pile.
🟢 `writing-fragments`: capture fragments from the very first user prompt (initial message).

## Gaps and Lacunas

🔴 **No automated governance**: there is no CI or linter that enforces the CLAUDE.md rules (e.g. verifying that every engineering/productivity skill appears in plugin.json). Compliance is manual and relies on contributor discipline.

🔴 **Skill versioning undefined**: no version field in SKILL.md frontmatter. No mechanism to pin or track changes to individual skills (beyond git history).

🔴 **No skill testing framework**: skills are natural language — there is no automated regression test suite to verify they still behave correctly after edits.

🔴 **in-progress graduation criteria undefined**: no documented threshold for when an `in-progress` skill graduates to `engineering/` or `productivity/`. The `review`, `writing-beats`, `writing-fragments`, `writing-shape` skills have no stated graduation condition.

🟡 **CONTEXT-MAP.md undefined for this repo**: `grill-with-docs` and `improve-codebase-architecture` reference `CONTEXT-MAP.md` for multi-context repos, but the skills repo itself has no CONTEXT-MAP.md — it uses a single CONTEXT.md.
