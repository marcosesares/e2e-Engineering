## 1. Resumo executivo

The `engineering/` bucket delivers 10 AI agent skills that assist developers during code-work sessions. It covers the complete engineering workflow: diagnosing bugs, writing tests (TDD), triaging issues, converting requirements to PRDs and GitHub/GitLab issues, prototyping, architecture improvement, and repository setup. Skills are natural-language programs executed by Claude Code (or equivalent agents) and must produce correct, traceable outputs when invoked in any mainstream-tracker repository.

## 7. Critérios de Aceitação

```gherkin
Cenário: Triage — issue transitioning to ready-for-agent
  Dado a GitHub issue is in needs-triage state with exactly 1 category label
  Quando the maintainer invokes /triage and classifies the issue as fully specified and AFK-safe
  Então the state label changes to ready-for-agent
    E exactly 1 state label exists on the issue
    E an Agent Brief comment is posted, opening with "> *This was generated by AI during triage.*"

Cenário: Triage — rejected enhancement creates out-of-scope entry
  Dado a GitHub issue is classified as category=enhancement
  Quando the maintainer invokes /triage and closes it as wontfix
  Então a file is created at .out-of-scope/<concept>.md
    E the issue label changes to wontfix
    E no .out-of-scope entry is created for bugs

Cenário: setup-matt-pocock-skills — idempotency on re-run
  Dado triage-labels.md already exists in the target repo
  Quando /setup-matt-pocock-skills is invoked again
  Então triage-labels.md is not overwritten
    E the user is informed the file was skipped

Cenário: to-issues — tracer bullet first
  Dado a PRD document is provided to /to-issues
  Quando to-issues creates vertical slices
  Então the first issue created is the tracer bullet
    E subsequent issues are ordered by layer (schema → api → ui → test)

Cenário: tdd — no horizontal slices
  Dado a developer is working on a new feature using /tdd
  Quando the developer writes the first test
  Então the skill requires that test to pass before writing the second test
    E the skill never suggests writing all tests upfront

Cenário: prototype — delete-or-absorb enforcement
  Dado a logic prototype TUI shell is running
  Quando the prototype answer is captured
  Então the TUI shell is deleted
    E the logic module may be promoted to production independently
```
