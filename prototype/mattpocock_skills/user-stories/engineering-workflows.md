## Epic: Repository Setup

### US-01 — First-time setup (🟢)
**As a** developer setting up a new repository for the first time,
**I want** to run a single setup command that seeds all the necessary configuration,
**So that** all engineering skills work correctly without manual configuration.

**Acceptance criteria**:
- [ ] `triage-labels.md` is created with correct label strings for my issue tracker
- [ ] `CONTEXT.md` is created as an empty glossary seed
- [ ] `docs/adr/` directory structure is created
- [ ] Re-running setup does not overwrite any existing files
- [ ] I am informed which files were created and which were skipped

## Epic: Requirements Elicitation

### US-02 — Elicit requirements from scratch (🟢)
**As a** developer starting a new feature with only a vague idea,
**I want** to be interviewed one question at a time,
**So that** I end up with a complete, unambiguous requirements spec without having to know the right questions upfront.

**Acceptance criteria**:
- [ ] I receive exactly one question per turn
- [ ] The session continues until the spec is complete (no artificial cap)
- [ ] Questions the AI can answer by reading the codebase are skipped
- [ ] I receive a structured output at the end of the session

### US-03 — Elicit requirements using existing documentation (🟢)
**As a** developer with an existing codebase and ADRs,
**I want** the elicitation to read and use my existing documentation first,
**So that** I'm not re-explaining things that are already documented.

**Acceptance criteria**:
- [ ] `grill-with-docs` reads CONTEXT.md, ADRs, and relevant docs before asking any question
- [ ] Questions that are answerable from existing docs are not asked
- [ ] The output incorporates domain vocabulary from CONTEXT.md

## Epic: PRD and Issue Creation

### US-04 — Convert requirements to a PRD (🟢)
**As a** developer or product owner,
**I want** to convert freeform requirements into a structured PRD,
**So that** I have a traceable specification before breaking work into issues.

**Acceptance criteria**:
- [ ] Input can be freeform text or an existing document
- [ ] Output contains: title, user stories (role/want/so-that), acceptance criteria
- [ ] The PRD is immediately usable as input to `to-issues`
- [ ] If information is missing, I am asked one question at a time

### US-05 — Break a PRD into vertical-slice issues (🟢)
**As a** developer with a PRD,
**I want** the PRD broken into GitHub/GitLab issues ordered as vertical slices,
**So that** I can implement the feature incrementally with a working end-to-end path from the first issue.

**Acceptance criteria**:
- [ ] First issue created is the tracer bullet (thinnest end-to-end path)
- [ ] Subsequent issues are ordered by layer (schema → api → ui → test)
- [ ] Each issue is typed: AFK (automatable) or HITL (requires human)
- [ ] Issues are created in my configured tracker (GitHub / GitLab / local-markdown)

## Epic: Issue Triage

### US-06 — Triage a newly opened issue (🟢)
**As a** maintainer,
**I want** to triage an issue through a structured state machine,
**So that** every issue ends up with the correct labels and the appropriate side effects (Agent Brief, Triage Notes, out-of-scope entry).

**Acceptance criteria**:
- [ ] Issue ends with exactly 1 state role label + exactly 1 category role label
- [ ] `ready-for-agent`: Agent Brief comment posted, opens with AI-generated header
- [ ] `needs-info`: Triage Notes comment posted, opens with AI-generated header
- [ ] `wontfix` (enhancement): `.out-of-scope/<concept>.md` created
- [ ] `wontfix` (bug): polite explanation only, no .out-of-scope entry

### US-07 — Detect a duplicate enhancement request (🟢)
**As a** maintainer,
**I want** the triage skill to check whether a new enhancement duplicates something already rejected,
**So that** I don't spend time re-evaluating already-rejected work.

**Acceptance criteria**:
- [ ] Before grilling, the skill scans `.out-of-scope/` for concept-similar entries
- [ ] If a match is found, the maintainer is informed and given the option to close immediately
- [ ] Matching is based on concept similarity, not keyword matching

## Epic: Test-Driven Development

### US-08 — Implement a feature test-first (🟢)
**As a** developer using TDD,
**I want** the AI to enforce the red-green-refactor discipline,
**So that** I never write implementation code without a failing test first.

**Acceptance criteria**:
- [ ] The skill never suggests writing more than one test before implementation
- [ ] Refactoring is only suggested while tests are GREEN
- [ ] The tracer bullet is the first slice implemented
- [ ] Tests target public interfaces only — never private methods

## Epic: Bug Diagnosis

### US-09 — Diagnose a bug systematically (🟢)
**As a** developer investigating a bug,
**I want** the AI to walk me through a structured 6-phase diagnostic process,
**So that** I confirm the root cause before proposing a fix.

**Acceptance criteria**:
- [ ] All 6 phases complete before any fix is suggested
- [ ] Repro is confirmed before hypotheses are generated
- [ ] If repro fails, additional repro info is requested

## Epic: Architecture and Design

### US-10 — Prototype a logic or UI design question (🟢)
**As a** developer facing a design question,
**I want** to build a throwaway prototype to answer it,
**So that** I have empirical evidence before committing to a production implementation.

**Acceptance criteria**:
- [ ] Logic questions → TUI shell prototype; TUI shell deleted after answer; logic module may be promoted
- [ ] UI questions → 2-3 variants behind a `NODE_ENV !== 'production'` gate; winner promoted; losers deleted
- [ ] No prototype artifact remains in production after the design question is answered

### US-11 — Improve codebase architecture incrementally (🟢)
**As a** developer working in a complex codebase,
**I want** the AI to help me identify seams and refactor toward deeper modules,
**So that** the architecture improves iteratively without a big-bang rewrite.

**Acceptance criteria**:
- [ ] Skill identifies seams in the area under review
- [ ] Each component assessed: deep or shallow module?
- [ ] Refactor proposed; applied if agreed; code re-read after each iteration
- [ ] Canonical vocabulary used: "seam", "deep module", "shallow module"
