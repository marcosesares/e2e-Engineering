## Deployment Overview

Superpowers is a **zero-dependency plugin/framework** distributed via:
1. **Plugin manifests** (per harness: Claude Code, Codex, Cursor, OpenCode, Gemini)
2. **Git repository** (skills stored as versioned Markdown)
3. **Bootstrap loader** (loaded at harness session start)

**No external runtime, package managers, or third-party services required.**

## Installation Models

### **Model 1: Plugin Marketplace (Preferred)**

```
User Flow:
  1. Open Claude Code / Codex App / Cursor IDE
  2. Go to Extensions / Plugins / Marketplace
  3. Search: "superpowers"
  4. Click Install
  5. Extension/Plugin auto-installed
     └─ .claude/plugins/superpowers/ (or harness-specific dir)
  6. At session start, bootstrap auto-loads
  7. Skills ready to use

Maintainer Flow:
  1. Update plugin.json in git repo
  2. Tag release (e.g., v5.1.0)
  3. CI/CD publishes to marketplaces:
     - Claude Code Plugin Registry
     - Codex Plugin Marketplace
     - Cursor Extension Store
     - OpenCode Plugin Hub
     - Gemini CLI Extensions
  4. Users auto-notified of update
  5. Auto-install or manual update available
```

### **Model 2: Git Submodule / Symlink**

```
User Flow:
  1. cd /my/project
  2. git submodule add https://github.com/obra/superpowers
     └─ Creates superpowers/ directory
  3. ln -s ../superpowers/skills .claude/skills/
     (or manual copy of skills/ directory)
  4. Add symlink to .gitignore
  5. At session start, bootstrap finds skills via search path
  6. Skills available

Advantages:
  - Full version control (can pin to specific commit)
  - Easy to test local changes
  - No marketplace dependency
  
Disadvantages:
  - Extra setup steps
  - Need to manage symlinks
  - Less discoverable than marketplace
```

### **Model 3: CLI Wrapper ❌ NOT SUPPORTED**

```
❌ Rejected approaches:
  - npx superpowers (runtime wrapper)
  - Manual skill file copying
  - User opt-in per session
  - Runtime shim before agent starts

Why: Bootstrap contract requires auto-load at session start.
     These methods don't auto-trigger skills, so they're dead weight.
```

## Bootstrap Loading Sequence

### **Claude Code: Session Start**

```
1. User opens Claude Code (CLI or IDE)
   ├─ Load harness config
   └─ Detect working directory (project root)

2. Harness reads: .claude-plugin/plugin.json
   ├─ bootstrap.skill = "using-superpowers"
   ├─ bootstrap.auto_load = true
   └─ bootstrap.trigger = "session_start"

3. Harness triggers session_start event
   ├─ Create new session context
   └─ Register skill search paths: [".claude/skills/", ".agents/skills/"]

4. Load skill: using-superpowers
   ├─ Search: .claude/skills/using-superpowers/SKILL.md
   ├─ Parse: triggers, dependencies
   ├─ Inject: harness type, available tools
   └─ Render: skill workflow to agent

5. Skill: using-superpowers
   ├─ Define trigger rules (intent → skill_id mappings)
   ├─ Register tool abstractions
   ├─ Initialize session state
   └─ Agent is ready for user input

6. User sends message: "Let's make a dashboard"
   ├─ Agent extracts intent: "make" → design
   ├─ Trigger matcher finds: "brainstorming" skill
   ├─ Load brainstorming/SKILL.md
   ├─ Render 9-step design workflow
   └─ Agent executes brainstorming phases
```

## Version Management & Updates

### **Semantic Versioning**

- **Major (5.x.x):** Breaking changes (skill removal, gate restructure, harness contract change)
- **Minor (5.1.x):** New features (new skill, new trigger, new harness support)
- **Patch (5.1.0):** Bug fixes (red flag wording, gate logic fix, doc clarification)

### **Update Notification**

```
Mechanism: User sees message at session start
  "💡 New version of Superpowers available: 5.2.0"
  "Run: npx superpowers update"

Update Command:
  npx superpowers update [--major] [--target-version X.Y.Z]
  
  Options:
    --major      Allow breaking changes
    --dry-run    Preview changes without applying
    --harness    Update only for this harness (claude-code, codex, etc.)
```

### **Backward Compatibility**

- New skills: additive, no breakage
- Modified skills: must maintain existing phases/gates (append new content)
- Removed skills: major version bump only
- Gate changes: major version bump if preconditions change

## Security & Supply Chain

### **No External Dependencies**

- **Dependency scanning:** `npm audit` returns 0 vulnerabilities (no packages)
- **Supply chain risk:** Minimized (all code in git repo, peer-reviewable)
- **Malware surface:** Reduced (no third-party packages to compromise)

### **Code Review Gates**

Every PR to core:
1. Read CLAUDE.md § "If You Are an AI Agent"
2. Search for existing PRs (duplicates rejected)
3. Complete PR template (all sections filled)
4. Show diff to human partner
5. Get explicit approval before submitting

**94% rejection rate** = deliberate quality signal.

### **Skill Content Integrity**

- Skills are markdown, version-controlled in git
- Changes tracked via git blame
- Red flags tables: intentionally tuned, high bar to modify
- All skill changes require eval evidence (before/after metrics)

## Disaster Recovery & Rollback

### **Scenario: Bad Skill Release (e.g., TDD gate broken)**

```
Detection:
  1. User reports: "Gate not enforced, code bypassed test"
  2. Team investigates: git log shows recent TDD skill change
  3. Root cause: phase_id typo in YAML frontmatter

Recovery Options:
  A. Immediate rollback (recommended)
     - Revert commit in git
     - Tag: v5.1.0-rollback
     - Publish update notification: "Security: downgrade to 5.0.5"
     
  B. Hotfix (if minor issue)
     - Fix: correct YAML typo
     - Test: run evals against fixed skill
     - Release: v5.1.1 with fix
     
  C. Graceful deprecation (if change is expected)
     - Announce: "Behavior change in 5.1.1"
     - Provide: opt-out in user preferences
     - Timeline: 2 releases before removal

Communication:
  - GitHub release notes
  - Email notifications to users
  - In-session warnings: "⚠️ Update recommended"
  - Clear diff in release: "Changed lines X-Y in SKILL.md"
```

## Monitoring & Observability

### **Available Metrics**

- **Skill invocation frequency** (trigger matches per session)
- **Gate enforcement rate** (gates passed vs overridden)
- **Red flag detection rate** (anti-patterns caught)
- **Checkpoint persistence success** (checkpoints saved/restored)
- **Tool abstraction success rate** (git, bash, file ops success)
- **Subagent dispatch success** (agents spawned / completed)

### **Logging**

All logged to:
- **Project-local checkpoints** (.reversa/state.json or equiv)
- **Git commit history** (who changed what, when)
- **User session transcript** (agent output in harness)

### **No Telemetry**

- ❌ No external metrics collection
- ❌ No user tracking
- ❌ No behavioral analytics sent outside project
- ✅ All data remains local or in user's git history
