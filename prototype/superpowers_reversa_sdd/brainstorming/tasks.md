# Brainstorming — Implementation Tasks

> **Feature:** Brainstorming Ideas Into Designs  
> **Generated by:** reversa-writer  
> **Date:** 2026-05-18  
> **Scope:** Tasks represent the execution checklist for an agent using the brainstorming skill to develop a feature spec.

---

## Prerequisites

- [ ] Brainstorming skill file (`skills/brainstorming/SKILL.md`) loaded and accessible
- [ ] Visual companion guide available (optional, for complex visual topics) — `skills/brainstorming/visual-companion.md`
- [ ] Project context readable (files, docs, git history accessible)
- [ ] User available for synchronous feedback (design validation requires user input)
- [ ] Git repository initialized (design spec will be committed)
- [ ] Output directory for spec file exists (`docs/superpowers/specs/` or user-configured location)

## Tasks

### Phase 1: Project Context Exploration

- [ ] **T-01: Read current project state**
  - Origin in legacy code: `skills/brainstorming/SKILL.md:22-23` (Explore project context step)
  - Done criterion: At least one of the following completed: (1) read key project files/docs, (2) reviewed recent git commits, (3) understood current architecture or constraints
  - Confidence: 🟢

- [ ] **T-02: Assess project scope early**
  - Origin: `skills/brainstorming/SKILL.md:73-74`
  - Done criterion: Scope assessed; if multi-subsystem, agent flags it and proposes decomposition
  - Confidence: 🟢

- [ ] **T-03: Detect over-scoped projects (if applicable)**
  - Origin: `skills/brainstorming/SKILL.md:72-74` (Scope Decomposition section)
  - Done criterion: If project spans 3+ independent subsystems, halt detailed questions and decompose; restart with first sub-project
  - Confidence: 🟢

### Phase 2: Visual Companion Setup (Conditional)

- [ ] **T-04: Offer visual companion (if visual questions anticipated)**
  - Origin: `skills/brainstorming/SKILL.md:146-154`
  - Done criterion: If mockups, wireframes, or diagrams will be discussed, standalone offer message sent; user consent obtained before proceeding
  - Confidence: 🟢

- [ ] **T-05: Load visual companion guide (if accepted)**
  - Origin: `skills/brainstorming/visual-companion.md` (external reference)
  - Done criterion: If user consents to visual companion, guide read and understood
  - Confidence: 🟡

---

### Phase 3: Clarifying Questions (Sequential)

- [ ] **T-06: Ask clarifying questions one-at-a-time**
  - Origin: `skills/brainstorming/SKILL.md:25` (Checklist item 3)
  - Done criterion: All questions asked and answered; agent has clear understanding of: purpose, constraints, and success criteria
  - Acceptance criteria:
    - Only one question per message (no bundling)
    - Wait for user response before next question
    - At least 3–5 questions asked (or fewer if user scope is very simple)
  - Confidence: 🟢

---

### Phase 4: Approach Proposals

- [ ] **T-07: Propose 2–3 alternative approaches**
  - Origin: `skills/brainstorming/SKILL.md:26` (Checklist item 4)
  - Done criterion: 2–3 distinct approaches presented, each with trade-offs, implementation effort, and agent recommendation
  - Acceptance criteria:
    - Each approach clearly differentiated
    - Trade-offs documented (e.g., simplicity vs. extensibility, speed vs. maintainability)
    - Agent recommendation stated with reasoning
    - User selects preferred approach
  - Confidence: 🟢

---

### Phase 5: Design Presentation (Incremental Sections)

- [ ] **T-08: Present design in logical sections**
  - Origin: `skills/brainstorming/SKILL.md:42-46` (Present design sections in process flow)
  - Done criterion: Design split into 3–5 sections (e.g., architecture, components, data flow, error handling, testing); each section presented and user approval obtained
  - Acceptance criteria:
    - Sections scaled to complexity (short for simple, 200–300 words for nuanced topics)
    - Approval requested after each section
    - Revisions made if user requests changes
  - Confidence: 🟢

- [ ] **T-09: Obtain incremental user approvals**
  - Origin: `skills/brainstorming/SKILL.md:90` (Design sections with approval)
  - Done criterion: User explicitly approves each design section before moving to next
  - Confidence: 🟢

---

### Phase 6: Design Specification Documentation

- [ ] **T-10: Write design spec to file**
  - Origin: `skills/brainstorming/SKILL.md:111` (Write design doc step)
  - Done criterion: File created at `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`; content matches approved sections from Phase 5
  - Acceptance criteria:
    - File path follows convention (YYYY-MM-DD-<topic>-design.md)
    - All approved sections from Phase 5 included
    - Markdown formatting correct and readable
  - Confidence: 🟢

- [ ] **T-11: Use elements-of-style skill (if available)**
  - Origin: `skills/brainstorming/SKILL.md:113`
  - Done criterion: If skill available, applied to spec for clarity and conciseness
  - Confidence: 🟡

### Phase 7: Spec Self-Review

- [ ] **T-12: Scan for placeholders and incomplete sections**
  - Origin: `skills/brainstorming/SKILL.md:119`
  - Done criterion: Search spec for "TBD", "TODO", incomplete sections; none remain
  - Confidence: 🟢

- [ ] **T-13: Check for contradictions between sections**
  - Origin: `skills/brainstorming/SKILL.md:120`
  - Done criterion: Each section consistent with others; no conflicting requirements or design choices
  - Confidence: 🟡

- [ ] **T-14: Resolve ambiguity in requirements**
  - Origin: `skills/brainstorming/SKILL.md:121`
  - Done criterion: Any requirement interpretable in 2+ ways is clarified; one interpretation chosen and made explicit
  - Confidence: 🟡

- [ ] **T-15: Fix issues inline**
  - Origin: `skills/brainstorming/SKILL.md:124`
  - Done criterion: All issues from T-12–T-14 fixed directly in spec file; no separate review loop created
  - Confidence: 🟢

### Phase 8: User Approval of Written Spec

- [ ] **T-16: Request user review of spec file**
  - Origin: `skills/brainstorming/SKILL.md:128-130`
  - Done criterion: User given spec file path and asked to review; explicit approval obtained
  - Acceptance criteria:
    - User reviews complete spec
    - User confirms no further changes
    - If changes requested, cycle back to T-10 (write) and T-12–T-15 (self-review)

### Phase 9: Git Commit & Transition

- [ ] **T-17: Commit design spec to git**
  - Origin: `skills/brainstorming/SKILL.md:111-114`
  - Done criterion: Design spec file committed to git with descriptive commit message
  - Acceptance criteria:
    - Commit message references design topic and date
    - Commit message includes "design approval" or similar language
    - File appears in git history
  - Confidence: 🟢

- [ ] **T-18: Invoke writing-plans skill**
  - Origin: `skills/brainstorming/SKILL.md:135-136`
  - Done criterion: writing-plans skill invoked; ONLY writing-plans (no other skill)
  - Acceptance criteria:
    - writing-plans is the next skill after brainstorming
    - No implementation skill (frontend-design, mcp-builder, etc.) invoked here

  - Confidence: 🟢

## Suggested Execution Order

1. **T-01–T-03** (Context & Scope): Must complete first; foundation for all subsequent phases
2. **T-04–T-05** (Visual Setup): If applicable; independent of other phases but should be done early
3. **T-06–T-07** (Clarification & Approaches): Sequential; user input required
4. **T-08–T-09** (Design Presentation): Sequential and interdependent; core design validation
5. **T-10–T-15** (Spec Writing & Self-Review): Sequential; produces durable artifact
6. **T-16** (User Approval of Written Spec): Must succeed before moving to git/transition
7. **T-17–T-18** (Commit & Transition): Terminal steps; invoke writing-plans

**Critical Blockers:**
- T-03 blocks T-06 if over-scoped (decompose first)
- T-09 blocks T-10 (spec not written until all sections approved)
- T-16 blocks T-17 (git commit only after user approval)
