# Refactor Tasks run the full flow, not a lite path

A `taskType: refactor` Task flows through the same phases as a feature: map-codebase (brownfield) → full PRD → to-issues → slices + TDD → mandatory e2e → review → human-QA. We rejected an earlier "lite PRD" shortcut for refactors. The reasoning that justified lite (a refactor's scope is pre-pinned by its candidate) is outweighed by two facts: e2e is the entire safety net that makes a refactor safe — "the behavior still works" is what lets old code be transformed without fear, so skipping or lightening it is backwards; and a single uniform pipeline is simpler than a path-with-a-threshold where someone must judge "trivial vs substantial".

To remove the only real friction (the full PRD's user-story sections feel fake for a behavior-preserving change), the PRD template allows refactor-shaped stories: behavior-preservation plus a structural goal, instead of forcing `As a user…`. Old code is transformed — modified OR removed depending on the refactor — and that transformation is captured as explicit acceptance criteria and its own migration-step slices (introduce new → migrate callers → modify/remove old). Hard gate 1 still applies: refactors are high blast radius and the human approves scope before slicing.
