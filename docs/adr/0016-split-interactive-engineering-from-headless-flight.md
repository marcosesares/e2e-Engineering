# Split the orchestrator into interactive /e2e-engineering and headless /e2e-flight

> **PARTIALLY SUPERSEDED by [ADR 0022](0022-flight-one-task-per-spawn-no-loop-no-checkpoint.md) (2026-05-31).** The split itself stands: `/e2e-engineering` = interactive front door, `/e2e-flight` = headless implementation. But `/e2e-flight` is NOT driver-run ‚Äî there is no external driver, no `claude --print` spawn, no `E2E_DRIVER` guard. Flight runs in the current session, one Task per invocation. Read driver/guard references below as history.

ADR 0003 chose skill composition over a monolith ‚Äî many sub-skills under one orchestrator. The orchestrator (`/e2e-engineering`) still spanned the whole flow: human grilling AND the automatable impl/post phases. The external driver (ADR 0015) exposed why that single span is wrong. The driver spawns `claude --print` ‚Äî headless, no stdin, no human. But pre-implementation is irreducibly human: grill-me and grill-with-docs loop *until the user approves*. A headless session cannot answer grill questions. The human/headless boundary is a real seam, and it falls exactly at hard gate 1 (PRD approved).

We split the orchestrator along that seam into two entry points. **`/e2e-engineering`** stays interactive and human-driven: it owns pre-implementation per feature (grill ‚Üí ‚Ä¶ ‚Üí to-prd ‚Üí gate 1), appends approved features to the [[Task queue]], shows the [[Run selection]] checkbox, and runs the batched [[QA sign-off session]]. **`/e2e-flight`** is headless and driver-run: it drains selected Tasks (impl ‚Üí gate 4 ‚Üí automated gate 5 ‚Üí review), parking human-QA. The driver only ever invokes `/e2e-flight`; humans only ever type `/e2e-engineering` (though they may invoke `/e2e-flight` directly ‚Äî see below).

Bootstrap is unified at one point via the **[[E2E_DRIVER guard]]** env var. `/e2e-flight` Step 0: if `E2E_DRIVER` is unset (a bare invocation ‚Äî by a human, or by `/e2e-engineering` at gate-1 approve), spawn the [[AFK wrapper]] and exit; if set (the wrapper re-entered it), do real work for one Task-step. This means `/e2e-engineering` needs zero awareness of the driver ‚Äî it just calls `/e2e-flight`, which self-bootstraps. The env var doubles as the nesting guard: worker sessions never re-spawn a driver.

This refines ADR 0003 (still composition, now two orchestrators not one) and physically realizes ADR 0014's gate-1 unconditional reset ‚Äî the phase boundary is now a boundary between two distinct skills/processes, not a soft in-session instruction. The cost is a second skill to maintain and a slightly less obvious entry model (two doors); accepted because faking headless grilling is worse, and the env-guard keeps the two-door model to a single branch point.
