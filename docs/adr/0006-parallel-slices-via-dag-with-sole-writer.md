# Parallel slice execution via depends_on DAG with orchestrator as sole writer

The implementation loop runs independent vertical slices in parallel rather than strictly one-per-iteration. to-issues emits a `depends_on` DAG; the tracer→schema→logic→api→ui ordering is encoded as dependency edges, so each feature stays sequential along its chain while independent branches fan out. Each iteration the orchestrator computes the ready set (deps satisfied, `passes: false`), dispatches each ready story to its own git worktree + subagent (red-green-refactor), and at fan-in serially merges the branch, writes `passes: true`, and appends to progress.txt.

This supersedes the earlier "one slice per loop iteration" rule. The single-writer invariant is preserved by making the orchestrator the only writer of prd.json + progress.txt — subagents return summaries and never touch shared state, which also keeps progress.txt genuinely append-only under concurrency. We rejected per-worktree state segments (concurrent appends break append-only and force a real merge/conflict resolver). Cost: to-issues must produce a correct dependency graph (Ralph flagged the absence of inter-story dependencies as a gap), and the orchestrator must handle branch merges at fan-in.
