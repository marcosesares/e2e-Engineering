# Task queue with a two-tier dependency model

The system was single-Task throughout: "new task = reset prd.json + progress.txt", one feature at a time, the human re-invoking for the next. The flight design (ADR 0015/0016) lets a human document several features, select a batch, and walk away while they all build. That needs a layer the system never had — a cross-Task backlog with ordering and dependencies — without colliding with the within-Task machinery that already exists.

We add a **[[Task queue]]** (`.e2e-engineering/queue.json`): an ordered list of [[Task]]s, each `{ id, priority, dependsOn, status, selected }`, with each Task's body under `.e2e-engineering/tasks/<id>/` (the existing prd.json + progress.txt + handoff.md, unchanged). `/e2e-flight` drains it serially — at most one Task `in-progress` at a time, picked by `priority` + satisfied `dependsOn`, so a fresh flight session can resume unambiguously by reading the queue first.

The trap is dependency scope. `depends_on` already exists at story level (snake_case, the tracer→schema→logic→ui DAG inside one PRD). The new cross-Task edge is a *second* DAG at a higher altitude. We name it **`dependsOn`** (camelCase) — the casing is the scope signal, reusing the snake/camel convention already in the schemas. Selecting a Task auto-includes its unmet `dependsOn` Tasks so a batch is never internally inconsistent.

Writers are split but safe: `/e2e-engineering` creates entries (at gate 1), `/e2e-flight` flips `status` — disjoint fields, and never concurrent because the interactive session fully exits before the driver starts. This bends the sole-writer invariant (ADR 0006) but doesn't break it: that invariant governs prd.json + progress.txt within a Task, which stay single-writer; queue.json is a new artifact with a documented disjoint-field, sequential-time contract. The cost is a new schema and a new directory convention to keep in sync; accepted because multi-feature walk-away is the whole point of flight, and a flat single-PRD model can't express "build these three, in this order, respecting that B needs A".
