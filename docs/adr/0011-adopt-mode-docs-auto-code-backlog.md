# Adopt mode bootstraps standards docs (human-reviewed) but never auto-refactors code

ship-it has an `adopt` mode for onboarding into an in-progress project, separate from the per-task flow. It splits onboarding into two halves of very different blast radius. The DOCS half auto-DRAFTS the standards scaffolding — CONTEXT.md glossary, constitution, and ADRs — from existing code and documentation, and presents them for human review/edit rather than silently committing. The CODE half runs map-codebase repo-wide to produce a prioritized refactor backlog routed through triage; code is never automatically rewritten to standards. The human picks which candidates become refactor Tasks, and each conforms incrementally through the normal gated flow (full PRD → hard gate 1 → slices → mandatory e2e). Refactor Tasks run the FULL flow like any feature — see ADR 0012.

Rejected: auto-committing the standards docs without review (an incorrect glossary or constitution is then injected into every future subagent, propagating wrong domain language everywhere) and auto-refactoring the whole codebase to standards (maximal blast radius, and explicitly forbidden by README Part 2 and the constitution's scope-discipline principle). Net: documentation conforms immediately under human review; code conforms over time, one human-chosen refactor at a time.
