# ARCHITECTURE.md — a durable project-architecture reference, read by the implementation loop

A 2026-05-28 session reverted a whole feature: parallel slice subagents, run against a stale snapshot of master, each independently guessed the project's architecture and produced a duplicate resource class at a URL an existing class already owned, two divergent copies of one component, a duplicate API-client block, and naming that broke the project convention. The merge stitched the contradictions together. Nothing in the flow held the project's architecture in a place the slices could read, so each slice reinvented it — differently.

The existing doc-types couldn't hold it. CONTEXT.md is a glossary and nothing else (ADR 0006). The constitution is *generic* engineering standards injected into every subagent — "test first", "real-interface tests" — cross-project rules, not "this project's completion endpoints live in EnrollmentResource". ADRs are scattered point-in-time decisions, not a consolidated current map an agent reads in one shot. codebase-map.md is per-task and rots. The architecture had no durable home.

We add **ARCHITECTURE.md** at repo root: project-specific structure + conventions (layering, ownership rules, naming, integration patterns, anti-patterns) — five sections, each anchored to a failure mode from the incident. It is the synthesized *current* architecture; ADRs remain the changelog behind it. The split from the constitution is the load-bearing distinction: generic best practice → constitution; project-specific rule → ARCHITECTURE.md.

We considered three lighter alternatives and rejected each. Bloating codebase-map §4 with conventions (the earlier "Proposal 3") — rejected because codebase-map is per-task and rots, so the conventions would be re-derived every task and could drift between them; a durable doc is the point. Folding architecture into the constitution — rejected because it muddies a clean generic/specific boundary and would inject one project's structure into a cross-project standards file. A new `tech-plan` sub-skill between PRD and to-issues — rejected because mattpocock's ADR-0014 already deleted exactly that (`prd-to-plan`) as a redundant middle step, and because it adds a per-task planning ceremony that fights this skill's lightness. A durable reference doc that existing steps merely *read* costs no new loop step and no new gate.

Two consequences carry the cost. First, ARCHITECTURE.md shapes every future slice subagent the way the constitution does, so a wrong entry propagates everywhere — which is why its writes are confined to human-present phases (seeded/reviewed in pre-impl, amended at the post-impl human-QA gate) and the automated implementation loop treats it as read-only. Drift a subagent notices is proposed in its summary and staged as a pending amendment, riding the same batched-approval chain as constitution amendments; the loop is never interrupted to edit it. Second, it is one more durable doc to keep honest — mitigated by lazy creation (greenfield projects have no file until a real structural decision exists) and by map-codebase flagging drift on brownfield tasks.

This also resolved where the brownfield ownership decision lives. to-issues now pins each story's integration decision (which existing owner/seam it extends) into a dedicated `integration` field in prd.json, reading ARCHITECTURE.md sections 1-2 — so the decision is made once, by the orchestrator, against the durable map, instead of guessed N times by N parallel subagents.
