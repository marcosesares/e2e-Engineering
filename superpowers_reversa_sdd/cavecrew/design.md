## Interface

**Input:** Task description + scope  
**Output:** Subagent type recommendation + spawn parameters (or vanilla agent recommendation)

**Decision Input:**

| Field | Type | Example |
|-------|------|---------|
| Task Type | enum: "locate", "edit", "review" | "Where is X defined?" |
| Scope | range: "1-2 files" to "5+ files" | "2 files" |
| Output Preference | enum: "compressed", "prose" | "compressed" |
| File Paths Known? | bool | true |

**Decision Output:**

| Field | Value | Example |
|-------|-------|---------|
| Selected Agent | enum: cavecrew-{investigator\|builder\|reviewer} \| {Explore\|Code Reviewer\|main} | cavecrew-investigator |
| Rationale | string | "Discovery task; compressed output saves context" |
| Spawn Parameters | object | `{ query: "where is X defined", search_type: "grep" }` |

## Decision Flow

```
1. Classify Task
   ↓
2. Known file paths?
   YES → 3a. Scope ≤2 files?
   NO  → 3b. Need discovery

3a. Scope ≤2 files + paths known
   → builder (surgical edit)

3b. Discovery needed
   → investigator (locate code)

4. Prefer compressed output?
   YES → use cavecrew variant
   NO  → use vanilla (Explore, Code Reviewer, main thread)

5. Spawn Subagent
   ↓
6. Parse Output Contract
   - Investigator: file:line table
   - Builder: change summary or terminal response
   - Reviewer: line-by-line findings + totals
   ↓
7. Handle Terminal Responses (builder only)
   - "too-big" → escalate to main thread
   - "needs-confirm" → request clarification
   - "ambiguous" → main thread decides
   - "regressed" → halt and investigate
   ↓
8. Chain Next Subagent (if applicable)
   - Locate → fix → verify pattern
   - Or stand-alone operation
```

## Main Patterns

### Pattern 1: Locate → Fix → Verify

**Scenario:** "Update all error handlers to include tracing"

**Flow:**
1. Spawn `cavecrew-investigator` with query "where are error handlers defined"
2. Receive file:line table (5–10 matches)
3. Main thread reviews matches, picks 1–2 most critical
4. Spawn `cavecrew-builder` with exact paths and change description
5. Receive change summary + verification status
6. Spawn `cavecrew-reviewer` on diff to audit for regressions
7. Receive findings with severity levels

**Evidence:** `SKILL.md` lines 62–65

---

### Pattern 2: Parallel Scout

**Scenario:** "Find all uses of the config system"

**Flow:**
1. Spawn 3 `cavecrew-investigator` calls in parallel:
   - Angle 1: "where is config defined"
   - Angle 2: "what calls config"
   - Angle 3: "list config tests"
2. Receive 3 results (file:line tables)
3. Main thread aggregates and deduplicates
4. Proceed to fix/verify if needed

**Evidence:** `SKILL.md` lines 67–68

---

### Pattern 3: Single-Shot Edit

**Scenario:** "Fix typo in error message at `src/errors.ts:42`"

**Flow:**
1. Skip investigator (path already known)
2. Spawn `cavecrew-builder` directly with exact path:line
3. Receive change confirmation
4. Done (no verification needed for trivial fixes)

**Evidence:** `SKILL.md` lines 70–71

## Subagent Output Contracts

### `cavecrew-investigator`

**Format:**
```
[Header describing search]:
- path:line — `symbol` — short note
- path:line — `symbol` — short note
...
totals: N results.
```

Or:
```
No match.
```

**Guarantees:**
- File path first, always
- Line number attached via `:line`
- Symbol backticked
- Each line fits in ~80 chars
- Total count included
- Regex-safe: `path:\d+` matches all site references

**Example:**
```
Where is `TddCycle` defined:
- skills/test-driven-development/SKILL.md:18 — `TddCycle` — State machine defining RED-GREEN-REFACTOR phases
- _reversa_sdd/state-machines.md:42 — `TddCycle` — Documented in specification
totals: 2 results.
```

**Evidence:** `SKILL.md` lines 38–44

---

### `cavecrew-builder`

**Format (Success):**
```
<path:line-range> — <change ≤10 words>.
verified: <re-read OK | mismatch @ path:line>.
```

Or **Terminal Responses:**
```
too-big.
needs-confirm.
ambiguous.
regressed.
```

**Guarantees:**
- Change summary ≤10 words
- Verification status explicit
- Terminal response is first token (recognized via `startswith()` pattern)
- If terminal, no change applied

**Examples:**
```
src/handlers.ts:42-48 — Added context parameter to error handler.
verified: re-read OK.
```

Or:
```
too-big.
```

**Evidence:** `SKILL.md` lines 46–51

**Handling Terminal Responses:**
- `too-big.` → Task spans ≥5 files or cross-cutting refactor; escalate to main thread or feature-dev skill
- `needs-confirm.` → Ambiguity in builder scope; request clarification from user
- `ambiguous.` → Multiple valid interpretations; main thread picks direction
- `regressed.` → Change introduces inconsistency; investigate before applying

---

### `cavecrew-reviewer`

**Format:**
```
path:line: <emoji> <severity>: <problem>. <fix>.
path:line: <emoji> <severity>: <problem>. <fix>.
...
totals: N🔴 N🟡 N🔵 N❓
```

Or:
```
No issues.
```

**Guarantees:**
- Findings sorted by file → line ascending
- Emoji prefix: 🔴 (critical), 🟡 (warning), 🔵 (info), ❓ (unclear)
- Severity explicit (e.g., "Critical", "Warning")
- Problem statement + fix suggestion always paired
- Totals included

**Example:**
```
src/auth.ts:18: 🔴 Critical: Missing null check on user token. Add guard before dereference.
src/auth.ts:42: 🟡 Warning: Error handler doesn't log context. Add context object to logger.
src/db.ts:15: 🔵 Info: Slow query detected. Consider indexing on user_id.
totals: 1🔴 1🟡 1🔵 0❓
```

**Evidence:** `SKILL.md` lines 53–58

## Decision Rules (Decision Matrix)

| Task | Scope | Files Known? | Prefer Compressed? | Recommendation |
|------|-------|-------|---------|---|
| "Where is X defined?" | — | No | Yes | `cavecrew-investigator` |
| "Where is X defined?" | — | No | No | `Explore` (vanilla) |
| "Fix these 2 files" | ≤2 | Yes | Yes | `cavecrew-builder` |
| "Implement new feature" | 3+ | Any | Any | Main thread or feature-dev skill |
| "Add error handling" | 5+ | No | Yes | `cavecrew-investigator` + main thread review |
| "Review this PR" | — | Yes | Yes | `cavecrew-reviewer` |
| "Code quality feedback" | — | Yes | No | `Code Reviewer` (vanilla, with prose) |
| "One-line answer" | — | — | — | Main thread (no subagent) |

**Heuristic:** "Use cavecrew if output should be ~1/3 the size of vanilla; use vanilla if prose/explanation is needed." (`SKILL.md` line 28)

## Dependencies

| Dependency | Role | How Used |
|-----------|------|----------|
| `cavecrew-investigator` subagent | Code location (grep/defs/callers) | Spawned for discovery tasks; output parsed for file:line sites |
| `cavecrew-builder` subagent | Surgical file editing | Spawned for ≤2 file edits with known paths; output parsed for change confirmation or terminal response |
| `cavecrew-reviewer` subagent | Diff/branch auditing | Spawned for code review; output parsed for findings + severity + totals |
| caveman compression engine | Output size reduction | Inherited via subagent definitions; no explicit dependency |
| Vanilla agents (Explore, Code Reviewer) | Fallback for prose output | Used when compressed output not suitable |

## Design Decisions Identified

| Decision | Evidence | Confidence |
|----------|----------|-----------|
| **Three-subagent model:** investigator, builder, reviewer | `SKILL.md` "Output contracts" section; explicit role definitions | 🟢 |
| **Structured output format:** file:line tables, not prose | `SKILL.md` output contracts; "rule of thumb" (compressed output) | 🟢 |
| **Builder precondition:** file paths must be known before spawn | `SKILL.md` line 75: "Don't use builder when you don't already know the file" | 🟢 |
| **Size threshold for compressed output:** ~1/3 token reduction via caveman | `SKILL.md` line 28 heuristic; investigator 700 vs. Explore 2k | 🟢 |
| **Terminal responses halt operation:** builder can return `too-big`, `needs-confirm`, etc. | `SKILL.md` lines 46–51 (builder output contract) | 🟢 |
| **Parallel investigators allowed:** different search angles in one message | `SKILL.md` lines 67–68 ("Parallel scout") | 🟢 |
| **Chaining pattern (locate → fix → verify):** most common workflow | `SKILL.md` lines 62–71 ("Chaining patterns") | 🟢 |
| **No prose from cavecrew:** output is structured, sometimes terse | `SKILL.md` line 78 ("Don't expect prose") | 🟢 |

## Internal State

**Cavecrew is stateless.** It does not maintain persistent state. All state is:
1. **Decision context** — Task description, scope, file paths (session-local)
2. **Subagent tool-results** — Structured output injected into main-thread context
3. **Cached grep/ast results** (within subagents) — Not exposed to main thread

---

## Observability

**Decision Logging:**
- Main thread logs which subagent selected and why
- Decision rationale captured in chat context

**Output Parsing:**
- Subagent results checked for expected format (file:line table, change summary, findings table)
- Terminal responses detected via `startswith("too-big")` etc.
- Malformed output would cause parsing error; escalate to user

**Metrics (Token Efficiency):**
- Vanilla agent result size: ~2k tokens
- Cavecrew result size: ~700 tokens
- Savings per delegation: ~1.3k tokens
- Cumulative savings (20 delegations): ~26k tokens

## Risks and Gaps

| Risk | Impact | Mitigation | Confidence |
|------|--------|-----------|-----------|
| Subagent output parsing fails | Main thread cannot use results | Format guaranteed by output contract; strict regex validation | 🟢 |
| Builder spawned for unknown file path | Wasted effort; context bloat | Precondition enforced in main thread (investigator first) | 🟢 |
| Over-scoped task assigned to builder | Builder returns `too-big`; wasted turn | Decision matrix and heuristics guide correct selection | 🟢 |
| Terminal response not recognized | Main thread assumes change applied; incorrect state | Terminal response is first token; explicit check required | 🟢 |
| Parallel investigators create merge conflict | Results overlap or contradict | Main thread responsible for deduplication and aggregation | 🟡 |
| Caveman compression loses important detail | Output too cryptic for downstream use | Compression is ~60%, not 95%; content preserved. Paraphrase if human will read | 🟡 |
