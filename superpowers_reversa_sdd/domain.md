## Executive Summary

Superpowers is a **zero-dependency agent behavior-shaping framework** that operates via skills — structured markdown documents that guide agent decision-making during software development workflows. The system enforces discipline over shortcuts through documented process patterns (TDD, systematic debugging, planning gates) rather than code.

**Core Domain Principle:** Skills are executable process specifications, not prose. Modifying skill content changes agent behavior at runtime, therefore requires evidence of improvement before acceptance.

## Glossary

### Skill
**Definition:** A self-describing markdown document that guides agent behavior during a specific workflow phase (planning, development, review, communication).

**Components:**
- **Metadata** — Name, triggers (when to use), category, version
- **Process** — Ordered steps, gates, decision trees, red flags
- **Examples** — Concrete code scenarios, anti-patterns, rationalizations

**Scope:** Skills are **general-purpose** (benefit all projects) or **zero-dependency** (no external tool requirements) or **harness-agnostic** (work across Claude Code, Codex, Gemini CLI, OpenCode).

**Domain Constraint:** Domain-specific skills (portfolio builders, game dev, specific tools) do NOT belong in core; they belong in standalone plugins.

🟢 **Confidence:** CONFIRMED (architectural pattern documented in multiple skills)

---

### Harness
**Definition:** The execution environment or IDE that loads and invokes skills (Claude Code CLI, Codex App/CLI, Gemini CLI, OpenCode, Cursor IDE, etc.).

**Harness Responsibilities:**
- Load the `using-superpowers` bootstrap at session start
- Auto-trigger skill invocation based on declared triggers
- Provide platform-specific tools (worktree management, git, subagent dispatch, hooks)

**Critical Contract:** A real harness integration auto-triggers `brainstorming` skill on "Let's make a react todo list" message. Manual tool copying or at-runtime shims are **not real integrations**.

🟡 **Confidence:** INFERIDO (from PR requirements and skill descriptions)

---

### Agent (in superpowers context)
**Definition:** An AI assistant (Claude model variant, different LLM family) operating within a harness session to complete development tasks.

**Agent Types in Superpowers:**
- **Implementer** — Writes code per plan (execute-plans, sdd-workflow)
- **Reviewer** — Code/spec quality gate (requesting-code-review, receiving-code-review)
- **Subagent** — Parallel task executor (spawned by superpowers:dispatching-parallel-agents, superpowers:subagent-driven-development)
- **Skill-Reader** — Uses this framework to look up process guidance (all agents reading any skill)

**Role Constraints:** An agent cannot skip skill process gates (TDD red phase, systematic debugging phases) without explicit human consent.

🟢 **Confidence:** CONFIRMED (skill enforcement + role separation documented)

---

### Skill Trigger
**Definition:** The condition under which a skill should be invoked by the harness or human.

**Examples:**
- Message contains "let's make" → trigger `brainstorming`
- Skill explicitly invoked via `/skillname` → priority load override
- Session starts → load `using-superpowers` bootstrap
- Agent says "I need a code review" → trigger `requesting-code-review`
- Build fails → trigger `systematic-debugging`

**Rule:** Skill triggers are **intent-based**, not keyword-sniffing. The system looks for semantic intention, not just substring match.

🟡 **Confidence:** INFERIDO (triggers documented, but sniffing heuristics not fully specified)

---

### Gate / Checkpoint
**Definition:** A mandatory decision point where an agent or human must verify a condition before proceeding.

**Hard Gates (non-negotiable without human consent):**
- TDD Red Phase: must write failing test before any production code
- Brainstorming: must write and approve design doc before starting implementation plan
- Debugging Phase 4: if fix #3+ fails, escalate to human (architectural issue signal)

**Soft Gates (can be overridden with justification):**
- Test coverage thresholds
- Code style checks
- Documentation completeness

**Domain Rule:** "No exceptions without your human partner's permission" applies only to hard gates.

🟢 **Confidence:** CONFIRMED (red flags tables + gate descriptions explicit)

---

### Plan
**Definition:** A structured implementation strategy with bite-sized, verifiable tasks.

**Required Sections:**
- Goal (one sentence)
- Architecture (2-3 sentences approach)
- Tech stack
- Tasks (checkbox list, depends_on relationships, estimated duration)

**Domain Rule:** Every task must have <= 2 hours estimated work. If task > 2h, break it down.

**Execution Model:** Plans are executed either via `subagent-driven-development` (each task → subagent + review loop) OR `executing-plans` (sequential agent execution within single session).

🟢 **Confidence:** CONFIRMED (plan structure detailed in writing-plans, executing-plans skills)

---

### Spec / Design Document
**Definition:** Pre-implementation documentation describing what will be built, why, and how.

**Required Sections:**
- Title & Goal (one sentence)
- Architecture (2-3 sentences)
- Tech Stack
- Key Components / Data Flow
- Error Handling Strategy
- Testing Approach

**Domain Rule:** Design doc MUST be approved by human before implementation starts (brainstorming skill hard gate).

**Path Convention:** `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

🟢 **Confidence:** CONFIRMED (brainstorming skill Step 8 gate)

---

### Worktree
**Definition:** An isolated git workspace linked to the main repository, created for feature work to avoid branch pollution.

**Creation Priority:**
1. **Native Tool** (Claude Code EnterWorktree, Codex App native management) — preferred
2. **Git Worktree Fallback** (`git worktree add ...`) — when no native tool
3. **Already Exists** — detect `GIT_DIR != GIT_COMMON`, skip creation

**Domain Rule:** Environment detection is mandatory before creating. Failing to detect existing isolation causes nested worktrees.

**Cleanup:** Linked worktrees owned by harness (detached HEAD on Codex App) are NOT removed by the skill — harness owns cleanup.

🟢 **Confidence:** CONFIRMED (environment detection, cleanup logic documented in codex-app compatibility spec)

## Business Rules

### 1. Zero-Dependency Architecture

**Rule:** Superpowers core adds zero external tool dependencies.

**Rationale:** Skills must work in any harness without forcing users to install npm packages, language-specific runtimes, or third-party services.

**Exceptions:** Support for new harnesses (IDE plugins, new CLIs) is acceptable if it extends platform-native capabilities, not adds external deps.

**Enforcement:** PR requirement — "no new dependencies except for new harness support."

🟢 **Confidence:** CONFIRMED (CLAUDE.md explicit rejection of third-party deps)

---

### 2. Skill Content Is Code

**Rule:** Skills are executable process specifications. Modifying skill text changes agent behavior at runtime.

**Consequence:** Skill changes require:
- Before/after eval evidence
- Adversarial testing across multiple sessions
- No casual rewording of red flags tables or rationalization lists
- High bar for behavioral modifications

**Non-Negotiable Content:** "human partner" terminology is intentional; careful tuning of Red Flags tables (feedback shows these prevent agent rationalization); process gates (TDD RED phase, debugging phases).

**Consequence of Violation:** Agents rationalize away process gates, causing regression to fast-but-careless behavior.

🟡 **Confidence:** INFERIDO (PR requirements state this; not codified in skill metadata)

---

### 3. Quality Gate: 94% PR Rejection Rate

**Rule:** Superpowers maintains deliberate high rejection rate as signal of quality.

**Factors Contributing:**
- Zero-dependency constraint eliminates many requests upfront
- Anti-slop requirements (no bulk PRs, no speculative fixes, no fabricated claims)
- Skill content bar (eval evidence required for behavior changes)
- Contributor understanding requirement (must understand WHY before contributing)

**Human Impact:** PRs are closed without review if they violate structural rules (missing sections, bulk requests, third-party deps). This saves maintainers' time and signals system maturity.

🟢 **Confidence:** CONFIRMED (CLAUDE.md § "If You Are an AI Agent")

---

### 4. Process Over Tooling

**Rule:** Superpowers optimizes for disciplined process (TDD, verification gates, systematic debugging) rather than build-tool automation.

**Example Contrast:**
- ❌ "Add linting" (tooling, not process)
- ✅ "Systematic Debugging" (4-phase root-cause process, verifiable without tools)

**Consequence:** Skills contain decision trees, red flags tables, and phase definitions rather than shell scripts or CI/CD configs.

🟢 **Confidence:** CONFIRMED (all 21 skills are markdown, zero-tooling, process-based)

---

### 5. Agent Behavior Shaping via Documentation

**Rule:** Agents follow documented processes because skills are loaded in context, not via code instrumentation.

**Mechanism:** During planning/coding, agent reads relevant skill (e.g., `test-driven-development/SKILL.md`), sees RED-GREEN-REFACTOR phases with red flags (e.g., "code before test? delete & restart"), and adjusts behavior to avoid violating those signals.

**Signal Strength:** Red Flags tables are the highest-fidelity behavior shaping tool (tested extensively across evals, reason not to modify casually).

🟡 **Confidence:** INFERIDO (mechanism inferred from skill structure; eval evidence not read by detective)

---

### 6. Session Transcript Requirement for New Harness

**Rule:** Any PR adding support for new harness (IDE, CLI tool) MUST include a **real session transcript** showing skills auto-trigger correctly.

**Acceptance Test:** Open clean session in new harness, send exactly "Let's make a react todo list", capture transcript showing `brainstorming` skill auto-triggered (not manual copy, not at-runtime wrapper).

**Failure Modes (rejected immediately):**
- Manual skill file copying into harness directory
- Runtime shim (`npx skills ...` wrapper)
- User must opt-in to skills per-session
- Brainstorming does not auto-trigger

**Rationale:** Bootstrap autoload is the contract between superpowers and harness. Without it, skills are dead weight.

🟢 **Confidence:** CONFIRMED (CLAUDE.md § "New Harness Support")

---

### 7. Cross-Platform Tool Abstraction

**Rule:** When describing tools in skills, use platform-agnostic action names (e.g., "enter worktree") rather than tool-specific commands (e.g., "EnterWorktree vs. git worktree add").

**Tool Mapping (Codex Tools):** Each platform maps generic actions to native tools:
- `enter-worktree` → Claude Code: `EnterWorktree` tool | Codex: native worktree mgmt | CLI: `git worktree add`
- `run-code` → Claude Code: `Bash` tool | Codex: native code runner | etc.

**Phase A-E Project (git history):** Multi-phase refactor to eliminate platform-specific prose from skills and centralize tool refs in platform docs.

🟡 **Confidence:** INFERIDO (git log shows Phase A-E commits; not fully read by detective)

---

### 8. Eval-Driven Skill Tuning

**Rule:** Superpowers maintains `evals/` directory as canonical behavior test harness. Skills are tuned based on drill results across multiple scenarios.

**Lift History (git log):** Bash tests were previously scattered; consolidated into `evals/` (drill framework) in 2026-05, eliminating per-test setup burden.

**Backend Support:** Evals run against Claude Code, Codex, Gemini 2.5 Flash, OpenCode backends.

🟡 **Confidence:** INFERIDO (evals/ structure visible; drill methodology inferred from commits + code-analysis)

---

### 9. Harness-Specific Integration Contracts

**Rule:** Each harness must provide specific tools/hooks; skills adapt via conditional logic, not fragmentation.

**Example: Codex App Worktree Contract**
- Codex App creates managed worktrees (detached HEAD)
- Blocks `git checkout -b`, `git push` via Seatbelt sandbox
- Provides "Create branch" native UI
- Skill detects `GIT_DIR != GIT_COMMON && BRANCH empty`, skips finishing menu, emits handoff payload instead

**Non-Negotiable:** Skills must NOT branch on "if Codex App, do X; if Claude Code, do Y". Instead: detect environment, adapt behavior, document in tool mapping.

🟡 **Confidence:** INFERIDO (Codex app compatibility spec details; not yet automated in skill logic)

---

### 10. PR Review Philosophy: Human Evidence Required

**Rule:** Before opening PR, agent MUST show human partner:
1. Complete diff (not summary)
2. Justification (why this solves the problem)
3. Search results (does identical PR exist? closed PR with different approach?)
4. Verification plan (how will this be tested?)

**Consequence:** Agent's job is to "protect your human partner from embarrassment," not to maximize PR throughput.

**Non-Negotiable:** "You will thank me for saving you the embarrassment" is the frame, not "I'll open this PR for you automatically."

🟡 **Confidence:** INFERIDO (CLAUDE.md § "If You Are an AI Agent" frames this; not explicitly in skills)

```
  │   └─ writing-plans (at end of brainstorm)
  │       └─ [execution choice]

  │           │   ├─ using-git-worktrees (Step 0: ensure isolation)
  │           │   └─ finishing-a-development-branch (Step 5: cleanup)
  │           │
  │           └─ executing-plans (sequential, same session)
  │               ├─ using-git-worktrees (before tasks)
  │               └─ finishing-a-development-branch (after tasks)
  │
  ├─ test-driven-development (when: "write code")
  │   └─ [RED phase] (mandatory gate)
  │       └─ [GREEN phase] (after RED passes)
  │           └─ [REFACTOR phase]
  │
  ├─ systematic-debugging (when: "test failed", "error occurred")
  │   └─ [Phase 1-4 sequence] (mandatory phases)
  │
  ├─ requesting-code-review (when: "need review")
  │   └─ [subagent spawn: external reviewer]
  │
  └─ dispatching-parallel-agents (when: independent tasks)
      └─ [spawn N subagents simultaneously]
```

## Anti-Patterns (🔴 Detected in Codebase)

### 1. Code Before Test

**Rule:** Writing production code before RED phase test is a skill violation.

**Red Flag:** "I'll test after" or "tests will follow."

**Detection in Superpowers:** TDD skill explicitly lists "code before test? delete & restart."

**Observed in Codebase:** ❌ None detected (TDD enforcement strong)

---

### 2. Guessing Without Investigation

**Rule:** Debugging is mandatory 4-phase process, not guess-and-retry.

**Red Flag:** "I think the issue is X, let me try this fix."

**Detection in Superpowers:** Systematic Debugging skill red flags "guessing", "multiple changes at once", "skip test."

**Observed in Codebase:** ❌ Codex app compatibility spec shows deliberate investigation (6 empirical findings before design)

---

### 3. Skill Content Casual Rewriting

**Rule:** Modifying red flags, rationalization lists, or phase definitions without eval evidence is not acceptable.

**Red Flag:** "Let's update the TDD guidance to be less strict" (without before/after data).

**Observed in Codebase:** ❌ PR history shows zero unjustified skill rewording; all changes tie to evals or bug fixes

---

### 4. Third-Party Dependencies

**Rule:** Zero-dependency core design.

**Red Flag:** "We should add [npm package] to simplify [feature]."

**Observed in Codebase:** ❌ Zero external deps in core skills; all complex behavior documented in markdown

---

### 5. Domain-Specific Skills in Core

**Rule:** Core contains only general-purpose skills usable across any project type.

**Red Flag:** "Let's add a skill for blockchain project workflows."

**Observed in Codebase:** ❌ All 21 skills tested across multiple domains (Go, JavaScript, Python, systems code)
