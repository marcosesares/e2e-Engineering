## 1. Issue Triage State Machine

🟢 CONFIRMED — defined explicitly in `skills/engineering/triage/SKILL.md`

### State roles (one active per issue at all times)

| State | Meaning | Trigger |
|-------|---------|---------|
| `needs-triage` | Maintainer needs to evaluate | Default on any new unlabeled issue |
| `needs-info` | Waiting on reporter | Maintainer decides issue is under-specified |
| `ready-for-agent` | Fully specified; AFK agent can pick up | Grilling complete; agent brief posted |
| `ready-for-human` | Needs human (judgment call, external access) | Maintainer decides it cannot be delegated |
| `wontfix` | Will not be actioned | Maintainer rejects |

### Category roles (one active per issue at all times)

| Category | Meaning |
|---------|---------|
| `bug` | Something is broken |
| `enhancement` | New feature or improvement |

### Transition diagram

```mermaid
stateDiagram-v2
    [*] --> unlabeled : issue opened

    unlabeled --> needs_triage : initial triage begins

    needs_triage --> needs_info : insufficient info
    needs_triage --> ready_for_agent : fully specified + AFK-safe
    needs_triage --> ready_for_human : requires human judgment
    needs_triage --> wontfix : maintainer rejects

    needs_info --> needs_triage : reporter replies with requested info

    ready_for_agent --> [*] : agent implements + closes
    ready_for_human --> [*] : human implements + closes
    wontfix --> [*] : closed (enhancement → .out-of-scope entry written)

    note right of ready_for_agent
        Side effect: Agent Brief posted as comment
    end note

    note right of needs_info
        Side effect: Triage Notes posted as comment
    end note

    note right of wontfix
        If enhancement: write .out-of-scope/<concept>.md
        If bug: polite explanation only
    end note
```

### Invariants
- A triaged issue carries **exactly 1 state role** + **exactly 1 category role** at all times.
- Maintainer may override to any state at any time.
- Conflicting state roles: flag and ask maintainer before acting.
- All AI-generated comments must open with `> *This was generated by AI during triage.*`

## 2. Caveman Communication Mode State Machine

🟢 CONFIRMED — defined in `skills/productivity/caveman/SKILL.md`

```mermaid
stateDiagram-v2
    [*] --> INACTIVE

    INACTIVE --> ACTIVE : trigger received\n("caveman mode" / "less tokens" / "be brief" / /caveman)
    ACTIVE --> INACTIVE : deactivation received\n("stop caveman" / "normal mode")

    state ACTIVE {
        [*] --> COMPRESSING
        COMPRESSING --> EXCEPTION : security warning\nOR irreversible action\nOR multi-step sequence risk\nOR user confusion / repetition
        EXCEPTION --> COMPRESSING : clear section complete
    }
```

### Invariants
- ACTIVE persists across **all** turns once triggered — never auto-deactivates.
- Exception sub-state is temporary; COMPRESSING resumes immediately after.
- Code blocks and exact error quotes are always reproduced verbatim regardless of state.

## 3. Prototype Branch State Machine

🟢 CONFIRMED — defined in `skills/engineering/prototype/SKILL.md`

```mermaid
stateDiagram-v2
    [*] --> ROUTING

    ROUTING --> LOGIC_BRANCH : question is about state/logic
    ROUTING --> UI_BRANCH : question is about appearance
    ROUTING --> AMBIGUOUS : question type unclear

    AMBIGUOUS --> LOGIC_BRANCH : surrounding code is backend
    AMBIGUOUS --> UI_BRANCH : surrounding code is page/component
    AMBIGUOUS --> AWAITING_USER : user reachable → ask

    AWAITING_USER --> LOGIC_BRANCH : user chooses logic
    AWAITING_USER --> UI_BRANCH : user chooses UI

    state LOGIC_BRANCH {
        [*] --> BUILDING_LOGIC
        BUILDING_LOGIC --> RUNNING : handed to user
        RUNNING --> CAPTURED : answer captured
        CAPTURED --> DELETED : TUI shell deleted
        CAPTURED --> PROMOTED : logic module lifted to production
    }

    state UI_BRANCH {
        [*] --> SUB_SHAPE_A : existing page available (preferred)
        [*] --> SUB_SHAPE_B : no existing page (last resort)
        SUB_SHAPE_A --> VARIANTS_BUILT
        SUB_SHAPE_B --> VARIANTS_BUILT
        VARIANTS_BUILT --> RUNNING
        RUNNING --> WINNER_CHOSEN
        WINNER_CHOSEN --> PROMOTED : winner folded into real page/route
        WINNER_CHOSEN --> LOSERS_DELETED
    }
```

## 4. Writing Session States

🟡 INFERRED — from `writing-beats`, `writing-fragments`, `writing-shape` SKILL.md files

### writing-beats journey state

```mermaid
stateDiagram-v2
    [*] --> AWAITING_PATH : article path not known

    AWAITING_PATH --> AWAITING_START : path confirmed
    [*] --> AWAITING_START : path provided upfront

    AWAITING_START --> PRESENTING_CANDIDATES : draft 2-3 starting beats

    PRESENTING_CANDIDATES --> WRITING_BEAT : user picks a beat

    WRITING_BEAT --> READING_DISK : re-read article from disk
    READING_DISK --> OFFERING_NEXT : offer 2-3 next beats

    OFFERING_NEXT --> WRITING_BEAT : user picks next beat
    OFFERING_NEXT --> ENDED : natural end of journey

    WRITING_BEAT --> EDITING : user requests rewrite / go back
    EDITING --> OFFERING_NEXT

    ENDED --> [*]
```

### writing-fragments accumulation state

```mermaid
stateDiagram-v2
    [*] --> CAPTURING_INITIAL : capture from very first user message

    CAPTURING_INITIAL --> FIRST_WRITE : write H1 title + first fragment
    FIRST_WRITE --> INTERVIEWING

    INTERVIEWING --> FRAGMENT_EMERGED : fragment surfaces
    INTERVIEWING --> EDITING_CMD : user gives cut/rewrite/merge command

    FRAGMENT_EMERGED --> READING_DISK : re-read file
    READING_DISK --> APPENDING
    APPENDING --> INTERVIEWING

    EDITING_CMD --> READING_DISK

    INTERVIEWING --> ENDED : user ends session
    ENDED --> [*]

    note right of ENDED
        Pile not necessarily empty at end —
        that is expected and fine
    end note
```
