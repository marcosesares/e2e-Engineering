### Control flows

#### `diagnose` — 6-phase linear state machine
```
Phase 1: build-feedback-loop (core investment)
  → Failing test | Curl/HTTP script | CLI invocation | Headless browser |
    Replay trace | Throwaway harness | Property/fuzz loop |
    Bisection harness | Differential loop | HITL bash script
Phase 2: reproduce
  → Confirm: correct bug? reproducible? symptom captured?
Phase 3: hypothesise
  → Generate 3-5 ranked falsifiable hypotheses BEFORE testing
  → Show to user (optional, non-blocking)
Phase 4: instrument
  → Map each probe to a hypothesis; change ONE variable at a time
  → Debugger > targeted logs > never "log everything"
  → Tag all debug logs [DEBUG-xxxx] for cleanup
Phase 5: fix + regression-test
  → Write failing test FIRST (if correct seam exists)
  → Red → fix → green → re-run Phase 1 loop
Phase 6: cleanup + post-mortem
  → Confirm original repro gone, logs removed, prototypes deleted
  → Ask "what would have prevented this?"
```
🟢 Non-deterministic bugs: loop 100×, parallelise, inject sleeps to raise reproduction rate.

#### `grill-with-docs` — iterative interview
```
start
  → Read CONTEXT.md (if exists) + any ADRs
  → For each question:
      IF answerable by codebase exploration → explore instead of asking
      ELSE → ask one at a time
          → Challenge against glossary
          → Sharpen fuzzy language
          → Stress-test with concrete scenarios
          → Cross-reference with code
          → [Resolved term] → update CONTEXT.md inline (immediately)
          → [Hard-to-reverse + Surprising + Real trade-off] → offer ADR
  → Continue until shared understanding reached
```
🟢 ADR trigger requires ALL THREE conditions simultaneously.

#### `improve-codebase-architecture` — depth analysis
```
start
  → Read domain glossary + ADRs
  → Explore via subagent (Explore type)
      → Look for: multi-file conceptual bouncing, shallow modules,
        no-locality pure functions, tightly-coupled seam leaks,
        hard-to-test interfaces
      → Apply deletion test to suspects
  → Present numbered deepening candidates
      → [contradicts ADR] → mark clearly, only surface if friction is real
  → User picks candidate → drop into grilling loop
      → [New domain term] → add to CONTEXT.md
      → [Fuzzy term] → update CONTEXT.md
      → [User rejects with load-bearing reason] → offer ADR
```
🟢 Deletion test: "If I delete this module, does complexity concentrate or disperse?"

#### `prototype` — binary branch router
```
start → identify question type
  QUESTION="Does this logic/state model feel right?" → LOGIC branch
  QUESTION="What should this look like?" → UI branch
  AMBIGUOUS → pick branch matching surrounding code type

LOGIC branch (LOGIC.md):
  → State question in writing
  → Pick host language (match project)
  → Isolate logic in PURE module (reducer | state machine | functions | class)
  → Build minimal TUI: clear screen + re-render on every action
       Frame = [current state (bold labels)] + [keyboard shortcuts (dim)]
  → One command to run (existing task runner)
  → Hand over → capture answer → delete or absorb

UI branch (UI.md):
  → Default 3 variants (max 5)
  → Sub-shape A (preferred): existing page + ?variant= URL param
  → Sub-shape B (last resort): throwaway route /prototype/<name>
  → Each variant: structurally different (not just color/copy)
  → Wire with switcher component
  → Floating bar: left-arrow | variant-label | right-arrow
       Keyboard ←/→ cycles; hidden in production builds
  → Hand over → capture winner → delete losers → promote winner
```

#### `triage` — issue state machine
```
State roles: {needs-triage, needs-info, ready-for-agent, ready-for-human, wontfix}
Category roles: {bug, enhancement}

Each issue: exactly 1 state role + 1 category role at all times

Normal flow:
  unlabeled → needs-triage → {needs-info | ready-for-agent | ready-for-human | wontfix}
  needs-info → needs-triage (on reporter reply)
  maintainer may override at any time

Per issue:
  1. Gather context (full issue body + comments + labels + .out-of-scope/*.md check)
  2. Recommend category + state + reasoning → wait for direction
  3. [bug] → attempt reproduction: trace code, run tests → confirm/fail/needs-info
  4. [needs fleshing out] → run /grill-with-docs
  5. Apply outcome:
       ready-for-agent → post Agent Brief (AGENT-BRIEF.md template)
       ready-for-human → same structure + reason why non-delegatable
       needs-info → post triage notes template
       wontfix + bug → polite explanation + close
       wontfix + enhancement → write .out-of-scope/<concept>.md + link + close

Prefix on ALL issue comments: "> *This was generated by AI during triage.*"
```

#### `tdd` — red-green-refactor
```
Anti-pattern: horizontal slices (all RED then all GREEN) → DO NOT USE

Correct pattern: vertical slices via tracer bullets
  1. Plan: confirm interface changes + behaviors to test + interface design
  2. Tracer bullet: ONE test → confirm ONE end-to-end path → fails → pass
  3. Incremental loop: for each behavior → RED → minimal GREEN
       Rules: one test at a time; only code to pass current test; no anticipation
  4. Refactor: duplication, deep modules, SOLID → only when all GREEN
       Never refactor while RED

Test quality checklist per cycle:
  [ ] Test describes behavior not implementation
  [ ] Test uses public interface only
  [ ] Test survives internal refactor
  [ ] Code is minimal for this test
  [ ] No speculative features
```

#### `caveman` — toggle state machine
```
State: {INACTIVE | ACTIVE}

Trigger to ACTIVE: "caveman mode" | "talk like caveman" | "less tokens" | "be brief" | /caveman
Trigger to INACTIVE: "stop caveman" | "normal mode"

While ACTIVE:
  Drop: articles (a/an/the), filler (just/really/basically), pleasantries,
        hedging phrases, conjunctions where possible
  Keep: all technical substance, code blocks unchanged, exact error quotes
  Use: fragments OK, short synonyms, abbreviations (DB/auth/config/req/res/fn/impl),
       arrows for causality (X→Y), one word when sufficient

Auto-clarity exception (temporarily exit caveman):
  Trigger: security warning | irreversible action | multi-step where order matters | user confusion
  → Write in full, clear language for that section
  → Resume caveman immediately after

Persistence: ACTIVE across all turns; never auto-deactivates
```

#### `handoff` — document generation
```
start
  → Receive optional focus argument
  → Compact entire conversation context
  → Generate temp path: mktemp -t handoff-XXXXXX.md
  → Read file before writing
  → Write handoff document:
       - Focus (if argument provided)
       - Key decisions made
       - Current state of work
       - Next steps
       - Skill suggestions for next session
       - References to existing artifacts (by path/URL — do NOT duplicate content)
  → Do not modify any existing artifact
```

#### `write-a-skill` — scaffold loop
```
1. Gather requirements:
   - Domain/task coverage?
   - Specific use cases?
   - Executable scripts needed?
   - Reference materials?

2. Draft:
   - SKILL.md with YAML frontmatter + concise instructions
   - Additional reference files IF content > 500 lines
   - Utility scripts IF deterministic operations needed
   [SKILL.md target: ≤ 100 lines]

3. Review with user:
   - Does this cover use cases?
   - Missing or unclear anything?
   - More/less detail needed in any section?
```


### Control flows

#### `review` — two-axis parallel review
```
1. Pin fixed point: commit SHA | branch | tag | main | HEAD~N
   Diff command: git diff <fixed-point>...HEAD (three-dot)

2. Identify spec source (priority order):
     Issue refs in commits (#123, Closes #45)
     → path argument from user
     → PRD/spec file under docs/, specs/, .scratch/
     → ask user (or skip Spec axis if none)

3. Identify standards sources:
     CLAUDE.md, AGENTS.md, CONTRIBUTING.md, CONTEXT.md,
     docs/adr/, .editorconfig, eslint.config.*, biome.json, prettier.config.*

4. Spawn PARALLEL sub-agents (general-purpose type):
     Standards agent: reads standards docs → reviews diff → reports violations
       (cite file+rule, distinguish hard violations from judgment calls)
     Spec agent: reads spec → reviews diff → reports missing/scope-creep/wrong-impl
       (quote spec line for each finding)

5. Aggregate: ## Standards + ## Spec sections, verbatim, NOT merged/reranked
   One-line summary: counts per axis + worst single issue

Rule: NEVER merge or rerank findings across axes
```

#### `writing-fragments` — fragment accumulator
```
State: {file_path, fragment_list}

start → ask file path once if not provided
       → capture fragments from INITIAL USER PROMPT immediately
       → on first write: H1 working title + nothing else

Loop per turn:
  → Interview user relentlessly (no phases, no outlines, no structure)
  → As fragments emerge from either side:
       re-read file from disk (preserve user edits)
       append: \n---\n + fragment text
  → Mention addition in passing ("adding that"), don't interrupt
  → Accept editing commands: "cut last", "rewrite that one sharper", "merge those two"

Fragment file format:
  # Working title
  fragment-1
  ---
  fragment-2
  ---
  ...
  (no headings inside body, no tags, no imposed order)

End condition: user ends session (no "pile empty" end state)
```

#### `writing-beats` — beat journey
```
State: {raw_material_file, article_file, beat_count}

start → ask article save path once if not provided

Loop:
  [beat_count == 0] → draft 2-3 candidate STARTING beats from raw material
                       preview what beats each might lead to
                       user picks one
  [beat_count > 0]  → re-read article file from disk
                       offer 2-3 candidate NEXT beats from article's current position
                       user picks one

  → write ONLY that one beat to article file (sentence to multiple paragraphs)
  → beat_count++
  → loop

End: natural end of journey (not when pile is empty)

User commands: "rewrite that beat", "go back to beat N, try different" → edit in place
```

#### `writing-shape` — article shaping
```
State: {input_pile (read-only), article_file}

start → read input pile in full
       → ask article save path once if not provided

1. Draft 2-3 candidate openings (different thesis/angle)
   → user picks or composes hybrid
   → chosen opening defines what the article must do

2. Growth loop:
   → "What does reader need to hear next?" → pull from pile
   → Argue about format out loud with user:
        prose vs list | inline vs callout | table vs repeated | quote vs paraphrase | code vs inline
   → Append agreed block to article file immediately
   → re-read article from disk before every write (preserve user edits)
   → loop until article is done (user decides)

Rules:
  Raw material file: READ-ONLY — never edit
  Article file: append per block; edit in-place only by user request
  Gap rule: "pile lacks X → name the gap explicitly, ask user to fill or cut section"
```

## Algorithms and Non-Trivial Logic

### `diagnose` — feedback loop construction
10-step ordered list for building a reproducible feedback signal. The ordering encodes priority (unit test first, HITL bash last). Non-deterministic variant: "loop 100×, parallelise, inject sleeps, narrow timing windows."

### `improve-codebase-architecture` — deletion test
Functional: `deletionTest(module) → bool`. Does deleting the module concentrate complexity (pass-through, delete it) or does it reappear across N callers (earning its keep)?

### `improve-codebase-architecture` — dependency classification
4-category classifier:
- `in-process` → always deepenable, no adapter needed
- `local-substitutable` → deepenable if stand-in exists (PGLite, in-memory fs)
- `remote-but-owned` → Ports & Adapters: define port, inject adapter, in-memory adapter for tests
- `true-external` → Mock: inject port, mock adapter for tests

Rule: one adapter = hypothetical seam. Two adapters = real seam. Do not introduce port without ≥ 2 justified adapters.

### `prototype` — question-type router
Binary routing: `routePrototype(question) → "logic" | "ui"`. Tiebreaker: surrounding code type (backend → logic; page/component → ui). When user unavailable, state assumption explicitly.

### `triage` — .out-of-scope matching
Concept similarity matching (not keyword matching): "night theme" matches `dark-mode.md`. Applied during step 1 before any grilling. Three outcomes: Confirm (add to prior requests + close) | Reconsider (delete/update .out-of-scope entry) | Disagree (proceed with normal triage).

### `review` — three-dot diff
`git diff <fixed-point>...HEAD` (three dots = compare against merge-base, not tip). Important distinction from two-dot diff when branches have diverged.

### `writing-beats` — beat-vs-end detection
End condition is "natural end of journey" not "pile empty." Pile is a quarry — leftover fragments that don't make it in are expected and acceptable.
